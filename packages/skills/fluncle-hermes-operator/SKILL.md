# Fluncle Hermes operator

Fluncle's Hermes agent is a self-hosted [Nous Research Hermes](https://hermes-agent.nousresearch.com) gateway that fronts Discord and acts on the archive through the authenticated `fluncle` CLI. This skill is the operator's map for **changing** it: which lever to pull for a given change, the invariants not to break, and how a change actually reaches the running bot.

The architecture, security model, and the **exact build / run / verify commands and secret locations** live in `docs/agents/hermes-agent.md` — that doc is the source of truth, and this skill routes to it rather than restating it. Read the doc's relevant section before you touch anything; come here for the decision and the guardrails. The build context (Dockerfile, the versioned `config.yaml`, `SOUL.md`) is `docs/agents/hermes/`.

This is the **chat presence only**. The async enrichment agent, the Friday newsletter agent, and the audio-observation agent are different agents with their own docs under `docs/agents/` — out of scope here.

## The mental model: the repo is the truth, the box is a deploy target

Make every change **in the repo and commit it** — never hand-edit on the box as the canonical version. An on-box edit drifts from git and is erased the next time the box is redeployed or rebuilt; keeping the change in the repo is what makes the security-relevant bits (the role boundary, the voice) reviewable and a rebuilt box identical to the committed state.

From the repo, a change reaches the live bot through **one of two deploy targets**, and picking the right one is most of the job:

- **The Worker** (`apps/web`) — _what the agent may do_. The operator/agent permission boundary is enforced **server-side**, not on the box. A push to `main` auto-deploys it. So "let the agent do X" or "stop the agent doing Y" is a code change to a route's role guard, reviewed in git.
- **The box** — _how the agent runs and sounds_. The model config, the Fluncle voice, the image, and the secrets live on the devbox (reached over SSH on the tailnet; the address is in the operator's ops notes). These ship by redeploying the changed file into the agent's state dir (`~/.hermes`), or rebuilding the image, then restarting the container.

If you remember nothing else: **the publish boundary is server-side.** Reaching for the box to change what the agent is _allowed_ to do is the classic wrong turn — that lever is a Worker role guard, in git.

## Invariants (the why, so you don't fight the design)

- **Changes go through git.** The repo is canonical; the box mirrors it. This keeps the role boundary and the voice reviewable and reproducible, and means a rebuilt box matches the committed state. An ad-hoc box edit is a one-off the next deploy erases.
- **The permission boundary is server-side, full stop.** The box deliberately holds only a low-privilege `agent`-scoped token; the Worker refuses publish-/irreversible-class actions for that role. You never make the agent "more powerful" by editing the box — you change a Worker role guard, in git. (This is also why a leaked box token can't publish.)
- **Pin the image and the model.** Hermes is pre-1.0, and a sub-64k-context model takes the whole gateway down at startup, not just one feature. Bump deliberately, never float `latest`, and review the upstream pin periodically for security patches.
- **Voice lives in git, never the agent's self-improve loop.** `SOUL.md` plus the `copywriting-fluncle` skill are the only voice source; letting the agent rewrite its own identity drifts it off-canon with no review.
- **Public repo: no host names, IPs, or secret values in committed files.** Those live in the operator's ops notes and the secret store. The committed docs and this skill stay at the architecture / procedure level — match that when you edit them.

## The change → ship → verify loop

1. **Edit in the repo** and commit; the relevant doc section names the exact file.
2. **Ship to the right target.** A Worker/role change deploys on push to `main`. A box change (config / voice / image / secret) is redeployed to the box and the container restarted — the doc's § Run has the exact commands. Rebuild the image **only** for image/CLI changes; config, voice, and secrets are redeploy-the-file (or re-populate-the-env) + restart, no rebuild.
3. **Verify.** Run the doc's § Verify smoke test (CLI present; an agent-allowed read returns `{ok:true}`; a publish-class command is refused with a 403). For a voice or model change, also run the **voice gate** (doc § Voice gate) and confirm it through the live bot in Discord, not just the container. For a Worker role change, confirm the agent token now gets the new allow/deny against the deployed API.
4. **Quality checks for Worker changes.** Per `AGENTS.md`, run the `apps/web` typecheck / lint / build before pushing — a push to `main` is a production deploy.

## Crons (automation) — live since the 2026-06-23 cutover

Hermes is also Fluncle's automation orchestrator: trusted, no-untrusted-input loops over the `fluncle` CLI. These crons are **live on the box** — deterministic `--no-agent --script` jobs (no agent loop, no model-call storms). Canonical source is the `.sh`/`.ts` pairs under `docs/agents/hermes/scripts/`; `docs/agents/hermes/cron/jobs.json` is `"jobs": []` **by design** — every automation cron now runs from a **repo-checked-in host systemd `.timer`/`.service` pair** under `docs/agents/hermes/<job>-timer/`, migrated off the gateway `hermes cron create` scheduler so the schedule is code and survives a re-provision (docs/hermes-durable-deploy-rfc.md § Unit E). The sweep bakes to `/opt/hermes-scripts` (+ the enrichment DSP skill to `/opt/hermes-skills`) and auto-updates from `main` via pin-watch; the timer only triggers it. The gateway holds NO automation crons — only the Discord chat agent (`gateway run`).

**Reset boundary:** a re-provision restores **code** (baked), **schedule** (the timers, via `docs/agents/hermes/install-host-timers.sh`), and **secrets** (`fluncle-secrets-sync`). Accumulated `state.db` state (sessions/memories/cron-output history) restores from the daily `fluncle-backup`→R2 backup — restore tooling is the one named follow-on.

- **The eight `--no-agent` sweeps:** `fluncle-enrich` (5m, `tracks enrich` self-heal), `fluncle-context-note` (5m, drain `hasContext=false`, Worker-side Firecrawl), `fluncle-note` (10m, hybrid — auto-author the written `note`), `fluncle-backfill` (30m, Last.fm / Discogs), `fluncle-observation` (60m, hybrid — author the recovered-audio script then `observe --script`), `fluncle-render` (60m, the video conductor), `fluncle-social-capture` (10m), and `fluncle-newsletter` (`0 15 * * 5`, hybrid). All are `--no-agent`: a pure trigger ships deterministic stdout; a **hybrid** (note / observation / newsletter) wraps exactly **one** bounded `claude -p` authoring call (subscription auth, zero OpenRouter tokens) inside a deterministic gather/persist/deliver shell. All sit under the agent ceiling — reversible, internal, no public footprint — so scheduling them raises no authority an injection could abuse.
- **The weekly newsletter (`fluncle-newsletter`):** Friday 15:00 Amsterdam, author the edition with the inline voice canon, persist a draft via `fluncle admin newsletter draft` (admin tier — the box's agent token can draft). Two operator-visible extras the other sweeps don't have: (1) its timer expresses the timezone DIRECTLY — `OnCalendar=Fri 15:00 Europe/Amsterdam` — so the slot is correct across the CET/CEST flip regardless of the box-clock TZ (the gateway used `0 15 * * 5` against the box clock, which is why the box was pinned `TZ=Europe/Amsterdam`; the timer makes that pin no longer load-bearing for the newsletter); (2) the **send stays operator-gated** — the agent token 403s on `newsletter send`, so the sweep delivers a one-line **Discord summary + the exact `fluncle admin newsletter send <id>` command** for the operator to run. Nothing auto-sends; an un-sent draft is re-offered by the next Friday run. `RESEND_*` stays a Worker secret.
- **Mechanism (host systemd timers):** each cron is a `Type=oneshot` `.service` that `docker exec`s the baked sweep + a `.timer` (`OnBootSec` staggered, `OnUnitActiveSec` cadence, `Persistent=true`); the newsletter uses `OnCalendar`. Timers run in parallel (a strict win over the gateway's one serial ~300s-budget runner). A `docker exec` sends stdout to journald, not the gateway's `~/.hermes/cron/output/<job>/` files — so each sweep SELF-WRITES the `/status` marker via the shared `cron-output.sh` helper (the `fluncle-healthcheck` prober is unchanged). The retired gateway cron mechanism is documented for reference in `docs/agents/hermes/cron/README.md` § The cron mechanism.
- **Operating them:** smoke-test each command by hand on the box before enabling its timer (`sudo systemctl start fluncle-<job>.service`, then `journalctl -u fluncle-<job>.service`), and watch the first ticks + the marker under `/opt/data/cron/output/fluncle-<job>/`; the observation sweep spends Cartesia credits per render. Install all timers with `sudo bash docs/agents/hermes/install-host-timers.sh`.

### Adding or changing a cron — the fan-out checklist

A cron is a fan-out surface like any other (the `fluncle-surfaces` skill's shape, for a `cron` kind): one job touches the sweep, the timer, the bake, the freshness marker, the registry, `/status`, and the on-box prober. Miss one and the cron runs but is invisible (no `/status` row) — or shows a wrong next-fire. Everything is **in the repo** (canonical) and reaches the box by baking on merge to `main`; nothing is created on the box by hand. To add a new `fluncle-<job>` cron, touch all of these:

1. **The sweep** — write `docs/agents/hermes/scripts/<job>-sweep.{sh,ts}` (a thin bash entry that execs the bun orchestrator — the `--no-agent` runner dispatches by extension, so a bare `.ts` is fed to Python). Model it on an existing sweep: `enrich-sweep` for a pure trigger; `note`/`observe`/`newsletter-sweep` for a **hybrid** wrapping exactly one bounded `claude -p` authoring call. If it needs creds, source the shared `0600 ${HOME}/.fluncle-secrets.env` (the runner withholds recognized provider creds from the script env — GHSA-rhgp-j443-p4rf — so a token can only arrive via that file). Pin `PATH` + `BUN_BIN`/`FLUNCLE_BIN` absolutely (the runner's PATH is minimal).
2. **The freshness marker** — the sweep MUST self-write its `/status` marker via the shared [`cron-output.sh`](../../docs/agents/hermes/scripts/cron-output.sh) helper (a `# Cron Job: fluncle-<job>` header + a JSON summary as the **last line**, `ok !== false` = healthy) to `~/.hermes/cron/output/fluncle-<job>/<ts>.md` (= `/opt/data/cron/output/fluncle-<job>/`). A `docker exec` sends stdout to journald, not to that dir, so without this the `/status` row is permanently "no data". (The `fluncle-healthcheck` prober is the ONE cron that writes none — a self-read would be circular.)
3. **The bake** — nothing to do: `Dockerfile` line ~191 is `COPY docs/agents/hermes/scripts/ /opt/hermes-scripts/`, so a new file in that dir bakes automatically (the `RUN rm -f` right after strips `*.test.ts` + the un-deployed `clip-drip-sweep.sh` — add to that filter only if the file must NOT ship).
4. **The timer** — add `docs/agents/hermes/<job>-timer/` with a `.service` (`Type=oneshot`, `ExecStart=docker exec -u hermes … bash /opt/hermes-scripts/<job>-sweep.sh`), a `.timer`, and a `README.md`. For a pure cadence use `OnBootSec` (staggered) + `OnUnitActiveSec` + `Persistent=true`. For a **fixed wall-clock** slot use `OnCalendar=*-*-* HH:MM:SS Europe/Amsterdam` — **the host clock is UTC**, so a wall-clock cron MUST carry the explicit `Europe/Amsterdam` TZ suffix (systemd ≥252; the box runs 255) or it fires at the wrong hour. Register the unit in [`install-host-timers.sh`](../../docs/agents/hermes/install-host-timers.sh).
5. **The registry surface** — add a `cron.<job>` entry to `@fluncle/registry` ([`packages/registry/src/index.ts`](../../packages/registry/src/index.ts)): `kind: "cron"`, `probeConfig: { kind: "cron", cronName: "fluncle-<job>", cadenceMs }`, `weights: { status: … }`, `exposedContent`. For a **fixed wall-clock** cron add `schedule: { time: "HH:MM", tz: "Europe/Amsterdam", weekday? }` so `/status` computes the TRUE next-fire (DST and all) instead of a cadence estimate. `cronSurfaces()` is the single source of truth every consumer reads.
6. **`/status`** — the registry entry flows into [`status.tsx`](../../apps/web/src/routes/status.tsx) via `cronSurfaces()` automatically (`CRON_SCHEDULE` is built from the registry `schedule`). Add the id to `OPS_AUTOMATION_IDS` if it's an ops cron (backups/self-deploy/audits/health) — otherwise `groupServices` files it under "Track automation" by default.
7. **The on-box prober mirror** — add a row to `AUTOMATION_CRONS` in [`fluncle-healthcheck.ts`](../../docs/agents/hermes/scripts/fluncle-healthcheck.ts) (it mirrors `cronSurfaces()`) so the box prober reads the new marker dir at the right cadence and POSTs its `/status` row.
8. **Verify** — smoke on the box (`sudo systemctl start fluncle-<job>.service` → `journalctl -u fluncle-<job>.service` + a fresh marker), confirm the `/status` row goes green, and run `bun packages/registry/src/index.test.ts` (catalog invariants) + `bun run --cwd apps/web typecheck`.

**Changing** an existing cron is a subset: edit the sweep and/or the `<job>-timer/` unit, keep the registry `cadenceMs`/`schedule` in step with the timer (a drifted cadence makes `/status` freshness lie), and re-verify. A merge to `main` bakes the sweep into the image (pin-watch auto-deploys); the box is the deploy target, the repo is canonical.

## When the bot misbehaves

- Runtime logs live under the agent's state dir (`~/.hermes/logs/`, root-owned); the container's stdout shows init + the banner + warnings, not the full agent log.
- A gateway that won't start is usually the model pin below the 64k floor — check `config.yaml`.
- A "refused" publish is the role boundary working as designed: a publish-class command **should** come back 403 for the agent, and the agent relays that in voice. The operator runs those actions with the full token, not the bot.
