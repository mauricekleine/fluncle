# Fluncle Video

Use this skill to make Fluncle's per-track social videos: 1080×1920 vertical clips that put one banger under the burning eclipse, every one unmistakably Fluncle, no two looking alike. **This skill is the constitution.** The package (`packages/video`) is a near-blank canvas — machinery and brand law, nothing styled — and the agent writes the scene and the shader for each track itself. More creative freedom, more diversity; the doctrine lives here.

This file is a router. Read it top-down, then load the reference for the phase you are in. The doctrine below is authoritative; where a living doc in the repo adds detail, follow it, but the rulings here win on conflict.

## Why this exists

A styled component library makes agents produce same-looking videos: identical text spots and timing, default eclipse discs, the same vehicle run after run. The best output comes from an agent writing its own shader from scratch. So there is no styled scene library — the package is the core surface, and the rules that components used to enforce are doctrine you uphold by hand. Diversity is the point. If your video could be swapped with another and nobody would notice, you have failed.

The aesthetic is already captured and it is strong — warm dark, one burning sun, grain, the through-the-glass calm. **Trust it, and use it freely.** The rulings below fence the brand; the space inside them is wide and it is yours. Lean toward the bolder, more alive idea over the safe repeat — a render that is unmistakably Fluncle yet unlike the recent output is the win. The doctrine is a short list of laws plus the failure modes that recur, not a recipe to follow step by step.

**The north star: FLUID, ORGANIC, ALIVE, and a little ALIEN.** This is the strongest pull of the whole brand — the footage should look like something Fluncle FOUND living on a far world, **recovered footage from the far edge of the map**, not something a machine drew. Favour flowing, breathing, almost-biological motion and form over the rigid, gridded, and mechanical. A structure is welcome (an iris, a weave, a dot screen) — but make it _breathe_: let lines curve and bow, let a lattice flex and ripple, let edges soften and reform. When a vehicle has a rigid state and a fluid state, LEAD with the fluid one. Caustics, smoke, liquid light, a curving membrane — these are the home register; treat hard geometry as something the music is melting, not holding still.

## The two things that are always true

1. **The video IS the output bundle.** Author each video as one self-contained composition at `src/remotion/workbench/<CompId>.tsx` — `export default` the component, import only the core from `../cosmos`. `workbench/` is gitignored and `root.tsx` AUTO-REGISTERS everything in it (the composition id is the filename), so you NEVER edit `root.tsx`, a render touches no tracked file, and there is nothing to clean up. After render, `ship` copies that exact source to `packages/video/out/<log-id>/composition.tsx` with `props.json`, `render.json`, and the footage; the R2 bundle is the durable archive (the local `workbench/` file is disposable scratch).
2. **You write the scene.** There are no prebuilt vehicles, no styled scene components, no static image assets (fonts excepted). You compose the scene and author the GLSL shader yourself, against the core surface. The core gives you a canvas and the brand law; the picture is yours.

## The core surface (what you may import)

Everything comes from `src/remotion/cosmos.ts`. Nothing else exists to import; if you reach for a styled vehicle or a static image, it is gone on purpose.

- **`ShaderLayer`** + the **`GLSL`** snippet helpers — the GPU fragment-shader layer and the composable GLSL function strings. This is where you draw. See [references/cookbook.md](references/cookbook.md).
- **Audio hooks** — `useAudioReactivity`, `useEnergy`, `useBeat`, `useBass`, `useOnset` — the only legal source of audio reactivity (they read the `audio.*` curves). Prefer `useAudioReactivity` for the shared bus (`hit`, `swell`, `drop`, `disturbance`, and ready shader uniforms) and then decide locally what material property those signals disturb. NOTE: these only drive VISUALS — they make no sound.
- **`TrackAudio`** — plays the analysed clip so the render carries SOUND. **MANDATORY in every video:** drop `<TrackAudio audio={audio} />` in once. The hooks above only animate the picture; without this the cut is silent (the render asserts on silence and fails).
- **`useJourney`** — the narrative clock (`progress` / `phase` / `phaseProgress` / `arc`). One shared timeline for the whole scene.
- **`Grain`** — the CSS film-grain overlay (the system base texture for non-shader layers).
- **`Starfield`** — the orbital star field.
- **`TypePlate`** — the FIXED informational type system (identity lower-left, telemetry upper-right, prescriptive timing). Drop it once with scene-derived inks; never hand-roll the facts layout.
- **`FloatingType`** — the underlying typography primitive (the sanctioned roles + the ink-halo legibility guarantee). `TypePlate` and `CloseCard` consume it; reach for it directly only for `body` text a scene genuinely adds beyond the plate.
- **`CloseCard`** — the locked brand ending.
- **`paletteMix`** — locks palette roles to canon (field stays Deep Field, sun stays Eclipse Gold, ink stays cream; artwork hues live only in secondary swatches).
- **color helpers** (`withAlpha`, `mix`, `luminance`, …), **fonts**, and the **props contract** types (`NostalgicCosmosProps` and friends).

That is the whole kit. The README has the exact signatures.

## See what's already been made (your live reference)

You do not start blind, and nothing about prior work is hard-coded into this doc — it lives in the data, discoverable at runtime. The already-published videos are BOTH your diversity ledger AND your quality bar. Before you pick a vehicle:

- **Read the vehicle ledger.** `bun run --cwd apps/cli fluncle recent --json` lists recent findings; any with a `videoVehicle` set already have a video, and that field names the vehicle used. (The public `/api/tracks` carries the same field.)
- **Look at the actual posters.** Every published video exposes a poster image at `https://found.fluncle.com/<log-id>/poster.jpg`. FETCH a handful and open them — `curl -s "https://found.fluncle.com/<log-id>/poster.jpg" -o /tmp/ref-<log-id>.jpg`, then view the file. Names tell you little; the pixels tell you everything.

Use them two ways: (1) pick a vehicle and a look clearly DIFFERENT from what you see — never the same medium twice in a row; (2) match or beat the best of them. As you look, the through-line is clear: the strongest Fluncle videos are flowing, organic, grainy, and alive (recovered footage), never clean, geometric, or CGI-polished.

## Doctrine

Eleven rulings. They are LAW. If a concept fights one, change the concept.

1. **One Driver, one climax — scene-led, NOT gold by default.** Exactly ONE travelling, beat-synced medium per video. orb / lines / fractal / glass / glitch are EXAMPLES, not an enum — invented vehicles are welcome, provided there is exactly one. The video carries ONE committed CLIMAX moment — the "pop" — but it is **the vehicle's OWN material intensifying, in the SCENE's own palette, NOT a gold/sun effect bolted on.** Eclipse Gold is OPTIONAL here, never the default. A warm scene MAY let the vehicle's material run gold _only when the gold IS the vehicle_ (the vehicle's own crests/nodes going molten, never a glow on top); a cool or non-gold scene climaxes in ITS OWN colours — a deep-blue structure brightening, a chrome mass surging steel, a teal bloom resolving — and reaching for yellow there is the error. BANNED outright: the soft gold/yellow **"sun"** — a round bloom, halo, glow, or warm wash that appears (almost always ~2/3 in) regardless of the scene's palette. That bolted-on golden sun is the single most common way these clips go off-brand; cut it. Test the climax two ways: (a) delete the vehicle and the bright moment vanishes with it — there is no separate light source; (b) the climax colour is drawn FROM the scene's palette, never reached for because "Fluncle = gold." Its FORM is free (a shimmer, a crest, a resolving front, a surge, a single cell), never a default circular disc; its COLOUR is the scene's, gold included only when the vehicle's own material genuinely runs hot. One driver, one climax, free form and free colour, AND it is the driver itself. Two drivers means one is wrong.

2. **Always-Visible Vehicle.** The vehicle is on screen and holds the center of attention from frame one to the final frame; it may fade/scale in over the first second, transform, travel, intensify — never a late reveal, never absent waiting for the drop.

3. **Vehicle diversity.** Use the live reference above — read `videoVehicle` across recent findings and look at their posters — then pick a vehicle clearly DIFFERENT from what's been used. Parallel runs tend to converge on the same idea (cellular/voronoi, horizontal lines) unless each deliberately diverges, so commit to a distinct medium and primitive. Honor the current brief and the failure modes below. Repeat a recent vehicle only if the music genuinely demands it.

4. **Type is FIXED; the art is the variable.** Predictable information gets a predictable home: every clip renders its facts through **`TypePlate`** — identity (Artist — Title, with the release credit `Label (Year)` beneath) lower-left, telemetry (Found date + Log ID) upper-right, two categories that never share a stack (the release year is a catalog credit beside the label, never confused with Fluncle's own Found date) — and ends on the **`CloseCard`** at the same lower-left anchor. Timing is prescriptive (identity in ~1s, telemetry staggered after it, both clear before the drop so the climax plays on pure art); you may nudge the in-points onto a musical seam via the plate's props, never re-place the blocks or hand-roll the layout with raw `FloatingType`. What stays YOURS: **the inks** — drawn from the composition's own palette (its light, a swatch, a counter-accent), with emphasis welcome but always a colour FROM the scene, never a fixed brand colour; this includes the `CloseCard` signature (pass it `palette`), NOT a hard gold — gold stays the SUN, never the type. Every word passes VOICE.md ("Artist — Title", "Found Jun 4", sentence case, no exclamation marks); render only fields the props expose. Placement is law and the scene's character lives in the art and the ink, not in re-arranging the facts.

5. **Metadata must visibly matter.** Facts come from the track's Spotify metadata, surfaced in the props file — never from web search. (a) FACTS that may render on screen: release year and label (when present in the props). They are authoritative — straight from Spotify, the source of truth — so they need no citation and carry no alias-collision risk; render only fields the props actually expose, and never invent one. Rendering a verified fact is ENCOURAGED. (b) CREATIVE FUEL that never renders as text: the artwork palette the pipeline extracts, the album and label names, the track title, and the shape of the energy curve. This shapes your vehicle, texture family, palette lean, and scene concept; your run report must name how it drove specific visual decisions ("a liquid-funk label and a drifting energy curve → a glass vehicle, fluent family") so the operator can see it in the pixels. BPM is measured by the audio pipeline (beat-grid aligned, more accurate than any metadata tempo) — that is what the vehicle syncs to; musical Key, if the props expose it, is an optional DJ touch on the overlay, never required.

6. **The quad law (no printed rectangle, no porthole).** A `ShaderLayer` must never show its quad as a hard rectangle — handled two ways by layer type. A **full-bleed background** (the vehicle field) FILLS the frame: opaque, a real warm-dark value at every pixel INCLUDING the corners — it satisfies the law with nothing to fade. A **localized layer** that does not cover the frame (an orb, a glow, a bloom) fades color AND alpha to exactly 0.0 before its quad edge (a radial fade in layer space, r reaches 1.41 at the corners). Do NOT apply that radial/circular fade — or a tight `vignette` — to a full-bleed background: it turns the 1080×1920 canvas into a circular porthole. Gentle, off-centre, or directional edge-darkening for mood is fine; a centred radial crop is not.

7. **Motion law — audio modulates MATERIAL, never motion.** Movement and animation must be SMOOTH: anything that travels or animates — a vehicle, a field, a streak, a star, a rotating iris, a drifting caustic — advances MONOTONICALLY and continuously, from `u_time`, the journey `arc`, or a steady rise, on an eased curve. Audio (`u_beatPulse`, onsets, energy) touches NEITHER **position** NOR **speed/rate** — driving either makes the picture lurch and jitter on every kick. Audio is the EQUALIZER: it rides **brightness, size/scale, thickness/width, curvature, exposure, glow, grain** — material properties that pulse on the beat while the motion itself glides on unbothered. So a kick makes a filament brighter, fatter, more sharply curved; it does NOT make it move faster or jump. Keep procedural detail low-frequency enough (or fbm-softened) that it doesn't shimmer as it travels. The starfield is the same law: monotonic orbital drift, audio touches twinkle only.

8. **Musical cut.** 20s default; choose `--duration-ms` (10–30s) from the waveform; end on a drop or just before a transition, never mid-build.

9. **The music leads, the video follows.** Audio reactivity is the point, not a finishing touch — the vehicle must visibly, legibly move WITH the track: lock to the beat grid, let the drop and the onsets HIT, map the energy curve to real swings in exposure / scale / width / density. The test, applied on the rendered MP4 (not the stills): **if the motion would look the same with the audio muted, the video has failed.** This stays inside the Motion law — reactivity rides MATERIAL (brightness, size, thickness, curvature), never position OR speed (doctrine 7), so the equalizer punch is visible but the motion stays glassy-smooth — but it must be FELT; ambient drift that merely coexists with the music is the failure this kills. The operating phrase: **audio disturbs the material, not just illuminates it — and never the motion.** At concept time, write a reactivity map with at least one structural reaction (width, density, radius, scale, thickness, curvature), one light reaction (glow, exposure, One Sun intensity), and one texture reaction (grain, dither, chroma, edge roughness) — and confirm not one of them touches animation speed or travel position. Sharp, structured textures carry this far better than soft haze.

10. **Movements.** A pattern, however good, exhausts the eye in ~10 seconds — so a clip is 2–3 MOVEMENTS, not one texture held for 20s. Each movement boundary lands on a seam the music actually has (a drop, a bar boundary, a breakdown, an energy shift — read the beat grid and energy curve), the transition STRETCHES over roughly a bar AND is EASED — start it on the seam and let the new regime ramp in on a smooth slow-in/slow-out curve (`Easing.inOut(Easing.cubic)`, not a linear A→B zoom/stretch/pan), arriving like weather, never a cut or a mechanical slide — and the shift is a legible change of regime inside the SAME theme: a palette lean, a scale or density jump, a zoom or reframe, the structure tightening or breaking open, a new behavior of the same vehicle. Nothing drastic — the One Driver persists (doctrine 1), the world doesn't change — but a viewer who has adapted to the pattern gets re-captured. A clip where second 18 looks like second 4 (just brighter) has only one movement, and that is the failure this rule kills.

11. **Compose to the vehicle's nature (symmetry + centre of gravity).** Two compositional disciplines, and they are CONDITIONAL — they apply only to vehicles that have a centre or an axis, never forced onto one that doesn't. (a) **Symmetry:** if the vehicle is symmetric — a kaleido/fractal fold, a radial mandala, a mirror-fold, a concentric or voronoi field grown from a centre — that symmetry HOLDS for the whole clip, through every movement. It must not drift off-axis as it travels; a centre that wanders off to one side reads as broken, not designed. (b) **Centre of gravity:** if the vehicle has a single focal mass — a bloom, a body, a convergence, a dot-bloom — that mass sits either dead-centre OR at a decisive, obvious off-centre (rule-of-thirds, a corner); a mass that sits _slightly_ off-centre reads as a mistake, not a choice. **The exemption is real and important:** a translational / continuous-motion vehicle (a horizontal light-front, a sideways roll, a drifting field) has no single centre and no symmetry axis, so neither discipline applies; don't manufacture one. Decide which kind of vehicle you have, then commit to its composition.

## Failure modes (recognize them, they recur)

Not new law — the shapes of getting it wrong, so you can see them coming. These are the ways a render drifts off-brand that the rulings above don't spell out:

- **The polished-CGI / videogame look (the big one).** A vehicle that is too CLEAN, GEOMETRIC, and HARD-SHADED — a tidy Voronoi/stained-glass tiling, a flat raster grid, a smooth faux-3D solid — reads as "a 2010s 3D video game" or "an HTML canvas demo," NOT Fluncle. The brand is **lost footage recovered from outer space** (the Light-Years Rule): the texture must be lossy, grained, degraded, ALIVE. Two cures, used together: (1) **lean flowing-organic over sharp-geometric** — Fluncle's best work is flowing membranes, blobs, and sheets, never clean tilings or rasters; if a vehicle has a geometric skeleton, make it FLEX, curl, drift, and reform organically, never sit as a rigid lattice. (2) **Bake the Light-Years degradation INTO the composition** — heavy grain, dither, chroma-bleed, soft lossy edges, analog wobble — so it reads as recovered footage even at high render quality. The polish is NEVER the encoder's job: a clean shader at high quality still looks like CGI; a grain-soaked, flowing one looks FOUND. Test: if the frame could be a desktop wallpaper render or a game asset, it has failed — it must look like something Fluncle pulled out of a degraded transmission from the far edge of the map.
- **The bolted-on golden sun.** A soft gold/yellow bloom, halo, glow, or warm wash that ignites ~2/3 into the clip no matter the scene's palette — reached for because "Fluncle = gold." BANNED (doctrine 1): the climax is the vehicle's own material intensifying in the SCENE's colours, and gold appears ONLY when the vehicle itself genuinely runs hot. A cool scene climaxes cool.
- **The fake glow (the gradient-orb / PowerPoint look).** A clean, smooth, lit-from-within glowing shape — a radial-gradient sphere, a softly-glowing ring, a tidy bloom — used as the bright focal element reads as a CSS `radial-gradient` or a PowerPoint glow: fake, weightless, non-organic. This bites even when it's NOT gold (a cool glow is just as fake) and even when the structure around it is good. The bright moment must be ORGANIC like the rest of the frame: carry grain, irregular or broken edges, internal filaments or dithered texture, a dissolve — never a clean gradient circle. Better still, make the bright moment the VEHICLE's OWN structure intensifying (the rings / blades / filaments themselves going hot and textured), not a separate smooth orb sitting at the centre. Test: if the bright element could be a CSS `radial-gradient()`, it has failed — degrade and texture it until it looks recovered, not rendered.
- **Lose the vehicle two ways — fog or chart.** A vehicle reads when it is STRUCTURE with DEPTH, and there are two opposite ways to lose it. A **formless fog**: soft warm gradients with no legible form — pretty, but nothing is actually _there_. A **flat chart**: thin, hard, sparse ruled primitives that reveal nothing — structure with no depth or meaning. The target sits between them: **structure that looks like a real thing, but only just so**, and degraded/grained enough to read as found (see the videogame failure above). The moodboard shows the register: a halftone dot-screen dissolving a bloom (`moodboard/halftone-tulip-bloom.png`), a CRT/scanline raster over a subject (`crt-scanline-roses.png`), contour lines that DISPLACE to sculpt a form out of an otherwise flat ruled field (`contour-eclipse-lines.png`) — sharp AND representational AND deep. Don't let "sharp" collapse to "horizontal lines + a horizontal gold bar"; vary the axis and primitive — dots, mesh, cellular, caustics, membranes — see the cookbook.
- **The copy-paste climax.** Every clip blooming bright in the dead centre and settling — the climax landing as a centred mid-clip glow every time. The moment is always present and always ONE, but its TIMING, PLACEMENT, FORM, and COLOUR are yours: ignite late, build slow and hold, sweep across as a front, sit off-centre, pulse and recede, crest at the very end. `useJourney`'s depart→travel→arrive with a centred settle is ONE option, not a mandate. If your bright moment lands where the recent posters' did, move it.
- **The porthole.** A tight radial `vignette` (or a circular `edgeFade`) on the full-bleed background darkens the corners until the 9:16 frame reads as a circle — the quad-law fade meant for a localized orb, misapplied to the whole field. The background must reach the frame's edges and corners; keep any vignette gentle and off-centre, never a centred radial crop (doctrine 6).
- **Copy-paste ink, and clashing ink.** Placement is fixed (doctrine 4: the `TypePlate` owns it) — what must NOT be copy-pasted is the COLOUR. Derive the ink from the scene's own palette — a dusty rose, an artwork counter-accent, the scene's own light. **Match the ink to the palette where the type ACTUALLY SITS — especially the `CloseCard`: pull from the scene's ENDING/dominant colours, not a transient early accent** (a cool ink chosen from a cold opening reads wrong over a warm close, and vice-versa). `FloatingType` carries its own contrast guarantee, so a non-cream ink still reads. Cream is a safe default; the type is never the brand gold — its ink follows the composition, the `CloseCard` included.
- **Beat-jitter — over-correcting the reactivity (flow, don't strobe).** Reactivity must be FELT (doctrine 9), but the common over-correction is pulsing the material HARD on every quarter-beat, so it strobes and jitters instead of flowing — at worst an actual epilepsy risk. The cure is not LESS reactivity but SMOOTHER, less literal reactivity: (1) make the PRIMARY pulse the slow energy ENVELOPE and bar-level swells, not the per-beat transient; (2) keep per-beat / onset hits SUBTLE in amplitude and SMOOTHED with an attack/decay so each lands as a soft swell, never an instant snap; (3) don't default every reaction to the 1/4-beat grid — vary the rhythmic target (the bar, the phrase, the energy shift) so the whole thing breathes rather than ticks. The clip must FLOW; if the material lurches or stutters between beats, you've over-corrected. This is distinct from doctrine 7 (there the MOTION must be monotonic) — here even the legal MATERIAL reactivity must be smoothed, not strobed.

## The constants you cannot touch

Grain over the whole frame, always. One committed CLIMAX moment, expressed through the vehicle's own material in the SCENE's palette — gold OPTIONAL, never the default, never a bolted-on "sun" (doctrine 1). Oxanium for brand marks and numerals only. `Artist — Title` with the one sanctioned em dash. `Found Jun 4` dates (UTC, tabular, never released/added). Warm darks only. The `CloseCard` ending. Determinism: remotion seeded `random()` and frame-derived values only — never `Math.random` / `Date.now` / `new Date()`. The Retint Rule: gold is one accent among the scene's palette, never forced as a "sun"; the type is never the brand gold — its ink follows the composition (the scene's ending palette), the `CloseCard` included.

## Commands

Run from `packages/video` (or `bun run --cwd packages/video <script>`). Use `bun`, never npm/pnpm/yarn. Exact flags live in `packages/video/README.md`.

- `bun run social:preview <trackId> --skip-render` — resolve + analyze audio (BPM, beat grid, onsets, energy/bass curves), extract artwork palette, write `out/<trackId>.props.json`. Run first; the props file is your ground truth.
- `bun run social:preview <trackId> --composition <CompId>` — the full pipeline through render to `out/<trackId>.mp4`, pointed at your composition.
- `bunx remotion still src/remotion/index.ts <CompId> out/still-N.png --props=out/<trackId>.props.json --frame=N --gl=angle` — the stills for the critique loop (GPU shaders require `--gl=angle`).
- `bun run studio` — Remotion Studio, to scrub a scene live with default props.
- `bun run --cwd packages/video typecheck` and `bunx oxlint packages/video` — the gates before any render. Format with `bunx oxfmt --write <files>`, never prettier.

## Hard rails

- **Never publish to social.** Rendering is local unless the operator explicitly asks you to upload the R2 bundle. Social publishing is a separate, approval-gated step.
- **Audio: the pipeline resolver only (Deezer/iTunes).** Never source audio from YouTube or rip full tracks. No legal audio means no video; stop and report.
- **One video per run. Never commit or push.** Delete only the temporary local composition you authored after it has been packaged into the output bundle.
- **Every fact on screen comes from the props' Spotify metadata; every word passes VOICE.md.** Release year and label are authoritative and need no citation. Render only fields the props expose — never invent a fact, never web-search for one.
