# The admin shell

The contract for every `/admin` surface. The workspace chrome splits across two levels so it stays mounted as the operator moves between stations. The **persistent shell** â€” the sidebar (`admin-sidebar.tsx`, the only navigation surface, with its live count query) and the full-viewport content plate (the translucent glass pane over the cosmos) â€” is mounted **once** in the `/admin` layout route (`apps/web/src/routes/admin/route.tsx`), wrapping the `Outlet`. It never re-mounts on navigation, so the sidebar's count badges never blink out and refetch and the plate's backdrop-blur never re-composites (no background flash). Each guarded page renders only its **header + body** through `AdminShell` (`apps/web/src/components/admin/admin-shell.tsx`) â€” the fixed header grammar (sidebar trigger, title + optional subtitle, the `headerActions` slot) plus an optional `subheader` and the page body. Login stays a bare centered card outside the shell (the layout bypasses it by path). The layout also owns the two cross-page structural bits the page can't set on a shared frame: it lights the active sidebar entry from the URL (`navKeyForPath`), and it switches the one viewport-height, self-scrolling station (the Studio) into fill mode by path.

## The persona law

**The web admin is the operator's platform Â· `fluncle admin` is the agents' tool Â· `fluncle` is the crew's toy.** One persona per surface, and the persona decides where an operation's primary home lives: an operator workflow becomes a button on the web admin, an automation verb lands on the `admin`-tier CLI/API for the box sweeps and agents, and the public `fluncle` CLI stays the crew's. The operator's enemy is tooling fragmentation, so machine-bound jobs centralize into the web admin â€” a headless poller behind the API does the machine's work; no machine needs a face (the live-show visuals stay the one local ritual). A job reachable from several surfaces still has exactly one primary home under this law; the others are conveniences.

## The placement contract

One home per kind of control. Put each control in its slot and nowhere else.

- **Page-level actions** go in the header slot, top-right (`headerActions`). One primary action per page keeps the One Sun budget (DESIGN.md).
- **Row actions** live inline on the row, right-aligned.
- **Bulk actions** live in a selection bar that appears with the selection and names its scope (`3 selected`).
- **Destructive actions** sit behind a confirm (`AlertDialog` from `@fluncle/ui`) that names the object it destroys.
- **A new page** gets exactly one sidebar entry, placed with its object group in `admin-sidebar.tsx`, and its route registered in `navKeyForPath` so the persistent sidebar lights the right entry from the URL (a page with no entry of its own â€” the Studio â€” maps to its parent object's entry).
- **Filters and view state** live in the `subheader` strip and deep-link through search params, so every view survives reload and pasteable URLs stay the operator's bookmarks.
- **Per-operator display preferences** live behind the sidebar's Display settings cog.

**The disclosure law â€” one primary per object, everything rare hidden by default.** When the operator lands on a page, the one thing they came to do is the loudest element and reads at a glance; nothing competes with it. Every infrequent or destructive action is demoted off the resting surface â€” behind a `â‹®` menu (Phosphor `DotsThreeVertical`, `DropdownMenu` from `@fluncle/ui`), a hover card, an expand, or a dialog. Never surface two ways to do the same thing, and never let a control the operator uses once a week sit at the same weight as the one they use every visit. Progressive disclosure over density: show what's needed now, reveal the rest on intent. This is why the Renders row's cover IS the play button (no second Watch control) with Requeue/Purge in a `â‹®`, why the Artists page collapses to summary rows with add/remove behind a dialog, and why a clip's TikTok and YouTube legs are separate one-action rows instead of one stuffed panel.

## The Object Row

Every index page â€” a page whose job is "see the objects, open one" (Renders, Mixtapes, Recordings, Playlists, Artists) â€” presents its objects through one shared primitive, `object-row.tsx` (`ObjectList` + `ObjectRow` + `ObjectLead` + `ObjectGlyph`), so a row reads and behaves the same wherever it lands: a bordered, divide-y list; each row a leading visual (a cover, or an `ObjectGlyph` fallback at the shared `size-11` footprint), an identity that grows, then a right-aligned zone for the object's quiet meta and its one primary action, with anything rare in a `â‹®`. Track-shaped rows feed `FindingIdentity` as the lead (its `plate` cover doubles as the play button when the object has a clip); set-shaped rows use `ObjectLead`. An **editor** page (Playlists, Studio) is an accordion/builder by nature â€” its collapsed summary row still matches the Object Row look, but expanding reveals the editor, not a detail view. Do not reach for a card grid or a bespoke list; extend the primitive.

## The nav model

The sidebar is a flat object nav: Dashboard, then the objects in pipeline order (Findings, Renders, Artists, Labels, Catalogue, Galaxies; the "Sets" group â€” Playlists, Mixtapes, Dream-weaver; the "Studio" group â€” Recordings, Clips; then Newsletter), then an "Ops" group (the Costs station and the Usage & cost station), then the System cluster pinned to the foot (**ChatDnB** â€” the admin-gated spike where Fluncle answers over his own archive tools â€” beside **Prompts** and `/status`). The Ops group pairs COST-02 and COST-01: **Costs** (`/admin/costs`) is the operator's private subscriptions ledger (recurring + one-off spend, entered by hand), and **Usage & cost** (`/admin/usage`) is COST-01's read on what the pipeline SPENDS per automation step and per finding, aggregated from the append-only `cost_events` ledger (docs/track-lifecycle.md). The two share only the operator's mental model, never data or a combined total: `/admin/usage` renders CASH (real incremental money â€” the headline) and SUBSIDIZED draw (fixed-plan usage â€” subscription LLM tokens + on-box compute) in SEPARATE columns, never summed, with unpriced rows counted rather than laundered to $0. Neither Ops entry carries a count badge (aggregates aren't cheap honest counts). Dashboard owns `/admin` â€” the attention queue: every action the system needs as a row, zero rows the success state, snooze/won't-do persisted client-side (one operator, one browser â€” a localStorage map, since a server column could not see this browser's working set). Its sources (each an `AttentionSource` in `lib/attention.ts`) span the pipeline: a fresh/bounced TikTok draft, a clip's TikTok/YouTube legs, a cue-less recording, a distributing mixtape, the empty IG drip, a **drafted-but-unsent newsletter edition**, an artist needing a link review, an **unruled label**, a **pending crew submission**, and an **auto-note the echo gate held back**. `AttentionSource` is an EXHAUSTIVE union mirrored in six places (the zod enum in `packages/contracts/src/orpc/admin-attention.ts`; the union + the priority order + the primary-action switch + the dispatch's phrase switch in `lib/attention.ts`; the dashboard's icon + label maps; the CLI's label map; the Raycast menu bar's meta map) â€” adding a source means all of them, or the build breaks. The newsletter row surfaces the Friday `fluncle-newsletter` sweep's persisted draft â€” it authored the edition and posted the send command to Discord, but `send_edition` is operator-tier, so the draft waits until the operator sends it; the row is anchored to when it was drafted and deep-links to `/admin/newsletter` where the review + Send control lives (the send authority stays on that page, never on the queue row, see [docs/agents/newsletter-agent.md](./agents/newsletter-agent.md)). The submission row deep-links to the exact candidate in the review tray (`/admin/findings?submission=<id>`) and carries the on-box `fluncle-triage` sweep's advisory verdict when it has one (the pre-chew read; approve/reject stays the operator's, see [docs/agents/triage-agent.md](./agents/triage-agent.md)). The **held-note** row is the auto-note's echo gate made visible: the gate refuses to STORE a note that echoes a sonic neighbour, but it no longer DESTROYS it â€” the line is kept with its reason and raised here, because a pipeline that throws work away without telling anyone cannot be supervised (nobody can tell a good rejection from a badly-tuned threshold once the evidence is gone). It deep-links to the finding's note dialog (`/admin/findings?note=<trackId>`), where the held note and the neighbour it echoed sit side by side with the lifted run marked in both, and the ruling lives â€” Keep it / Edit it / Bin it, operator-tier, since accepting publishes a line to `/log`. The gate's thresholds are operator-tunable at runtime through the `settings` KV (a flip, not a deploy) and render beside the score. See [docs/agents/note-agent.md](./agents/note-agent.md). Labels owns `/admin/labels`, the record-label entity and the operator's CRAWL-SEED control: every label a finding has carried is a row, and its three-state ruling (`enabled`/`disabled`/`undecided`) says only whether the future catalogue crawler may seed from it â€” **crawl scope, never storage**: skipping a label removes it from the NEXT crawl's seeds and touches nothing already stored. A brand-new label enters `undecided` and surfaces as a queue row ("a new label to rule on"); ruling rides the operator-tier `update_label` op (an agent token 403s). See [docs/label-entity.md](./label-entity.md). Catalogue owns `/admin/catalogue` â€” **The Ear**: every track the archive knows and Fluncle never certified (a `tracks` row with no `findings` row), ranked by cosine similarity to its single NEAREST finding, with each row carrying the finding it matched as its WHY. It is a TELESCOPE, not a conveyor belt â€” a short, high-conviction list he wants to open â€” so it deliberately carries **no count badge** (the honest number is "how many are worth your time", and a `COUNT` cannot answer that; a telescope with a backlog badge is a conveyor belt). The lenses deep-link through `?lens=` â€” the telescope, the capture queue (the rows with no audio yet, which the ranking structurally cannot reach), the wrong-audio quarantine, and the dismissed restore pile â€” and the operator verdicts live on the rows: the cover is an inline 30s audition, **Log it** certifies the row in place (behind a confirm â€” certifying creates lore), **Not for me** dismisses it (restorable from the dismissed lens), and a wrong capture is flagged into the quarantine. Nothing on it is lit like a finding â€” no coordinate, no gold, no note â€” because those columns live on `findings` and a catalogue row has none. The ranking is PRECOMPUTED by the agent-tier `rank_catalogue` sweep (a request-time ranking would be a 10k Ă— 60 cross join over 1024-d vectors, per page load), so the page does no vector math at all. See [docs/the-ear.md](./the-ear.md). Galaxies owns `/admin/galaxies`, the naming gate for the sonic map: the nightly on-box cluster sweep keeps the sound-derived galaxies assigned (the k-means fit itself is an operator act at the held k â€” [docs/agents/cluster-engine.md](./agents/cluster-engine.md)) and the Worker mints each new galaxy a permanent machine handle, and the operator names the unnamed ones here â€” after auditioning each galaxy's member covers through the shared `/api/preview` relay (the same preview singleton `/mix` uses) â€” via the operator-tier `update_galaxy` op (naming is publish-class, so an agent token 403s; a machine handle never renders publicly). The header carries the naming progress (n of N named) because the public `/galaxies` lens gates on a fully-named map; rename and request-a-split are the rare acts, off the resting surface behind the row's `â‹®`. Findings owns `/admin/findings`, the pipeline board (the queue's deep-link target for the publish loop; the board's `?stage`/`?mix` deep-links survive â€” old `/admin?stage=â€¦` links redirect). An entry whose station doesn't exist yet points at the best current home for that object and stays unlit there; it lights only on the page that declares it as owner. Count badges carry live, cheap, honest server counts (a scoped `COUNT` per number); a number that can only be estimated stays off the rail.

### Prompts (`/admin/prompts`)

**Prompts** owns `/admin/prompts` â€” the prompt registry, the station where the operator edits **what Fluncle tells the models**, with no deploy and no box rebake ([docs/agents/prompt-registry.md](./agents/prompt-registry.md)). Every prompt that authors a Fluncle artifact in production is a row: the finding note, the spoken observation, the Logbook entry, the triage verdict, the newsletter edition, the context distil, the search filter. Each row says where an edit LANDS ("Live on the next sweep, no rebake" for the on-box five; "Live on the next request" for the two the Worker runs) and whether the repo's baked default or an operator override is running.

A prompt IS code â€” a bad edit silently degrades every artifact it touches until a human notices â€” so this station's primary goal is not "edit" but **"see what changed and put it back."** Opening a prompt shows the `{{variables}}` it may reference, an editor with a **live diff against the running body**, and the full version history with a per-version diff. **Roll back** is one action behind an `AlertDialog` that names what it does; because the history is append-only a rollback MINTS a new version rather than rewinding, so nothing is lost and the rollback is itself undoable. Reset-to-the-repo-default is the rare act, demoted behind the row's `â‹®` per the disclosure law. All three writes ride the single operator-tier `update_prompt` op. The station carries **no count badge** (there is no queue here â€” seven prompts, always).

## Auth

One identity, two carriers: the CLI and agents send `FLUNCLE_API_TOKEN` as a `Bearer` header; the browser carries a signed grant cookie (`{ role: "admin", iat }`, HMAC'd with `ADMIN_SESSION_SECRET` via `signState` in `env.ts` â€” the key never reaches the client). `requireAdmin` accepts either. The browser proves identity with **Login with Spotify**, allow-listed to the operator through `ADMIN_ALLOWED_EMAILS` (+ optional `ADMIN_ALLOWED_SPOTIFY_IDS`) in `admin-auth.ts`; the login exchanges the code only to read the profile and discards the tokens â€” it never touches the publish refresh token in `spotify_auth`. On success it sets the grant cookie (`Path=/`, 30-day window) and redirects to `/admin`. Login at `/admin/login`, sign out at `/api/admin/logout`; the gate is active in dev too (just without `Secure` on localhost).

- Admin UI ships through the impeccable flow (`shape` â†’ build â†’ `audit`) and honors DESIGN.md, PRODUCT.md, and VOICE.md.
