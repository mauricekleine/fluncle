## 0. Summary / the reframe

- **The unifying simplification: the app is _thin over data, deep in exactly two places_.** Almost everything (feed data, archive, deep-link resolution, video/poster URLs, palette, DTOs) is _consumption_ of surfaces the web already ships — `GET /api/v1/tracks` (findings + mixtapes, cursor-paginated), `/api/v1/tracks/{idOrLogId}`, `/api/v1/preview/{idOrLogId}`, the `TrackListItem`/`MixtapeDTO` types in `@fluncle/contracts`, the Media Transformations recipe in `apps/web/src/lib/media.ts`, the tokens in `packages/tokens`, the `fluncle://` scheme in `apps/extension/src/coordinate.ts`. The app adds exactly **two** load-bearing things, and that's where the real work and risk live: **the native feed** and **push**. Everywhere else: port the contract, don't re-invent it.
- **The whole risk surface collapses to two things:** (1) **blur/effects over playing video** (a native perf cliff), and (2) **App Store 4.2 "thin app" rejection**. Both share one answer — _make the experience genuinely native and media-rich_ (native preview audio + the brand videos + a real gesture feed), never a webview wrapper. Designing for (2) de-risks (1): fake glass over video, real blur only on static overlays.
- **The feed reads the findings feed, not the video-only `/api/v1/stories`.** `/api/v1/stories` filters `video_url IS NOT NULL`, so a freshly-logged finding can never appear there — which would break the locked "every finding feed-able from the moment it's logged." So the **Stories pager reads `GET /api/v1/tracks` (findings)**, newest-first, and resolves each card's media at render time (below). The video-only `/api/v1/stories` becomes an optional "videos only" mode, not the default source.
- **The media ladder (per card), and the baked-text correction:** the web Stories player uses `footage.social.mp4` — the **baked-text** portrait cut — because its overlay is light. The locked decision is a **native overlay over clean footage**; native overlay on baked text double-prints. So the ladder is: **(a)** clean square master present (`videoSquaredAt` set) → `videoCrop(logId,"portrait")` clean portrait + native overlay; **(b)** else → the **cover card** (a designed type, not a fallback): the cover with slow eclipse motion under the 30s preview audio; **(c)** else (no preview either — `previewUrl` is optional and live-resolved) → a **beautiful silent cover** with the in-fiction line ("the rest of this one hasn't beamed back yet", the Light-Years Rule), optionally Fluncle's spoken observation (`observationAudioUrl`) if present. All three rungs share **one overlay system** so the feed reads as one card with three fills, never "some cards failed to load."
- **Navigate the archive by the Galaxy, not a search box.** No public archive-search endpoint exists (the public `/search` is Spotify-candidate search for submission), and a search box is a DESIGN.md anti-reference. The DTO carries `galaxy:{key,name}` + `vibeX/vibeY` — so the archive's nav is **browse + a four-galaxy lens**, served by a defined `galaxy=` filter param on `/api/v1/tracks` (client-side-over-loaded-pages is a trap on a phone).
- **Decomposition (truly-coupled vs independent):** seven units. Unit 0 (scaffold), Unit 5-server (push backend), and the web server bits (AASA/assetlinks/`/privacy`) are independent and ship first. **Unit 1 (shared media kit + the feed-filter change) is a serializing dependency** for both the feed and the push client — not parallel; sequence it before Units 2/3. Units 2/3/4 (feed / archive+deep-link / design) parallelize after Unit 1. Unit 6 (ship) gates on paid accounts.
- **Stack (corrected from the panel):** **Expo SDK 56** (stable, 2026-05-21 — **RN 0.85.2, React 19.2.3, Reanimated 4.3, Hermes v1 default, New-Architecture-only**) + Expo Router v6 (native tabs) + **NativeWind v4 stable** (v5 is still preview; v4.2+ already supports Reanimated v4 — the "v4 pins Reanimated v3" claim was false) + **Reanimated v4 with its required `react-native-worklets` peer** + expo-blur/image/video/audio/notifications/font. **`react-native-skia` is out of the _feed_** (the brand videos carry the aliveness); a thin Skia layer earns its keep only on the no-video screens (Unit 4, D-list). Install everything via `bunx expo install`; verify the exact pins at `eas init`.

## 1. Context & goals

**Why now.** Fluncle spans web, API, CLI, SSH, MCP, Tor, DNS, an LLM agent, mixtape distribution, and a game — every surface a point of light over one public API under shared Log IDs (`PRODUCT.md`, the Galaxy). The surface a phone does uniquely: a **lean-back, full-screen feed** of the brand videos, and **push** — the only capability that reaches the crew when they aren't already looking. One more surface, same bar: dark-only, cover-led, fully in-fiction, fast.

**Goals (in reach this build):** feed-first Expo app; native playback folded into the feed; every finding feed-able from log time (cover/silent rungs); push on a new finding + a new mixtape from the `apps/web` publish boundary; native-adapted Nostalgic Cosmos; both-platform EAS builds; deep links resolving `fluncle://<coordinate>` and `https://www.fluncle.com/log/<id>`.

**Honest calibration (outside our control / gated):** store _acceptance_ is a judgment call (4.2) — we minimize rejection risk, we can't promise approval. Paid enrollment + `eas submit` are external/paid/operator-gated. And some submission artifacts are **account-blocked** (AASA Team ID, assetlinks SHA, App Store Connect privacy/age/screenshots): scaffolded pre-gate, finalized post-enrollment. The standing review-risk read (5.2 music rights, 4.2 thin-app, the pre-submit moves) lives in [docs/app-store-review.md](../app-store-review.md).

**Out of V1 (deferred, as scope):** submit-a-track, accounts + sync, the native Galaxy game (links out). Note `Save` is auth-gated → deferred; the feed still delivers real actions without auth (Open in Spotify via `spotifyUrl`, Share via `logPageUrl`), so it isn't crippled.

## 2. Unit 0 — Scaffold `apps/mobile` (the zero-decision unblock)

- New Bun workspace `apps/mobile` (root `apps/*` glob matches; add to `turbo.json`). Native deps via `bunx expo install`; JS-only via `bun add`.
- **Expo SDK 56**, Expo Router v6, TypeScript, New Architecture ON (always-on since SDK 55).
- **Scaffold deps (corrected):** `bunx expo install react-native-reanimated react-native-worklets react-native-safe-area-context react-native-screens expo-video expo-audio expo-image expo-font expo-blur expo-notifications expo-linking expo-constants @shopify/flash-list`; `bun add @tanstack/react-query nativewind` (NativeWind **v4 stable**) + Tailwind dev deps; `@fluncle/contracts` + `@fluncle/tokens` as `workspace:*`.
- **babel.config.js = `{ presets: ['babel-preset-expo'] }` and nothing else** — `babel-preset-expo` auto-configures the Reanimated/worklets plugin; **do not** hand-add `react-native-reanimated/plugin` (it moved to `react-native-worklets/plugin` in Reanimated 4.3 and the preset owns it; adding it manually double-registers and breaks the bundle). Reanimated 4 fails to bundle without `react-native-worklets` installed.
- **Metro monorepo wiring** (`metro.config.js`): `watchFolders=[monorepoRoot]`; `resolver.nodeModulesPaths=[app node_modules, root node_modules]`; wrap with `withNativeWind(config, { input: "./global.css" })`. `@fluncle/contracts`/`@fluncle/tokens` ship raw `./src/index.ts` (pure TS, no JSX/node) — Metro/Babel transpile them once the root is watched; no `transpilePackages`.
- **Bun linker = hoisted (APPLIED 2026-06-21).** The root `bunfig.toml` had no `linker` set → bun used its **isolated** store layout (`.bun/<pkg>@<ver>/…`), and Metro's bundle failed with `Cannot read properties of undefined (reading 'transformFile')` / a `@babel/core` plugin `MODULE_NOT_FOUND`, because Babel can't resolve its plugins across bun's isolated silos (`@babel/core` wasn't hoisted to top-level `node_modules`). Fix (done): `[install] linker = "hoisted"` in the root `bunfig.toml` + reinstall → `@babel/core` hoists, the Metro bundle succeeds (4.7MB iOS Hermes bundle), and `apps/web` still typechecks clean (verified — no regression to the production app). Repo-wide but hoisted is the more compatible layout; the other apps (Vite/TanStack/tsc) are unaffected. The standard requirement for an Expo app in a bun monorepo.
- **Skia native binaries need `trustedDependencies` (APPLIED 2026-06-21).** `@shopify/react-native-skia` downloads its prebuilt binaries in a `postinstall`, which **bun skips by default** (lifecycle scripts run only for trusted deps) → `pod install` fails with "Skia prebuilt binaries not found. Run npx install-skia". Fix (done): `"trustedDependencies": ["@shopify/react-native-skia"]` in the root `package.json` so fresh installs run the postinstall in the store package dir (where it lands correctly). **Two-copy gotcha under the hoisted linker:** there are two physical skia copies — the hoisted `node_modules/@shopify/react-native-skia` (real dir) and the store `node_modules/.bun/@shopify+react-native-skia@<hash>/.../@shopify/react-native-skia`. CocoaPods (`use_native_modules!`) resolves the **store** copy; the podspec checks `libs/ios`+`libs/macos` relative to itself. The `.bin/install-skia` symlink targets the _hoisted_ copy, so backfilling the current install must run the **store** copy's `scripts/install-libs.js` directly (`node node_modules/.bun/@shopify+react-native-skia@<hash>/node_modules/@shopify/react-native-skia/scripts/install-libs.js`). On a fresh `bun install` the trusted postinstall handles it; if a reinstall ever recurs, re-run that store script. (Any future native dep with a binary-fetching postinstall wants the same `trustedDependencies` treatment under bun.)
- **`app.config.ts` CANNOT import the TS tokens package** — Expo evaluates the config transpiled standalone ("doesn't support external TypeScript modules"). For the 3–4 config-time hexes (splash/adaptive-icon/notification-icon backgrounds) **hardcode them with a `// from packages/tokens` comment**, or read a generated `.json` via `fs` at config time. The tokens _import_ belongs in the app bundle (`src/theme/`), never in `app.config.ts`.
- **Fonts (with the Tabular-Rule fix):** Oxanium + Monaspace Krypton via the `expo-font` config plugin (build-time embed). Native needs TTF/OTF (prefer the upstream Monaspace OTF release over lossy woff2→TTF). **Critical:** `fontVariant:['tabular-nums']` does **not** apply to custom fonts on iOS (expo/expo#20048) — so the Log ID would jitter, breaking DESIGN.md's Tabular Rule on the hero numeric. Fix: **vendor an Oxanium subset with the `tnum` OpenType feature baked as default** (renders tabular without `fontVariant`). That font file is a Unit 0 deliverable. (Also: the token key is `fontVariation: "tabular-nums"`, non-standard — don't copy it verbatim into RN.)
- Layout: `app.config.ts`, `metro.config.js`, `babel.config.js`, `tsconfig.json` (extends root, `@fluncle/*` paths), `global.css`, `assets/fonts/`, `src/{api,lib,theme}/`, `app/{_layout.tsx,(tabs)/_layout.tsx,(tabs)/index.tsx,(tabs)/archive.tsx,log/[id].tsx,galaxy/[coordinate].tsx}`.

**Acceptance:** `bunx expo start` runs; dev build boots; `@fluncle/*` import + typecheck; Oxanium digits render visually tabular on **both** platforms (no width jitter on update).

The "port the contract" move + the one feed-source change. Web-side, gates the feed and the push client.

Gate: the moved `media.test.ts` passes and `bun run --cwd apps/web build` is green. **Acceptance:** `media.test.ts` green in contracts; web builds; the findings feed + `galaxy=` filter return correct pages.

## 4. Unit 2 — The native feed (the hero)

A full-screen vertical pager over `GET /api/v1/tracks` (findings), cursor-paginated, newest-first.

- **Pager: `@shopify/flash-list` v2** (New-Arch-only, production-grade), vertical, one cell = screen height, `pagingEnabled`, `decelerationRate="fast"`. **Known Android risk:** FlashList #1200 — `pagingEnabled` can jump to the last index on a small gesture on Android, and #2050/#1966 (visible-content-position / nested-scroll loops) bite full-screen paging. So the de-risk spike **must verify one-card-per-swipe snapping on a low-end Android device specifically**; pre-decide a `FlatList pagingEnabled` fallback as the Android path if #1200 reproduces.
- **Player model (corrected): one `useVideoPlayer` per mounted cell, paused unless viewable.** FlashList recycling bounds the live count (~3 mounted). The draft's "3 shared players keyed by visible index" violates hook rules (hooks can't be created in recycled cells/`.map`) and fights recycling — rejected. `onViewableItemsChanged` (`itemVisiblePercentThreshold:80`) drives play/pause; the visible player `.play()`+`loop`, neighbors pause; reassign source on recycle via `player.replace()`.
- **Audio session (one coordinator owns it — net-new module):** expo-video + expo-audio both write the OS audio session; calling `setAudioModeAsync` "once" is insufficient when video cards and cover-audio cards alternate. Build a single **audio-session coordinator** every player routes through. Config: `duckOthers` (interrupt other apps when a card has sound; not `mixWithOthers`), `shouldPlayInBackground:false`, `playsInSilentMode:true` after unmute. expo-video config plugin: `supportsBackgroundPlayback:false`, `supportsPictureInPicture:false`; **no** iOS `UIBackgroundModes:["audio"]`. **Muted autoplay → unmute on first gesture** (mirrors web `stories-player.tsx` `soundOff`); a speaker toggle after. On every swap, explicitly `.pause()` the outgoing player/audio so only the visible card emits sound. Specify the interruption matrix: incoming call, route change, app background, iOS silent switch, Spotify-already-playing. The clean square master **has licensed audio** (`docs/video-variants.md`: "audio, clean"), so the visible video card emits song audio when unmuted.
- **The media ladder per card** (from §0): clean portrait (`videoCrop` + native overlay) → cover + preview audio → silent beautiful cover (+ `observationAudioUrl` if present). **Cold-MT latency:** a never-seen `videoCrop`/`videoPoster` URL transcodes on first hit (H.264 MP4 + range — natively playable by expo-video, verified) — pre-warm current+next crop/poster URLs on viewability and paint `videoPoster` first. `onError` is an expo-video status/error **listener** → `player.replace(rawMaster)` (not a DOM `onError`). **Preview-audio cost:** `/api/v1/preview/{id}` is a live, `no-store` relay (re-resolves Deezer/iTunes per hit) — do **not** preload neighbor audio; only the visible cover card fetches, and cancel on swap.
- **Native overlay** (RN layers over `VideoView`/cover, shared across all three rungs): Log ID (Oxanium, tabular via the tnum subset), `Artist — Title`, note, Found date, action row — **Open in Spotify** (`Linking.openURL(spotifyUrl)`), **Share** (`Share.share({ url: logPageUrl })`), mute toggle. Save deferred (auth). Styled from `@fluncle/tokens`; reduced-motion via `AccessibilityInfo`.
- **Data: TanStack Query (react-query) is the API seam (locked).** Mirror web's conventions — one `QueryClient` via `useState` in the root layout; `useInfiniteQuery` on the findings feed, `getNextPageParam: p=>p.nextCursor`; URL shareable per-flick via `router.setParams`. **Every API touch goes behind typed hooks** (`useFindingsFeed`, `useFinding`, `useRegisterDevice`) returning `@fluncle/contracts` types, with their `queryFn`/`mutationFn` internals isolated in one `src/api/` client module. This is what lets Phase 0 build against typed mocks and Phase 1 swap to the real transport without touching the UI. **Forward-compatible with the oRPC migration by design:** oRPC ships `@orpc/tanstack-query` (`createTanstackQueryUtils(client)` → `.queryOptions()`/`.infiniteOptions()`/`.mutationOptions()`/`.key()`), so when oRPC lands the hook internals become `orpc.tracks.list.infiniteOptions(...)` and the consuming components don't change (verified against `/dinwwwh/orpc`, June 2026).
- **Offline/empty/error states** (also a 4.2 defense): specify a cold/offline/empty feed in-fiction ("quiet sector tonight", per VOICE.md) — a real brand surface, not an afterthought. Cache the first page for offline open.
- **Mixtapes** don't appear in the pager (no per-finding video); they live in the archive as F-marked checkpoints.

**Acceptance:** ≤~3 live players while scrolling; only the visible card emits sound across the interruption matrix; all three media rungs render with one overlay system; Android one-card-per-swipe verified; reduced-motion stops ambient motion.

## 5. Unit 3 — Archive + deep-link finding screen

- **Archive tab:** `useInfiniteQuery` over `GET /api/v1/tracks` (`FeedItem = TrackListItem | MixtapeDTO`, discriminate `type === "mixtape"`). Findings → native track rows (Log ID, `expo-image` on `albumImageUrl`, Artist — Title, chips); mixtapes → checkpoint variant (`coverImageUrl`, `memberCount` · run time, the literal `F`). Native idiom: `log/[id]` as a pushed native screen or swipe-to-dismiss sheet — not a web "plate" clone.
- **`log/[id].tsx`:** `GET /api/v1/tracks/{idOrLogId}` → `{ ok, track }` or `{ ok, mixtape }` (parse the `ok` envelope). The deep-link target.
- **Deep linking:** `app.config.ts` `scheme:"fluncle"`; iOS `associatedDomains:["applinks:www.fluncle.com"]`; Android `intentFilters` (`autoVerify`, `https`, host `www.fluncle.com`, `pathPrefix:/log`) + the `fluncle://` scheme. **The real pattern** (apps/extension/src/coordinate.ts L22) is `/fluncle:\/\/(\d{3,4}\.(?:\d\.\d[A-Z]|F\.\d[A-F]))(?![0-9A-Z])/gi` — note the `fluncle://` prefix, the boundary lookahead, and the **`g`+`i` flags**. The `g` flag makes `.test()` stateful (`lastIndex` advances between calls) — use a **fresh non-global** copy or `matchAll`. The `i` flag means it matches lowercase; **uppercase the coordinate before resolving** (the `/log/$logId` DB lookup `where track_id=? or log_id=?` is case-sensitive, and `isLogId`/`isMixtapeLogId` require an uppercase mark).
- **Archive navigation = the Galaxy lens (defined server param, not client-side).** A `galaxy=` filter on `/api/v1/tracks` (Unit 1) backs four galaxy entry points → filtered list (client-side-over-loaded-pages is a broken phone UX — empty results filling as you scroll). The four galaxy names are canon — use `galaxy.name` from the DTO, don't invent. Findings with no placement (`galaxy` null, `vibeX/vibeY` null until tagged) need an in-fiction **"uncharted"** state. The draggable energy×mood vibe-map plane is a deliberate fast-follow, not crammed into V1.

**Acceptance:** cold-start `fluncle://<coordinate>` and `https://www.fluncle.com/log/<id>` land on the right finding; mixtapes render as checkpoints; the galaxy lens returns server-filtered findings + names the uncharted state.

## 6. Unit 4 — The native Nostalgic Cosmos (design system)

Native-adapted (palette/voice/cover-led + native nav; not a web "plate" clone).

- **Version matrix (corrected):** Expo SDK 56 + New Arch + **NativeWind v4 stable** (production track; v4.2+ supports Reanimated v4) + **Reanimated v4 (+ `react-native-worklets`)** + expo-blur/image. Adopt NativeWind v5 only if a needed `@theme`/CSS-first feature forces it (it's preview). RN 0.85's new animation backend + Hermes v1 default lower the perf cost of the static-screen Skia layer below.
- **Styling: NativeWind v4 stable**, fed by `packages/tokens` via a **token→RN adapter** (mandatory, not "import the colors"): **four tokens carry 8-digit-hex alpha** (`dustLine #d0b99029`, `goldVeil #f5b8001a`, `dustVeil #d0b9901a`, plus glows) — RN's `#RRGGBBAA` parsing is unreliable cross-platform, so the adapter converts them to `rgba()` (or base+opacity). Web units don't port (`rem`→×16 px, drop `cubic-bezier`/font stacks). Recommend importing the TS tokens object into `tailwind.config.js theme.extend` (stable on v4). Map: `eclipseGold→primary`, `starlightCream→foreground`, `deepField→background`, `stardust→muted-foreground`, `dustLine→border`, `reentryRed→destructive`.
- **Grain (Light-Years Rule):** a single baked tiled PNG (`grain-160.png`, ~3–6KB) under content, every layer (faithful to the web's "one tile" discipline). AA verified against the _pane+grain composite_, not the raw sky.
- **The chrome-liveliness call (panel pushback, reconciled):** Skia stays **out of the feed** (full-screen video carries it; don't spend the cosmos-drift budget there). But on the **no-video screens (archive, log)** the only life would otherwise be one opacity-breath on a baked PNG — the static/HTML-y look the operator's taste explicitly rejects. So **the static-screen backdrop must feel alive**, via one of (decision D5): **(a, recommended) a thin `react-native-skia` layer on archive/log only** — real `FractalNoise` grain + a true `RadialGradient` bloom that moves and **heats on interaction** (small static surfaces, affordable cost); or **(b)** stay Skia-free but raise the bake: 2–3 grain frames cross-faded (shimmer, not a static tile) + a compound bloom (slow scale+translate+opacity parallax). Either way, add the **Ignition heat-on-touch** the draft omitted: the Log ID/coordinate and primary action catch Gold-Veil/Eclipse-Glow on press.
- **Cover backdrop + bloom:** `expo-image` cover at `opacity:0.5` mounted once in the navigator background slot (RN has no `position:fixed`). Bloom = off-center radial (baked PNG, or Skia per D5), gold ≤13% alpha, sun fixed where the cover's sun sits; One Sun Rule (~10% gold) is a review gate.
- **Glass:** **never blur over playing video or the animated cosmos** (per-frame GPU readback tanks framerate, worst on Android). Fake panes natively (Sleeve-Black 48–86% + 1px Dust-Line + grain under) — the half-opacity cover shows through, a faithful Through-the-Glass translation. Real `expo-blur` BlurView (intensity ~40–60, dark tint, Android `blurMethod:"dimezisBlurView"`) only on static overlays (sheets, nav bar over a non-animating list).

## 7. Unit 5 — Push notifications end-to-end (the new server capability)

Fires on a **new finding** and a **new mixtape**. `apps/web` owns it. Server is independent of the app, ships/tests first.

**Status (Phase 1B — server backend LANDED):** the `register_device` backend + send-on-publish is built, tested, and merged (this RFC's §7 server half). It ships as a **NO-OP until `EXPO_ACCESS_TOKEN` is set**, so it's safe in production unprovisioned. What landed: the `devices` domain (`register_device` / `deregister_device` + the admin `sweep_push_receipts`), the `push_tokens` + `push_receipts` tables (migration `0026`), the `lib/server/push.ts` send module (chunk ≤100, per-category mute, parallel `Promise.allSettled` POSTs, `waitUntil`, immediate + receipts-driven dead-token reaping, never-throws), the publish + mixtape hooks (the mixtape one transition-guarded against the per-platform double-fire), the GDPR sweep, the `/privacy` push-token note, and unit tests for chunk/mute/no-op-when-unset/`DeviceNotRegistered` + the mixtape single-owner guard. **One deliberate deviation from the draft below:** the endpoint is authored **contract-first in the oRPC shape** (a `devices` contract + handler module), NOT the `routes/api/devices.ts` + `aliasHandlers` mirror the §7 draft describes — the public API surface migrated to oRPC after this RFC was written (the spec flip, #78), so the oRPC pattern (mirroring `submit_track`) is now the canonical one; the dual `/api` + `/api/v1` mount is handled by the oRPC rails. **Still deferred (client-side or follow-up):** coalescing/quiet-hours (the brand-quiet debounce), the external cron that calls `sweep_push_receipts`, the `lastSeenAt` staleness reaper, and the final copywriting-fluncle pass on the notification strings.

**Transport: Expo Push Service** (`POST https://exp.host/--/api/v2/push/send`) — keeps `apps/web` a single non-platform HTTP caller (the Telegram/Postiz/Loops side-channel pattern); APNs/FCM credentials upload once to EAS (`eas credentials`). Hand-rolled chunking in the Worker (the `expo-server-sdk` is Node-oriented); honor **≤100 messages/request** and the **600 notifications/sec project ceiling** (never bites at Fluncle scale, but note it).

**Server (apps/web):**

- **Endpoint** (Convention B): op **`register_device`**, `operationId: registerDevice`, `POST /api/v1/devices` (idempotent upsert on token); op **`deregister_device`**, `DELETE /api/v1/devices/{token}` (opt-out). **Author the bare `apps/web/src/routes/api/devices.ts` as source-of-truth** (`export const serverHandlers: ApiHandlers`) and the `apps/web/src/routes/api/v1/devices.ts` **mirror** via `aliasHandlers(serverHandlers)` — the bare file is canonical, the v1 file is the mirror (verified direction; `submissions.ts ↔ v1/submissions.ts` is the template; the dual-mount preserves POST bodies). Anonymous-compatible.
- **Rate limit (corrected):** use the generic `rate_limit_events` limiter — `enforceRateLimit({ action: "register_device", limit, windowMs, request })` in `apps/web/src/lib/server/account-data.ts` (NOT `submissions.ts`, which counts rows in the submissions table). Rate-limit hard: the endpoint is anonymous and a flood of format-valid fake `ExponentPushToken[…]` tokens would bloat the table and slow every fan-out.
- **Storage** `push_tokens` in `apps/web/src/db/schema.ts`: `token` TEXT (natural key), `platform` notnull, `appVersion`, `userId` nullable, `mutedJson` TEXT JSON (the `tracks.featuresJson`/`artistsJson` precedent — _not_ `userSavedFindings`, which has no JSON column), `createdAt`, `lastSeenAt`. Natural-key precedent is `userGalaxyState.userId` PK (not `userSavedFindings`, which uses surrogate id + unique index). **Runtime upsert is raw SQL** `db.execute({ sql: "insert into push_tokens (...) values (...) on conflict(token) do update set last_seen_at=..., app_version=..." })` — the repo writes via the raw `@libsql/client` (`getDb()`), not drizzle's builder (drizzle is Better-Auth-only); precedent `setMixtapeSoundcloud` (mixtapes.ts). **Migration generated, never hand-written:** `bun run --cwd apps/web db:generate` → `0026_*.sql` (next after `0025_sudden_killmonger.sql`); commit SQL + `meta`; Cloudflare auto-applies.
- **GDPR sweep (net-new, required):** add `delete from push_tokens where user_id = ?` to `accountDeletionStatements` in `account-data.ts` (covers future userId-bound rows). Anonymous rows (the V1 default) are reaped by a **`lastSeenAt` staleness policy** (prune rows not seen in N days) — state it in §6 privacy.
- **Send module** `apps/web/src/lib/server/push.ts` (mirrors `telegram.ts`): `notifyNewFinding` / `notifyNewMixtape`. Read tokens, drop those whose `mutedJson` includes the category, build `{to,title,body,data:{url},channelId}`, chunk ≤100, `Authorization: Bearer ${EXPO_ACCESS_TOKEN}` (add to `envKeys` in `env.ts`, read via `readOptionalEnv` — no-op when unset, like `LASTFM_*`; **verify whether Expo's "Enhanced Security for Push" is now on-by-default for new projects** — if so the token is required, not optional). **`waitUntil` discipline + its 30s ceiling:** `import { waitUntil } from "cloudflare:workers"` **in `push.ts`** (it's not re-exported by `edge-cache.ts`, only used there). `waitUntil` extends execution **only ~30s after the response** — a large fan-out can be killed mid-send → silent partial delivery. So **parallelize chunk POSTs** (`Promise.allSettled`) and batch prune-DELETEs into one statement; add: "if `push_tokens` exceeds a few thousand, move the fan-out to a Cloudflare Queue or a cron-driven admin route — `waitUntil` is not a durable job runner."
- **Dead-token reaping (corrected):** `DeviceNotRegistered` arrives via **receipts** (`getReceipts`, ~15min+ later), **not tickets** — pruning "on tickets" catches almost nothing. V1: prune what tickets do surface (invalid format / `PushTooManyExperienceIds`) **and** ship a small **receipts-sweep admin route hit by an external cron** (TanStack has no `scheduled()`; same pattern as `enrich-sweep`) that prunes `DeviceNotRegistered`. This is required V1 work, not deferred.
- **Hooks (corrected sites + transition detection):**
  - **Finding:** `lib/server/publish.ts` `publishTrack`, immediately after `posted_to_telegram=1` (L252, committed L257) alongside `purgeLogCache(logId)` (L267) / `lastfmLove` (L272). No extra gate needed: the `duplicate`/`incomplete_duplicate` guards (L56–69) throw _before_ this point, so a retry can't re-reach the hook (the draft's "gate on the transition" described protection that already exists upstream).
  - **Mixtape:** the `published` flip is in **`lib/server/mixtape-social.ts` `finalizeMixtapeDistribution`** (a SQL `CASE` in a `db.batch`), called **once per platform** (YouTube _and_ Mixcloud finalize routes) — a naive hook double-fires. **Detect the transition:** change the flip to a guarded `update ... set status='published' where status='distributing'` and check `rowsAffected`/`returning` (the `updateSubmissionStatus rowsAffected===0` guard is the precedent); only the call that _owned_ the `distributing→published` transition calls `notifyNewMixtape`.
- **Coalescing + quiet-hours (net-new, brand-required):** the brand is "quiet" (PRODUCT.md) and the crew cadence is weekly (newsletter); per-finding push is defensible at this low volume **but** an operator adding several findings back-to-back would fire several pushes in minutes — an uninstall driver. Ship a **short server-side coalescing window** (debounce findings published within N minutes into one "n new findings" push) + **quiet-hours suppression**, alongside per-category mute. (A daily digest would over-correct given the genuinely low volume.) Optional **idempotency ledger** (`notified_at` column or a `push_sends` table) for at-most-once under future refactors + partial-send resume.
- **Copy** is a separate voice pass (`copywriting-fluncle`): sentence case, no exclamation marks, "finding"/"banger". Constraint only.

**Client (Expo — requires a dev/prod build):** push needs a build (removed from Expo Go on **Android since SDK 52**; iOS Expo Go still works via EAS auto-config — build a dev client anyway for parity). Android channels first (`findings`, `mixtapes`, `HIGH`) — the Android 13+ prompt needs a channel to exist. **In-app consent language before the system prompt** (Apple 4.5.4 treats content-drop push as promotional → requires consent language in-UI + opt-out): a small surface naming what push is for, requested **after first value**, not cold on launch. `getExpoPushTokenAsync({ projectId })` → POST `/api/v1/devices`; **re-register on launch if the token changed** (Expo tokens rotate). Foreground handler: banner on, sound + badge off. Tap → `router.push(data.url)` (`/log/<logId>` or `/mixtapes/<logId>`); cold start `getLastNotificationResponseAsync()`. (Note: a push for a brand-new finding routes to `/log/<id>`, which renders the cover rung — it can't "jump into the feed" if not yet in the pager's loaded pages; route to the finding screen.)

**Acceptance:** `POST /api/v1/devices` upserts (raw on-conflict); publishing a finding sends once; the mixtape fires once across both finalize routes (transition-guarded); the receipts sweep prunes `DeviceNotRegistered`; account deletion removes the user's tokens; coalescing collapses a burst; unit tests cover chunk/coalesce/prune/mute; migration `0026` generated + committed.
