# The artist relationship

Fluncle keeps a canonical **artist entity** (`artists`, keyed on the Spotify artist id) with a `track_artists` many-to-many and an identity graph (`artist_socials`). The archive gets an artist entity, resolution, the `/artist/<slug>` pages, and the SEO/AEO graph.

This doc covers the **identity graph** — how `artist_socials` is resolved, reviewed, and rendered into the public artist page + `sameAs` JSON-LD. The entity + tables it builds on are Unit 1.

> **The relationship is identity-only.** Fluncle knows who an artist is and links out to them; Fluncle-the-account does not _follow_ them. The follow/champion motion was retired in two steps — the auto-follow cron (2026-07-08), then the operator's manual Follow-now / Undo (2026-07-09) — and what remains is the identity graph below: resolve → review → confirm → the public page. The `cron.artist-sweep` resolution was never part of the follow strand and is untouched. See [Why there is no follow](#why-there-is-no-follow).

## The data model

`artist_socials` is the identity-graph store — one row per `(artist_id, platform)`:

- `platform` — the socials only (`spotify | youtube | soundcloud | instagram | tiktok | mixcloud | twitch | bandcamp | beatport | twitter | facebook | homepage`). Beatport is a link-only store platform (the key DnB store/profile) — it was promoted out of the MB aggregator denylist to a first-class social. Twitch is the live-stream surface (many DnB acts stream there, and Fluncle itself lives at `twitch.tv/flunclelive`) — its channel handle is the first path segment, like SoundCloud/Twitter. The KG anchors (`mbid`, `wikidata_qid`) live as `artists` columns, not here.
- `status` — `auto` (MusicBrainz/operator, trusted) · `candidate` (Firecrawl-only, awaits a human glance) · `confirmed` (a candidate promoted). Only `auto`/`confirmed` render on the public artist page + `sameAs`.
- `source` — `musicbrainz | firecrawl | operator`.
- `reviewed_at` — when the operator last acknowledged THIS link, or NULL when it's still fresh. **Review lands on the link, not the artist**: a link is fresh (needs a look) iff `reviewed_at IS NULL`. A resolver INSERT (a new link) or a URL CHANGE nulls it (born/re-armed unreviewed); an untouched re-resolve keeps it; an operator add/edit is born reviewed (`reviewed_at = now`, they just wrote it). This replaced the old per-artist `artists.reviewed_at` acknowledgment so a single new link (e.g. a fresh Twitch link on an already-reviewed artist) surfaces on its own instead of re-flagging the whole artist. `artists.reviewed_at` survives only as the one-time backfill source (`scripts/backfill-artist-social-reviews.ts`), droppable once that has run in prod.

There is no follow/champion state on the row — the graph is purely who the artist is, not what Fluncle did about it.

## The `track ↔ artist` edge (how a crawled track earns its artists)

The `track_artists` edge is what lets `/artist/<slug>` show the rest of an artist's catalogue by an indexed seek (`track_artists_artist_id_idx`) rather than a full `artists_json` scan (the shape AGENTS.md forbids). A CERTIFIED track gets its edge at publish (`upsertTrackArtists`, folded on the stable `spotify_artist_id`). A CRAWLED track earns it two ways, in priority order:

1. **By stable Spotify id, at the anchor step (the primary path).** A catalogue track's Spotify anchor is filled by the agent-tier `anchor_track` op (`lib/server/anchor.ts`), which the box's Apify anchor sweep drives (docs/catalogue-crawler.md § the anchor). The verified candidate carries the track's artists, each with its `{ id, name }`, so `anchorTrack` connect-or-creates them by `spotify_artist_id` via `upsertTrackArtists` (`connectAnchorArtists`) with **no extra Spotify call**. This MINTS `artists` rows for artists Fluncle has never certified — the fragile name-fold is replaced by the stable id for every Spotify-present track. Avatar fetches are deferred (`fillImages: false`) to the batched `backfill-artist-images` sweep.
2. **By name-fold, at write time (the fallback).** A track with NO Spotify presence gets no id, so `linkTracksToArtistEntities` links it by a case-insensitive name match against artists that ALREADY have a row — minting nothing. Nothing hard-requires the Spotify id.

The recurring deploy backfill (`backfill-artist-links` in `db:backfill`) is retired: the inline paths cover the common case. Its name-fold reconcile is kept as a one-off operator script (`scripts/backfill-artist-links.ts`) for the case the inline paths miss — an artist entity minted AFTER catalogue tracks crediting it were already written/anchored. The one-off `scripts/backfill-artist-graph.ts` re-keys history onto the stable id: for each catalogue track with a `spotify_uri` but no stable-id artist link, it re-fetches the Spotify artists and connects them by id (resumable, rate-limited, hits prod — operator-run, never in the deploy chain).

**A crawl-minted artist is PUBLIC on its content, exactly as a discovered label/album is.** A findings-free artist with a row renders its `/artist/<slug>` page (its findings lead, then the rest of its catalogue), and once it clears the thin-content floor it enters the sitemap too — reachability follows the label/album rule. There is no certified-finding gate on the artist surface: a slug with no `artists` row 404s, a row renders, and the thin-content floor decides whether it indexes. That floor is now catalogue-aware — `resolveArtistPageData`'s `indexable` and `listArtistSitemapRows` count RENDERABLE tracks (certified findings via `countArtistFindings` PLUS the quieter catalogue rows via the catalogue's SQL-counted `totalTracks`), both read through the canonical `track_artists` join so an indexable page is never orphaned from the sitemap. The label/artist catalogue-heading link lights whenever the credited artist has an entity. The `/artists` INDEX is ONE unified A–Z list of every artist Fluncle holds (`listArtistsHubPage`) — certified findings and the wider catalogue in one alphabetical `?page=N` surface, a certified artist's name lit in Eclipse Gold, the rest unlit — so a crawler-discovered artist page is linked rather than orphaned. The unified-index shape is documented once, in [docs/album-entity.md](./album-entity.md#the-sitemap-carries-every-page-the-hub-is-one-unified-index).

### The history backfill (`backfill_artist_edges`, agent tier)

The inline paths above only edge a track WRITTEN after the graph existed (the graph is crawl-era-only, born 2026-07-15), so history was left carrying artist names in `artists_json` with no identity link — only ~12.3k of ~37.5k tracks had edges. That gap matters because the capture-authorization model matches BY IDENTITY through this graph (a track's audio may be bought iff a credited artist is qualified), so it needs the graph as full as honest matching allows. `backfill_artist_edges` (`lib/server/backfill-artist-edges.ts`) closes it: one bounded, agent-tier pass folds the whole `artists` + trusted-alias corpus into one in-memory `fold → artist_id` map, then, for every track with no edge and no backfill stamp, matches each `artists_json` name against it — first by exact case-insensitive fold on the canonical name, then via `artist_aliases` (`kind='name'`, `status in ('auto','confirmed')` — the search resolver's alias semantics) — and writes the `track_artists` edges `insert or ignore` (position from array order, `role` null). It runs on the `fluncle-artist-edges` HOST timer (docs/agents/hermes/artist-edges-timer/), one HTTP request per tick.

**The no-minting rule.** A bare name string is not enough identity to CREATE an `artists` row — a row is an entity with a public page — so the backfill mints nothing; it only links names to identities that already exist, exactly like `linkTracksToArtistEntities`. A fold that two distinct identities share is ambiguous and matches nothing (fail-closed); a primary `artists.name` beats an alias for the same fold. It makes NO vendor call (pure DB matching), writes graph identity only (no publish, no certification), and stamps every VISITED track (`tracks.artist_edges_backfilled_at`) — matched, partial, OR zero-match — so a zero-match row is not re-chewed each tick and a re-run over drained history is a cheap no-op (the `mb_recording_id_attempted_at` discipline).

**The residual is the point of the report.** Each pass returns tracks fully matched, partially matched, and zero-matched, plus `unmatchedNames` — the credited names that folded to no existing identity. That residual (a name Fluncle has never certified an artist for) is what decides whether a later paced MusicBrainz credit-sweep to MINT those artists is worth running; the backfill itself deliberately stops at existing identities.

### The MB credit sweep (`backfill_artist_credits`, agent tier)

Slice 0's residual ratified this sweep (`lib/server/backfill-artist-credits.ts`): it completes the graph for the ~14.3k **zero-matched** tracks — the ones slice 0 stamped but wrote no edge for, because no credited name folded to an existing identity. Those are exactly the tracks capture-authorization cannot reason about, so leaving them nameless leaves a hole in the gate. Where slice 0 could only LINK to identities that already existed, this sweep may MINT — and the difference is the licence a real MusicBrainz artist id gives that a bare name does not. For each zero-matched track carrying a MusicBrainz **recording** identity — `tracks.mb_recording_id`, or the `mb_<recording-mbid>` PK a crawler-born row carries (both checked, via the shared `recordingMbidFromTrackId`) — one paced `/recording/<mbid>?inc=artist-credits` lookup through the shared MusicBrainz client (`lib/server/musicbrainz.ts`, 1 req/s, circuit-broken on a throttle) names its credited artists WITH their MB artist ids.

**The mint rule.** Each credited artist is matched to an existing `artists` row by `mbid` (the `artists_mbid_idx` seek), else **minted** via `ensureArtistByMbid` (`lib/server/artists.ts`) — the one canonical mbid mint, the sibling of `upsertTrackArtists`'s Spotify-keyed mint, keyed instead on the MB artist id. An MB artist id IS identity (a curated, dereferenceable MBID), so a row born from one is honest — this is the RFC's identity-true mint, and the reason slice 0's no-minting rule does not bind here. The `track_artists` edges are then written `insert or ignore` (position from credit order, `role` null, the slice-0 edge shape). The MusicBrainz "Various Artists" placeholder is never minted or edged. A zero-matched track with **no** MB recording identity is **terminally skipped** — stamped so it drains, never retried.

**Its own reliability stamp.** The per-row stamp is `tracks.artist_credits_backfilled_at` — DISTINCT from slice 0's `artist_edges_backfilled_at`, which this sweep never writes or re-nulls (slice 0's semantics stay exactly as shipped). Every visited row is stamped — edged OR terminally skipped — so the worklist drains and a re-run is a cheap no-op (the `mb_recording_id_attempted_at` discipline). Because it makes vendor calls, the pass is SMALL (`MAX_BATCH` 40) and carries the label-lineage protections: a 60s response budget that pauses mid-page and resumes on the next request, and the `rateLimited` circuit breaker that stops the pass without stamping the throttled row. It writes graph identity only (no publish, no certification), so the box's agent-token `fluncle-artist-credits` HOST timer drives it (docs/agents/hermes/artist-credits-timer/) at a 5-minute cadence — the ~14.3k residual drains at ~40/tick over ~a day and a half. Minting these edges re-stales The Ear's ranking automatically: the rank corpus fingerprint (`rankCorpus`, `lib/server/catalogue.ts`) already folds the total `track_artists` edge count, so every new edge moves it and the catalogue re-ranks against the fuller graph — no separate fingerprint component is needed for the minted artists, because authorization depends on the edges (captured) and findings/label rulings, never on the raw `artists`-table cardinality.

## Resolution (`resolve_artist`, agent tier)

The box's `fluncle-artist-sweep` cron reads the resolve worklist (`list_unresolved_artists`) and calls `resolve_artist` per row: a MusicBrainz url-rels walk (→ `status=auto`, trusted) plus a Firecrawl gap-fill (→ `status=candidate`, operator-confirm before public). The gap-fill backfills **every missing social platform except `homepage` and `spotify`** (`instagram, tiktok, youtube, soundcloud, bandcamp, twitter, facebook, mixcloud, twitch, beatport`) — `homepage` is excluded because MB already covers it, and `spotify` is always known (it's the identity key). Everything Firecrawl returns lands as `status=candidate`, so the wider net stays behind the operator-review gate. `wikidata` classified during the MB walk is routed to the `wikidataQid` KG anchor on `artists`, not into `socials`.

The gap-fill spends the fewest Firecrawl credits by going cheapest-source-first: **(1)** scrape any **link hub** MB already carried — a `linktr.ee`/homepage — with one `/v2/scrape` JSON call (one page → every social it lists); the MB walk captures these hubs as scrape seeds rather than discarding them. **(2)** No MB hub → `/v2/search` for the artist's hub, then scrape it. **(3)** Still missing → a **disambiguated per-platform `/v2/search`** (`"<artist>" drum and bass <platform>` — the whole roster is DnB, which disambiguates hard against same-name acts in other genres), taking the first host-matching result that reduces to a profile root, guarded by a name-relatedness check on the platforms whose handle is the first path segment (so a bare-platform search can't attach a label/namesake account). The key insight: `/v2/scrape` and `/v2/search` act on **the URL you hand them** — the retired `/v2/extract` was fed the Spotify SPA (which lists no socials) and never polled its async job, so it returned nothing for every artist.

The MB walk finds the artist's MBID by **name search cross-referenced with the artist's Spotify URL** — the primary resolver: it queries `/ws/2/artist?query=artist:"<name>"`, deep-fetches each top candidate's `inc=url-rels`, and accepts the candidate whose MB Spotify url-rel's artist id equals the artist's stored `spotify_artist_id` (an exact identity match, accepted even over a higher-scored candidate). A candidate whose Spotify rel is present but DIFFERS is a namesake and is rejected. Only when NO candidate exposes any Spotify rel to cross-check does the walk fall back to a strong MB `score` (≥ 90) plus an exact normalized name match; otherwise the artist stays unresolved rather than resolve to a namesake (a wrong social link on a public artist page is worse than a missing one). The earlier ISRC→recording→artist-credit MBID lookup was **retired** — DnB ISRCs are frequently absent from MB's index and the walk landed on empty/wrong MBIDs, so most artists resolved to zero socials.

The worklist is **self-healing**: `list_unresolved_artists` returns artists with `resolved_at IS NULL` PLUS artists resolved to zero socials whose stamp is older than 30 days, so a transient MB failure (a 503 window, a namesake not yet disambiguated, an artist MB has since gained a Spotify rel for) re-tries at most once per window instead of sticking on 0 socials forever. Artists that already have socials are never re-queued. For an IMMEDIATE flush of a freshly-stamped empty backlog (which the 30-day window wouldn't touch yet), the operator runs `bun run apps/web/scripts/requeue-empty-artists.ts --confirm` (dry-run without `--confirm`), which clears `resolved_at` for every artist with zero `artist_socials` rows.

## Artist avatars (`artists.image_url`)

Each artist carries a canonical avatar in `artists.image_url` — the largest Spotify profile image (an `i.scdn.co` URL, the same host and attribution-by-link precedent as `tracks.album_image_url`; there is no re-hosting). Two write paths keep it filled, both idempotent and best-effort so a Spotify hiccup never blocks the fast synchronous add:

- **Create/resolve** — `upsertTrackArtists` (the ingest + entity-backfill path) calls `fillMissingArtistImages` after upserting the rows: for any of the track's artists still lacking an image (a freshly-minted artist always does), it batch-fetches the largest Spotify image (one `/v1/artists?ids=` call, ≤50 ids) and stamps `image_url`. An artist that already has one costs a single indexed read and no API call.
- **Backfill** — `backfill_artist_images` (agent tier, `POST /admin/backfill/artist-images`; CLI `fluncle admin backfills artist-images`) is the bounded, cursor-resumable sweep over artists minted before the column (`image_url IS NULL` + a Spotify id to look up). The on-box `fluncle-artist-sweep` cron drains one page (50 = one Spotify batch) every tick — best-effort, so a pinned box CLI predating the subcommand is logged and skipped, self-healing after the CLI re-bake. Spotify carrying no image for an artist leaves the column null (counted `skipped`, not `failed`).

Render: the `/artists` index is avatar cards (round Spotify image over name + finding count) and the artist page's "Similar artists" chips carry a small round avatar, both via `ArtistAvatar` — a graceful monogram tile (first letter) stands in when `image_url` is null. The image is decorative (empty `alt`, the name is adjacent); WCAG-AA, keyboard-reachable, reduced-motion-safe.

## Artist aliases (`artist_aliases`) — DnB's many-names problem

A DnB act records under many names (an alias, a real name, a sort name, a locale variant). `artist_aliases` is the store that reconciles them all to ONE artist entity without ever rewriting `artists.name` — the exact structural twin of `label_aliases` (docs/label-entity.md): a per-artist side table of alternative spellings, keyed `(artist_id, alias_slug, source)` unique, INSERT-only (`on conflict do nothing`), so a re-resolve never duplicates one and never disturbs an operator ruling.

**MusicBrainz curates them directly, so the resolve pipeline harvests them for free.** The `resolve_artist` walk already deep-fetches each candidate's `/artist/<mbid>?inc=url-rels`; adding `+aliases` brings the artist's aliases back on the SAME lookup (no extra MusicBrainz call), and `extractAliasesFromArtistData` folds them onto the matched candidate — skipping any alias that folds to the canonical name's slug, deduping by slug, and splitting a MusicBrainz "Search hint" (kept as `kind='hint'`, never rendered) from a real display name (`kind='name'`).

**Trust: `auto`, not `candidate`.** Unlike `label_aliases` (whose Apple×ISRC candidate is a fuzzy cross-source INFERENCE that awaits an operator glance), a MusicBrainz alias is a DIRECT statement of identity — the same authority that makes MB **socials** born `auto` (trusted/public) rather than `candidate`. So `artist_aliases.status` is `auto | confirmed` (no `candidate` tier): MB-harvested aliases are born `auto`, an operator-added alias is `confirmed`, and both feed the public graph. **Already-resolved artists pick this up automatically** — the self-healing worklist (`list_unresolved_artists`) re-queues an artist resolved to zero socials after 30 days, and any fresh resolve harvests aliases on its next pass; a re-resolve of an artist that already has socials is a genuine no-op only because it is not re-queued (aliases ride the same walk, so there is no separate backfill knob to add).

**Emit:** the public `/artist/<slug>` `MusicGroup` JSON-LD carries `alternateName` — one alias a scalar, several an array, omitted entirely when empty (byte-identical to before) — read from `getPublicArtistAliasNames` (`status in ('auto','confirmed') and kind='name'`), mirroring exactly how the label page emits `alternateName` from `label_aliases`. **Not yet threaded into the artist search FTS index:** the FTS5 index is built over `tracks` (title/artists/album/label) and synced by triggers on `tracks`, keyed on the tracks rowid; artist aliases live in a separate table keyed by `artist_id`, so folding them in needs new triggers on a different table plus a many-to-many fan-out join — not a small addition, deferred as a follow-up.

## The `/admin/artists` review queue (the operator motion)

`/admin/artists` is the stable MANAGE surface for every artist Fluncle features — one card per artist, name-sorted. The **review model**: review lands on the LINK, not the artist. A link "needs a look" while its `reviewed_at` is NULL — a fresh resolver insert, or a machine re-resolve that changed its URL (`artistNeedsLook` is now just `socials.some(s => s.reviewedAt === null)`). A **Fresh links** section at the top of the board lists exactly those unreviewed links, grouped by artist, each with **Approve** (`review_artist_social`: stamp `reviewed_at = now` AND promote a `candidate` to `confirmed`), **Edit**, and **Remove** — so a single new Twitch link surfaces on its own instead of re-flagging the whole already-reviewed artist. **Edit is inline**: the pencil turns the link text into an input IN PLACE (matched to the row's button band so the row never jumps), Approve becomes **Save**, and Save corrects the URL AND approves it in one act (`update_artist_social`) — the entered URL is validated + normalized against the row's platform on the server (reusing the resolver's `classifyMbUrl` + `normalizeProfileUrl`: a YouTube row rejects an `instagram.com` URL; a deep link collapses to the profile root, or is rejected with a quiet inline reason), then stored `source=operator`, `status=confirmed`, born reviewed (the operator just fixed it). Escape/blur restores the original. This is the daily fix for a resolver miss (a `music.youtube.com/search` page, a label's Bandcamp mistaken for the artist) without leaving the row. The per-artist **"Looks good"** (`review_artist`) stays as the BULK: it stamps every one of an artist's links reviewed at once and promotes any surviving candidates (reviewing IS the trust gate that lets a link onto the public page + `sameAs`). Add/remove a platform inline behind the **Manage links** dialog: a Shadcn `Select` (the platform logo next to `SelectValue`) + a URL `Input` → `add_artist_social` / `remove_artist_social` (an operator-entered link lands `source=operator`, `status=confirmed`, born reviewed). The WORK surfaces as an `/admin` attention row (source `artist-review`, counting UNREVIEWED links per artist) that deep-links here with `?artist=<id>`, auto-expanding that artist.

### The board's automated-socials cell

The findings board's old **LFM** cell is repurposed into an **automated-socials** aggregate showing the Last.fm love (`done` once the backfill RAN). A `HoverCard` reveals the breakdown. There is no per-track artist column — the identity graph is per-artist, owned by the `/admin/artists` queue.

## Why there is no follow

Fluncle once championed the artists it featured — following them back on Spotify and YouTube, first as an on-box auto-sweep, then as a per-row operator tap. Both are gone. This section is the record, so the decision is not relitigated.

**Spotify was never possible.** Its artist-follow endpoint (`PUT /me/following?type=artist`, scope `user-follow-modify`) 403s for the app, and it is provably not our side: with the exact same token a `playlist-modify-public` write returns 200 while the artist-follow 403s — so it is neither scope, nor account allow-list, nor Premium (verified 2026-07-07, after a full remove-app + re-auth and even a Premium upgrade). It is the **Development-mode endpoint gate**. The only lift is Extended Quota Mode, which since 2025-05-15 is **org-only** (≥250k MAU, a registered business entity) and unreachable for Fluncle.

**YouTube worked, but not well enough to keep.** With Spotify gated, YouTube was the sole working platform, and its batched follows mostly failed on quota — a flaky sweep buying a marginal upside. So the motion was wound down rather than propped up:

- **2026-07-08 — the auto half.** The YouTube-only `follow_artist` sweep, its `fluncle-artist-follow` box timer, and the `cron.artist-follow` registry / `/status` surface. A `RETIRED_SERVICE_IDS` tombstone in `apps/web/src/lib/server/status.ts` still suppresses the stale `service_status` row the box's old image keeps upserting; it goes when that row is dropped.
- **2026-07-09 — the manual half.** The operator Follow-now / Undo, the `follow_artist_social` / `unfollow_artist_social` / `record_operator_follow` ops, the Spotify `user-follow-modify` scope, the YouTube subscribe helpers, and the `followed_at` / `muted_at` columns. **`mute` went with them** — muting meant "don't follow this platform", so once nothing followed, it had nothing left to mean.

Reverse this only if the gate lifts — Spotify reopening broader Web API access to dev-mode apps, or a Fluncle business entity qualifying for Extended Quota. Nothing depends on it.

## Remixer credits (the `track_artists.role` column)

The `track_artists` join gains a nullable **`role`** (null = a performer — a lead/featured artist, the only kind v1 had; `'remixer'` the first non-null value). It is DERIVED, never guessed. `deriveRemixerNames` (`track-match.ts`) reads the title's version descriptor — "(Calibre Remix)", "Song - S.P.Y VIP" — strips the version word off it, and returns only the credited `artists` it EXACTLY folds to. So an uncertified remixer with no `artists` row is never invented (there is no row to stamp), and a co-remix ("Calibre & Fabio Remix") resolves to BOTH when both are credited. It is the SINGLE source of truth: the DB stamp and the JSON-LD emit both read it, so the column and the markup agree by construction.

`stampRemixerRoles` (`artists.ts`) stamps `role='remixer'` **fill-empty-only** at every write path that mints a `track_artists` edge — the publish path (`upsertTrackArtists`), the crawler's name-fold link (`linkTracksToArtistEntities`), and the Spotify-anchor step (`connectAnchorArtists`) — and `scripts/backfill-remixer-roles.ts` (deploy `db:backfill`) catches history up, keyset-paged over the titles that plausibly carry a descriptor. Idempotent and non-clobbering.

**Emitted, markup-only.** On the `/log` `MusicRecording` and the graph-page tracklists a remixer renders as a schema.org `contributor` **Role** (the roles.html pattern: `contributor` → a `Role` carrying `roleName: "remixer"` and REPEATING `contributor` for the `MusicGroup`, with its `@id` when the remixer is a known artist). Additive to `byArtist`; the visible page is unchanged (credits are markup-only for now).
