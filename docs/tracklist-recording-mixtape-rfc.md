## 0. Summary / the reframe

- **The primitive is the _recording_ cue — but a mixtape's published tracklist is a separate, frozen thing.** My draft's "one cue table for everything" was **falsely coupled** [panel: data-model S1]. A recording's cues are _mutable working state_; a published mixtape's tracklist is _immutable canon_. Merging them lets editing a promoted take silently rewrite the published mixtape and re-point every live clip's coordinate — destroying immutability the current code carefully guards (`assertDraftMixtape`, `recorded_at_immutable`, the `setMixtapeCues` exact-match backstop). So: **`recording_cues` (one owner, mutable) for plans+takes; keep `mixtape_tracks` (frozen) for the published tracklist; `promote` COPIES cues into it.** They look alike but serve different consumers — decouple.
- **Two tables, three concepts** [panel: taste #1, brand-voice H2]. A **plan** is just _a recording with no video and untimed cues_; a **take** is _a recording with video and timed cues_. Collapse them into one `recordings` table with a `parentId` (take → plan) — no separate "Tracklist" object (which also collided with "tracklist," the breadcrumb word). **Mixtape** stays its own table (it earns the coordinate mint + immutability). Concepts stay three (plan / take / mixtape); tables are two + `mixtapes`.
- **This is mostly connective work on shipped machinery.** The recording primitive (#239/#240/#244) already gives the Studio, clip library, `presign_recording_upload` (no web caller yet), `promote_recording` (idempotent), and the clip→finding algorithm `resolveClipTracks` (`packages/contracts/src/util.ts:235`). Net-new: `recording_cues` + the finding link, the plan lifecycle on `recordings`, the Rekordbox derivation script, the plan→tools export, and a right-sized migration.
- **The load-bearing rules hold:** coordinate-purity (`.F.` mints only at publish, date-based sequential, no reservation — the drift bug dies by _deleting_ the prediction); a clip links only findings its window actually overlaps (no misattribution), and its coordinate line is **frozen into the stored caption at publish** [panel: taste #5].
- **Decomposition:** the truly-coupled spine is `recordings` + `recording_cues` + the migration. The Rekordbox derivation, the plan export, the slug, and the clip-caption work are independent units that parallelize once the schema lands.

## 1. Context & goals

Today "mixtape" is one row that is draft, plan, published artifact, and take-link at once; the draft carries publish-time cruft; the reserved Log ID **drifts** with the date (operator re-recorded Mixtape #2 twice, renaming Beatport/Rekordbox/USB each time); and a recording reaches a mixtape only via `promoteRecording`, which always mints anew and — confirmed live — resolves hand-authored cues to _zero_ findings (`getTrackByIdOrLogId(cue.id)` where `cue.id` is a random UUID → `no_resolvable_members`).

**Goals (all in reach):** compose a plan with a stable handle; capture takes, each with its real played tracklist auto-derived from Rekordbox; clip any take, every clip honestly linked to canon; publish one winning take. **Honest scope:** whether search/AI engines index the emitted coordinates is theirs, not a ship gate. Given the tiny dataset (~2 mixtapes, 1–2 recordings, a few drafts, 3 clips), the migration is _careful but small_ [panel: taste #3] — the split earns a real migration because it removes four standing debts + unblocks the workflow, not for schema elegance.

## 2. Data model (the truly-coupled spine)

Drizzle, `apps/web/src/db/schema.ts`; repo convention: plain-text ids, **no declared FKs** (cascades in `db.batch`), but **do add `check()` constraints** — the panel confirmed the repo has none and the ownership invariant must be structural [panel: data-model S2]. All DDL via `bun run --cwd apps/web db:generate`.

- **`recordings`** [plan|take] — existing table + `parentId` (nullable: a take → its plan; NULL for a plan or an orphan take like the rolling set) + `version int` (a **display label** only, see §3) + make **`r2Key` nullable** (a plan has no video; "has video" = `r2Key IS NOT NULL`, an explicit signal) [panel: taste #1]. Indexes: `parent_id`, unique `(parent_id, version)` (NULLs distinct → orphan takes coexist — verified SQLite semantics). `tracklist_json` retired after backfill.
- **`recording_cues`** (NEW — the recording-side unified cue): `id` PK (stable clip-overlay ref), `recordingId` **NOT NULL** (one owner — no XOR), `findingId` (nullable — NULL for a Rekordbox track that isn't a finding), `artistsText`/`titleText` (snapshot: clip overlay + non-finding survival), `position` NOT NULL, `startMs` (NULL until marked), timestamps. Unique `(recording_id, position)`; index `finding_id`. Replaces `recordings.tracklist_json` **only**.
- **`mixtapes`** — unchanged mint (`publishMixtape` CTE, `sequence_number`-unique, `.F.` tail, ≤54). `recordingId` → **unique** (one mixtape per take). `status` → `["distributing","published"]` (TS-narrow only; migrate all `draft` rows out first). `planned_for` **moves to the plan-recording**, not dropped blindly (see §6/§9 — `/calendar.ics` reads it) [panel: taste #6].
- **`mixtape_tracks`** — **KEPT** as the frozen published tracklist (this is the S1 fix). Add nullable `finding_id` (rename of the NOT-NULL `track_id`) + `artistsText`/`titleText` snapshot so a published mixtape can carry a **non-finding** live-added track as a plain unlinked row [panel: DJ-workflow R7]. `promote` copies the winning take's `recording_cues` → `mixtape_tracks` (today's `setMixtapeMembers` path); immutability guards stay exactly as they are.
- **`mixtape_clips`** — `mixtapeId` nullable (retire the `''` sentinel); `recordingId` → target NOT NULL. A clip's findings are **derived** from `recording_cues(recording_id)` overlapping `[inMs,outMs]`; the coordinate line is **frozen into `caption` at publish** (no persisted-link column, no cache to drift) [panel: taste #5, data-model S7].

**Ownership invariant:** `recording_cues` has a single NOT-NULL owner (no XOR to enforce). The published copy lives in `mixtape_tracks`. This is the whole S1/S2 resolution: mutable cues and frozen tracklist are _different tables for different consumers_.

**Guards** [panel: staff-eng SF-5/SF-6]: `promote_recording` must **claim the link before minting** — atomically insert `recording_id` (or a per-recording lock), _then_ mint. The current check-then-mint path + a `unique(recording_id)` only errors on the _second_ update, i.e. **after** a scarce coordinate is already burned (unrecoverable on the 54-cap spine). `deleteRecording` (today `recordings.ts:224` does neither) must **block** if a `distributing|published` mixtape links it and **cascade** `recording_cues` + `mixtape_clips`. `deleteMixtape`'s draft-only guard becomes dead code once drafts are plans — repoint or remove.

## 3. Versioning (plan = v0 conceptually; takes = v1…vN)

- A **plan** is the parent recording; **takes** are child recordings (`parentId`). `version` is a **human label** ("v2" in the UI), assigned by an **atomic** `INSERT … SELECT coalesce(max(version),0)+1 … WHERE parent_id IS ?` (mirror the mint CTE — a TS read-then-insert races) [panel: data-model S6].
- **No append-only / immutability tax** [panel: taste #2, DJ-workflow R6]. Editing a cue edits the cue. Take-diffing is _already_ free (takes are separate rows with separate cue sets) — build the diff UI if/when wanted; impose no storage constraint for it now.
- **Cement** = `promote` copies the winning take's cues into `mixtape_tracks` (frozen there). "Base = winner" is `mixtapes.recording_id`.
- Take labels ("v1/v2/v3") can also fall out of `created_at` order; the `version` column just makes them stable/explicit. **Decision D-version** below.

## 4. Rekordbox auto-derivation + finding matching

- **Take cues come ONLY from Rekordbox** — drop "pre-seed the take from the plan" [panel: DJ-workflow R2]. Two writers to one cue-set at the divergence seam is the bug the RFC exists to prevent; the take's authoritative order _is_ Rekordbox. The plan's findings are used **only to resolve `finding_id`** during derivation: played+on-plan → link; played+off-plan → match the full catalogue, else NULL; planned+never-played → correctly absent. Divergence (drop/reorder/add-live) works by construction.
- **~90% reuse:** order+identity from `rekordbox-tracklist.py` (`session_tracklist` over the live `master.db` `djmdSongHistory` by `TrackNo` — add `Content.Key.ScaleName` + `BPM/100`); matching from `key_backfill.py` (`_fold`, `_normalize_artists`, `_split_title` remix-guard, `match_key`, `reconcile`'s 3-bucket model). Net-new ≈ 40 lines: fetch the catalogue (`fluncle admin tracks list --json`), invert the index, emit cues, and write them via a **new `replace_recording_cues` op** (operator-tier — inserts/replaces `recording_cues` rows) — **not** `tracklist_json`, which is retired [panel: staff-eng SB-2: the draft's "reuse `update_recording` with a `tracklistJson` body" contradicts the retirement and would misdirect the build].
- **Never auto-create findings** (findings are Spotify-backed). Unmatched → `finding_id=NULL` + snapshot text (renders, flagged). Ambiguous → flagged. Auto-prune: collapse _consecutive_ same-identity rows (re-load); keep non-consecutive repeats flagged.
- **Delete `getTrackByIdOrLogId(cue.id)`** in `resolveTracklistMembers` → read `cue.finding_id` directly [panel: data-model S4]. Adding the column without repointing the resolver leaves the trap live.
- **Source = live `master.db`, not the `.txt`** (stable ids, key, BPM, `TrackNo` order). Operator-Mac script (Rekordbox lives on the Mac); the CLI stays a thin HTTP client.
- **Cue timing:** Rekordbox gives order+identity+key, **not** mix-in start (load-time ≠ mix-in). Derivation writes the rows + order; the operator marks `startMs` in the Studio (`C`/`X`/↑/↓ loop, unchanged).

## 5. Clip → canon (honest, never misattributed)

- **`resolveClipTracks`** (`util.ts:235`) already returns single-finding / blend (window straddles ≥2 cues) / end-clamped. A recording's cues feed it with a small shim — the cue stores `artistsText` (a single string) but `ClipTrackInput.artists` is `string[]`, so `.split(", ")` at the boundary, not a straight pass-in [panel: staff-eng N-8]. **Guard the nullable `setDurationMs`** (`recordings.durationMs` is nullable → `Math.max(setDurationMs,outMs)` = NaN) [panel: data-model S7].
- **Only findings the window overlaps** — never resolve to the _nearest_ finding outside the window [panel: DJ-workflow R4]. A hype/transition clip cut before that section is cued must not borrow an unrelated track's coordinate; honest silence beats misattribution.
- **`buildClipCaption`** (new `apps/web/src/lib/server/clip-caption.ts`): caption stored clean; at **publish**, the coordinate line is baked into the stored caption string (frozen) — promoted → one `fluncle://<mixtape .F.>`; else the covered findings' `fluncle://<logId>` line(s). Before publish, re-derive live for display. This resolves the persist-vs-rederive debate: freeze the _artifact_ (caption), not the derivation.
- **Never-orphan (RESOLVED: block until cued)** [panel: DJ-workflow R1/R4, data-model S7; operator's "everything links to canon"]: **clip creation is blocked until ≥1 cue overlaps the clip window** (the Studio already gates re-sync this way) — the hard guarantee that every clip carries a real coordinate. A clip then links every finding under its window (blend → multiple lines), plus the mixtape's `.F.` once published. This deliberately reverses the shipped recording-primitive "un-promoted clips point home" behavior (the operator chose "link the tracks played"; blocking ensures there's always something honest to link).

## 9. The migration (backfill FOLDED into the deploy)

The deploy ordering is the real risk [panel: staff-eng SB-1]: `deploy:cf` = `db:migrate && wrangler deploy` runs **DDL only**; the backfill is a manual script with **no merge signal**, and every push to `main` auto-deploys — so a cutover deploy can land _before_ the operator runs the backfill and blank the published `/log` pages (a silent public data-disappearance, not a 500). "Space the merges" does not fix this. **Fix: fold the idempotent backfill into the deploy** — add a `db:backfill` step to `deploy:cf` (after `db:migrate`, before `wrangler deploy`) that runs the guarded (`where not exists`) Phase-2 script every deploy (a no-op once done), so DDL + backfill ship atomically and cutover reads are guaranteed populated. Keep **dual-read** (a reader falls back to the legacy column when a row's cues are empty) as belt-and-suspenders.

Two mechanism corrections the panel verified by running `db:generate`:

- **(SF-3)** drizzle-turso emits a libSQL `ALTER COLUMN` wrapped in a **full-DB index drop/recreate (~39 indexes)** per rebuild — _not_ a create-copy-drop-rename. Data + indexes survive, but the blast radius is DB-wide, so **consolidate every nullable/drop into ONE cleanup migration** to churn the index set once (not once per phase).
- **(SF-4)** narrowing `mixtapes.status` emits **zero DDL** (the enum is compile-time only; the column is plain `TEXT`, no CHECK anywhere in the repo). It's not a migration step — instead the backfill must drain **all** `draft` rows and cutover code must stop writing `draft` **before** the TS type narrows, or a stray `draft` becomes a runtime type-lie.

- **Deploy 1 — additive DDL + folded backfill + dual-read.** DDL: create `recording_cues` (+ `check(recording_id IS NOT NULL)`); add `recordings.parent_id`/`version`, `r2_key` nullable; add `mixtape_tracks.finding_id`(nullable)/`artists_text`/`title_text`; `mixtape_clips.mixtape_id` nullable. The `db:backfill` step runs in the same deploy. Readers dual-read (new table → fallback legacy).
- **Backfill (idempotent TS, run once post-Deploy-1, every step `where not exists`):**
  1. Drafts → **plan recordings** (`r2_key=NULL`, generated/`slugify` handle, `planned_for`, `note`); copy their `mixtape_tracks` → `recording_cues(recording_id=plan, finding_id=track_id, position, snapshot from tracks)`.
  2. Published/distributing mixtapes: reuse #1's existing `recording_id=38a4cb58…`; synthesize a take-recording for others. **Seed #1's `mixtape_tracks` stays untouched** (it already holds exact `track_id` links — do NOT rebuild it from `tracklist_json`, which discarded `track_id`) [panel: data-model S3]. Only _recordings_ get `recording_cues`.
  3. `recordings.tracklist_json` (incl. the empty rolling set `f8f555d2…`) → `recording_cues`, resolving `finding_id` by **normalized title+artist** (the `key_backfill.py` matcher), never by `cue.id`; unresolved → NULL + snapshot [panel: data-model S4].
  4. Normalize the 3 clips — they carry **both** `mixtape_id` and `recording_id` today (the 019 backfill set `recording_id` without nulling `mixtape_id`) [panel: data-model S5]: `set mixtape_id = NULL where recording_id is not null`; also clear the `''` sentinel.
- **Deploy 2 — cutover + ONE consolidated cleanup migration.** Repoint all consumers (below) to `recording_cues` / `mixtape_tracks`; `createClip` stops writing the sentinel; then a **single** migration drops `recordings.tracklist_json` + `mixtapes.planned_for` (after re-pointing `/calendar.ics`+`getUpcoming` to the plan) + `mixtape_clips.mixtape_id` (one DB-wide index churn, per SF-3). `status` narrowing is TS-only (no DDL) and safe once all drafts are drained.
- **Consumers to repoint before dropping `mixtape_tracks`/`tracklist_json`** [panel: data-model S8 — KEEP `mixtape_tracks`, so most stay]: `getTracksForMixtape` (`tracks.ts:341`, INNER JOIN → must LEFT JOIN + carry snapshot for non-finding rows), `listMixtapeMembershipsForTracks`, `setMixtape{Members,Cues,Cue}`, `addTracksToMixtape`, `deleteMixtape`, the `member_count` subquery (`mixtapes.ts:892`), `getRecording`/`RECORDING_SELECT`/`parseTracklist`, the CLI `recordings.ts`, `studio.$recordingId.tsx`, `getUpcoming`+`/calendar.ics`.

## Decisions — RESOLVED with the operator (2026-07-02)

1. **D-collapse → TWO tables.** A plan is a videoless `recordings` row (`parent_id` NULL); a take is a `recordings` row with video + `parent_id` → its plan; `mixtapes` stays separate. Three concepts, two tables. No `tracklists` table.
2. **D-handle → the auto Galaxy-vocab slug IS the handle** (not operator-typed). Curated allow-list pool + the load-bearing exclusion test (§6) — `banger`/`tune`/`sector`/`eclipse`/`void` + the standard banned set are hard-excluded.
3. **D-orphan → BLOCK clip creation until ≥1 overlapping cue.** Hard never-orphan guarantee; deliberately reverses the shipped "un-promoted clips point home."
4. **D-version → keep a stored `version` int** (stable, explicit labels; ~free), assigned by the atomic CTE.
5. **D-export → Beatport search-list first**, then Rekordbox `.m3u8`/XML, then a plain checklist.
6. **D-tail → keep base-6 `A–F`** (cap 54 is years out).
7. **D-plannedFor → moves to the plan, stays wired** to `/calendar.ics` + `getUpcoming` (repoint, don't drop).

All seven are settled — the RFC is build-ready with no open forks.
