# Newsletter: Loops → Resend + Spinup → Hermes (brief)

**Status: scoping, not built.** Move the weekly newsletter draft compute off Spinup and onto the Hermes box as an **agent cron**, swap the send provider from **Loops → Resend**, and gate the send behind a **Discord interactive button** (the Hermes `clarify` tool) on a **Friday 15:00 Europe/Amsterdam, DST-aware** schedule. This is the Hermes-side execution of two already-ratified threads: the provider/archive swap in [docs/rfcs/newsletter-own-the-stack.md](./rfcs/newsletter-own-the-stack.md) (the `editions` table + `/newsletter` archive + the Resend Broadcast send) and the agent-drafts/operator-sends tiering in [docs/hermes-automation-brief.md](./hermes-automation-brief.md) § Newsletter. It mirrors the cron pattern proven in [docs/spinup-to-hermes-enrichment-brief.md](./spinup-to-hermes-enrichment-brief.md). Per `AGENTS.md`, a `*-brief.md` is non-canonical planning; the codebase and canon win on any conflict, and this file gets deleted once the migration ships.

> **Relationship to the RFC (read this first).** The RFC owns the **server build**: the `editions` table, the `list_editions` / `get_edition` / `create_edition` / `update_edition` / `send_edition` oRPC contracts, the Resend Broadcast send path, the subscribe repoint, and the `/newsletter` archive. This brief does **not** re-specify any of that — it consumes it. What this brief owns is the **Hermes side**: which compute moves off Spinup, that the draft is an **agent** cron (creative authoring, not `--no-agent`), the **DST-aware Friday 15:00** schedule and its timezone mechanics, the **`clarify` send gate** design (the part the automation brief left as "a Discord nudge"), and the **stale-doc cleanup**. Where the two overlap, the RFC is canonical for the server contracts; this brief is canonical for the cron + the send gate.

## Why

Three things move at once because they are one weekly job:

- **Spinup is the wrong home.** The newsletter agent today runs on Spinup (slug `fluncle-s-newsletter-97bwtd`), the last automation that still lives off-box now that enrichment is migrating to Hermes ([spinup-to-hermes-enrichment-brief.md](./spinup-to-hermes-enrichment-brief.md)). The newsletter draft is light (read an API window, author copy, persist a draft) and already runs `claude-sonnet-4.6`-class authoring — it belongs on the Hermes box, the single automation home, alongside the observation cron it most resembles.
- **Loops blocks auto-send; Resend doesn't.** Loops has no programmatic campaign-send (dashboard-only — CLI/SDK/API/docs all confirm), which is exactly why the current flow ends at "operator opens the Loops dashboard and taps Send." Resend has a real Broadcast send API (`POST /broadcasts/{id}/send`), so once the edition is persisted (the RFC), the send becomes one Worker-side HTTP call — and the operator gate can become a deliberate **button** instead of a dashboard chore.
- **The gate should be a button, not a nudge.** PRODUCT.md keeps publishing operator-controlled, and a send to the real list is publish-class — so the send must stay human-gated. The automation brief scoped that gate as "a Discord nudge" (the agent posts, the operator runs a command). Hermes' `clarify` tool renders **tappable buttons** and pauses the agent for the click — a strictly better gate: the operator taps **Send** in Discord, no command to remember.

## The core flip

- **Today:** the **Spinup newsletter agent** authors the edition, then `loops campaigns create` + `loops email-messages update` stages a Loops campaign draft; the operator reviews it in the Loops dashboard and presses **Send**. The window cutoff is parsed out of the Loops campaign name; only sent campaigns anchor the window.
- **After:** a **Hermes agent cron** (Sonnet + `copywriting-fluncle`) fires **Friday 15:00 Europe/Amsterdam**, authors the edition, persists it as a **draft edition** via `fluncle admin newsletter draft` (the RFC's `create_edition`), then calls `clarify("Send edition #N?", [Send, Hold])`. **Tap Send** → the agent calls `fluncle admin newsletter send` → the Worker creates + sends the Resend Broadcast and mints the edition `number`. **Miss the window** → the draft is already persisted in our DB (and in the `/newsletter` archive machinery as a draft), and the send is re-triggerable. Spinup and Loops both leave the loop.

## Decisions (locked)

- **Agent cron, NOT `--no-agent`.** The newsletter draft is creative authoring — it writes the intro, the per-track "why", the subject, the tidbits, all in the Email register through `copywriting-fluncle`. That needs the LLM (Sonnet), so it is an **`agent` job carrying a `prompt`**, the same shape as the observation cron. (Contrast: the enrichment cron is `--no-agent --script` — pure deterministic compute, zero tokens. The newsletter is the opposite end of that axis; it is the _most_ authoring-heavy cron of the set.)
- **Friday 15:00 Europe/Amsterdam, DST-aware** — the locked send moment. Hermes cron has **no timezone field** (verified against the upstream cron docs; schedules are relative / `every Nh` / a cron expression / an ISO timestamp, all evaluated against the box clock with no documented `TZ` parameter or DST handling). So DST-aware is solved at the schedule layer, not by a TZ option that doesn't exist. **Recommended approach: pin the box's system clock to `Europe/Amsterdam` and schedule a single local-time cron expression `0 15 * * 5`.** See [The DST schedule](#the-dst-schedule) for why this beats the UTC-with-guard and two-seasonal-entries alternatives.
- **The send stays operator-gated via a `clarify` button.** The agent never sends unprompted. `clarify` renders the buttons and **blocks the agent** until a tap (or timeout). The gate is **persist-first, then offer**: the draft edition is written to the DB _before_ the button appears, so a missed button never loses the authored work — only defers the send.
- **Persist-then-offer, draft survives a missed window.** Because `clarify` blocks only for `agent.clarify_timeout` (default **600s / 10 min** — far too short for "respond whenever"), the design must not depend on the operator being at the keyboard in that window. The edition is persisted as a **draft first**; the button is a convenience for an immediate send. Miss it → the draft is saved and re-triggerable (a re-offered button next tick, or `fluncle admin newsletter send` by hand). The send is **idempotent** on the edition id — re-offering never double-sends.
- **Secrets posture unchanged — the agent holds no vendor key.** `RESEND_API_KEY` (+ the audience/segment id, already set as `RESEND_SEGMENT_ID`) stays a **Worker secret**; the Broadcast create + send happens **Worker-side** inside `send_edition`. The Hermes box holds only its `agent`-scoped admin token — the same boundary as every other cron (the newsletter was historically the _last_ raw-vendor-key risk; this closes it). The agent calls `fluncle admin newsletter draft|send`; the Worker holds the key.
- **Skill installed the `copywriting-fluncle` way.** The authoring voice rides the host-mounted `~/.hermes/skills/copywriting-fluncle` (→ `/opt/data/skills/`), updatable without an image rebuild ([hermes-agent.md](./agents/hermes-agent.md) § Voice). The same skill the observation cron already loads — no new skill install, the box already has it.

## The DST schedule

The constraint: fire at **15:00 in Amsterdam** every Friday, correctly through the DST flip (CEST = UTC+2 in summer, CET = UTC+1 in winter). Hermes exposes no per-job timezone. Three honest options, with the recommendation:

1. **Pin the box clock to `Europe/Amsterdam` + a single local cron expr (RECOMMENDED).** Set the container/host timezone to `Europe/Amsterdam` (a `TZ=Europe/Amsterdam` env var on the `docker run`, or the host's `/etc/localtime`), then schedule `0 15 * * 5`. The OS tz database handles DST; "15:00 local" is always 15:00 Amsterdam, summer or winter, with **no app-level DST logic**. One cron entry, no seasonal edits, no guard. **Caveat to verify at build time:** confirm the cron scheduler evaluates expressions against the _box local clock_ (it has no TZ field, so it must read _some_ clock — almost certainly the system clock the process sees). A 60-second smoke test (set the box TZ, schedule a one-shot `0 <nextminute> * * *`, watch it fire) settles it before relying on it. **Watch-out:** changing the box TZ also shifts _every other_ cron (enrich/context/observation are intervals — `every 1h` — so unaffected; but any future absolute-time cron inherits Amsterdam local). Document the box as "Amsterdam-local time" once set.
2. **UTC cron expr + a DST guard in the prompt.** Schedule the UTC instant — `0 13 * * 5` (CEST/summer) — and have the agent's prompt **check whether Amsterdam is currently on CET**; if so (winter), 13:00 UTC is 14:00 Amsterdam (an hour early) so it would need a different entry. This collapses into needing two entries anyway, or a self-aware no-op/defer, which is fragile for a once-a-week job. **Not recommended** — it pushes timezone math into an LLM prompt that runs 52 times a year, the worst place for it.
3. **Two seasonal entries (`0 13 * * 5` + `0 14 * * 5`), each guarded to only fire in its season.** Summer entry fires 13:00 UTC (= 15:00 CEST) and no-ops in winter; winter entry fires 14:00 UTC (= 15:00 CET) and no-ops in summer. Correct, but it needs a season check to suppress the wrong one (else double-fire near the boundary), and it's two jobs to maintain. **Fallback only** if option 1's clock assumption proves wrong.

**Recommendation: option 1 — pin the box to `Europe/Amsterdam`, schedule `0 15 * * 5`.** It moves DST to the one component that already solves it perfectly (the OS tz database), keeps the cron entry trivial and self-documenting, and adds zero LLM-side timezone reasoning. The intervals (enrich/context/observation) are TZ-agnostic so the box-clock change is invisible to them; the only discipline is to write future absolute-time crons in Amsterdam local and note it.

## The clarify send gate

This is the piece the automation brief left as "a Discord nudge." `clarify` is the upgrade: it renders up to 4 tappable buttons (+ an "Other" free-type), **blocks the agent** until the operator taps one, and on no-answer-within-`clarify_timeout` **unblocks with a sentinel message** rather than hanging.

The flow, inside the Friday agent cron:

1. **Author + persist the draft first.** The agent writes the edition and calls `fluncle admin newsletter draft` (`create_edition`) → a **draft** row in `editions` (no `number` yet). This is the durable artifact; everything after it is convenience.
2. **Offer the button.** `clarify("Send edition #N? — <subject>, <track count> finds, window <since>..<until>", choices: ["Send", "Hold"])`. The message doubles as the operator's review summary (the one-minute read that the Friday Loops review used to be), with a link to the draft's archive preview if available.
3. **On `Send`** → the agent calls `fluncle admin newsletter send <edition-id>` (`send_edition`). The Worker renders the email HTML from `contentJson`, creates the Resend Broadcast (API-created, so it is API-sendable), sends it, and mints `number` + `sentAt` + `sendProvider: "resend"` + `sendExternalId`. The edition appears in the archive.
4. **On `Hold`** → the agent does nothing; the draft persists, re-offerable later.
5. **On timeout (the sentinel)** → the agent treats the missed window as `Hold` (do **not** auto-send — silence is not consent for a publish-class action), reports "drafted edition #N; send pending" to the channel, and exits. The draft is in the DB and re-triggerable.

**`clarify_timeout` posture.** Default is 600s (10 min). For a "respond whenever" send gate that is too short — but **do not lean on a long timeout as the mechanism**, because the persist-first design already makes a missed button harmless (the draft survives). Two coherent postures:

- **Keep the default 600s** and rely on persist-first + re-trigger. Simplest; the button is a 10-minute fast-path and anything slower falls back to a re-offer or the manual `fluncle admin newsletter send`. **Recommended.**
- **Bump `agent.clarify_timeout`** (e.g. to a few hours) _only if_ the operator wants the single live button to stay tappable across a longer Friday-afternoon window. **Caveat:** `clarify_timeout` is a **global** `agent.` config, not per-call — bumping it lengthens how long _every_ clarify (including chat) blocks the agent session, which can wedge the shared process. So prefer the default + persist-first; reach for a bump only if the re-offer path proves annoying in practice.

**Re-trigger path (the miss recovery).** Two options, pick at build time: (a) the next Friday tick re-offers the button for any unsent draft before authoring a new one (the window self-heal already folds the un-sent edition's tracks back in — see below), or (b) a tiny on-demand path — `fluncle admin newsletter send <id>` by hand, or a `/cron` one-shot that re-offers the button. Option (a) is the durable default; the un-sent draft is simply re-offered (and updated, never duplicated) on the next run, matching today's "update the stale draft" rule.

## Gotchas

- **`clarify_timeout` is global, not per-call.** It is an `agent.` config (default 600s); bumping it lengthens _every_ clarify block including chat, which can wedge the shared session. Prefer the default + persist-first over a long timeout. (Source: the upstream tools/config docs.)
- **No native cron timezone — DST is solved by the box clock, not a TZ field.** Verified against the upstream cron docs: no timezone parameter, no DST handling. The recommended fix (pin the box to `Europe/Amsterdam`) shifts the clock for the _whole_ scheduler — fine because the other crons are intervals, but any future absolute-time cron inherits Amsterdam local; document it.
- **Cron sessions are fresh and isolated.** Each tick is a brand-new agent session with no carried conversation, so the Friday prompt must be **fully self-contained** (restate the window logic, the voice rails, the persist-then-clarify flow) — exactly like the observation cron's self-contained prompt.
- **Agent cron, not `--no-agent`.** Easy to mis-pattern this after the enrichment brief's `--no-agent --script` job. The newsletter _authors copy_, so it carries a `prompt` and runs Sonnet — the opposite kind of job. A `--no-agent` newsletter would have no voice.
- **Persist BEFORE the button, always.** If the button is offered before the draft is saved, a missed tap or a session crash loses the authored edition. The draft row is the durable artifact; the button is convenience. (This is the whole reason the gate is persist-then-offer.)
- **The send is idempotent on the edition id; never auto-send on timeout.** A re-offered button must not double-send; the timeout sentinel is treated as `Hold`, not `Send` (silence ≠ consent for a publish-class action).
- **Resend Broadcasts: only API-created broadcasts are API-sendable** (RFC-verified vendor constraint). Satisfied because `send_edition` creates the broadcast via the API and then sends it — but if anyone ever creates the broadcast in the Resend dashboard, the API send will refuse it.
- **Window cutoff moved from a parsed string to a column.** The current doctrine parses the cutoff out of the Loops campaign name; the new flow reads `editions.windowUntil` from the last **sent** edition. The self-heal (a skipped/un-sent week widens the next window) is preserved — but it now keys off `status: "sent"`, not "a campaign exists," so an un-sent draft correctly re-enters the window.
- **Don't rewrite the LMX template in two places.** If the RFC's server build re-homes the template into `apps/web`, deleting the `docs/` copy, coordinate so this brief doesn't also rewrite the `docs/` file. One owner for the template.
- **Spinup teardown after, not before.** Same as the enrichment cutover: prove one good Friday edition from Hermes before deleting the Spinup agent, so the newsletter never has a gap.
