## 0. Summary / the reframe

- **The unifying simplification: the sim contract is the portable unit, and the terminal is a third adapter.** PR #7 already split the game into a pure sim core (`sim.ts` + `placement.ts` + `types.ts` — deterministic, no DOM/audio/`Date.now`/`Math.random` in the hot path) and platform adapters (`render.ts`, `audio.ts`, `input.ts`). The SSH variant preserves the same rules and projection contract, then adds one new adapter pair: a Bubble Tea **scope renderer** and a **terminal input layer**. Nothing about the entities, galaxy, or product feel is re-decided.
- **The stock-Javy WASM path is killed; default to a Go engine port unless Maurice explicitly buys the plugin cost.** Gate-0 spike 2 proved that stock `javy build` cannot support the RFC's FFI/state contract: WIT imports are rejected, and stock static/dynamic builds trap on repeated zero-arg export calls. The old "one literal TS source in QuickJS" path now requires a **custom Rust Javy plugin** (`initialize-runtime`) that installs host callbacks and owns a persistent runtime. That is no longer a contingency; it is the required cost of keeping JS-in-WASM. The recommended path is now a **small Go port behind the same `boot/tick/projection` contract**, synchronized by golden vectors. This is less flashy, but it is cheaper, easier to test, pure-Go, and avoids owning a custom Javy toolchain indefinitely.
- **The galaxy is identical across surfaces, and that part _is_ free.** `apps/ssh/main.go` already fetches the same public `/api/tracks` with the exact `GameTrack` fields, and `placeStars` is pure. Same tracks → the same star at `fluncle://241.7.3A`. The Log ID is pure integer `fnv1a` (bitwise-identical in any engine); derived float positions are visually identical (the cross-engine ULP difference is invisible). **Map knowledge is portable across surfaces** — the Log ID spine paying off, verified, at no extra cost.
- **The Minimum Lovable galaxy (name it, ship it):** the SSH sim engine + a top-down **scope renderer** + the **read-the-log orbit card with an OSC-8 Spotify link** + the **`scopeContacts`** hazard selector + **repeat-decay input** + the **menu entry** + the **in-register telemetry** + the **crew stat you return to on the menu** (free — it already renders). That ships the flex. **Fast-follow v2:** the scannable QR block, the Kitty-protocol input path, and ambient crew-blips — each gated by the dogfooding census.
- **Coupling tiers, lowest-risk-first:** Gate 0 (same-continent latency PASS; engine KILL recorded) → (A) the **SSH sim engine** (recommended Go port, parity-tested over the _existing_ public surface) → (B) the **scope renderer + orbit card + `scopeContacts`** → (C) **input + control tuning** → (D) **canon integration** (mostly independent copy + wiring, parallel from day one).
- **The make-or-break is input over SSH, and the first draft got it wrong.** Terminals deliver no key-release events; the naïve repeat-decay window _stutters at the start of every turn_ (the OS initial-repeat delay is a 250–500 ms dead zone). The fix is a hold-state machine that bridges that delay (Unit C), and the ROADMAP's "lonely operator reading instruments" becomes the design — but as _texture, not the emotional landing_ (Unit D resolves the cold-sci-fi trap).

## 2. Unit A — The SSH sim engine (the keystone)

**Direction:** implement the SSH-side sim engine behind the same `boot/tick/depart/reset/projection` contract the WASM bridge was meant to expose, with golden-vector parity against the web sim before any rendering leans on it. Recommended default: a focused Go port of the pure sim and placement logic.

**The path:** port the deterministic core (`sim.ts` + `placement.ts` + `types.ts`) to Go inside `apps/ssh` or a small internal package owned by `apps/ssh`, preserving the same public engine contract and using generated golden fixtures from the TypeScript sim as the authority. Rejected for now: AssemblyScript (a rewrite = a second source of truth) and stock Javy (Gate-0 KILL). Optional but no longer recommended: a custom Javy plugin path, which keeps one literal JS source but adds Rust plugin ownership, host-callback bindings, runtime lifecycle code, and CI/toolchain risk.

**The state contract (replanned after the engine-model KILL):**

- **Keep the same shape:** `boot(catalogue, seed)`, `tick(input, elapsed)`, `depart()`, `reset()`, and `projection()` operating on one per-session `SimState`. In Go this is normal object state, not a QuickJS module-global.
- **The lean projection out (per frame):** `ship.{x,y,heading,speed,boosting,fuel}`, `phase`, `collectedCount`, `stars.length`, `time`; selectors computed inside the engine — `nearestCarrier()`, `radarBlips()`, and `drainEvents()` (the `SimEvent[]` since last tick). A few hundred bytes as a Go struct that can still be JSON-serialized for tests. (`scopeContacts()` is **Unit B's** addition — not in A's parity surface; see re-scope below.)
- **`boot()` payload:** the full `GameTrack[]` catalogue — unbounded, ~75 KB at 500 tracks (fine, once per session). **This requires new SSH client code: the full paginated `/api/tracks` fetch loop** the web's `loadCatalogue()` does (the SSH app currently fetches only `?limit=16`/`?limit=1`). No new endpoint, but new pagination code.
- **The seed:** the **Go host generates the per-session seed** (`crypto/rand`) and passes it to the engine. Doesn't affect star/Log-ID determinism.
- **Sim-step rate — pin to `1/60`** (the panel: a parity-breaker otherwise). `stepSim` is dt-dependent (`heading += steer*TURN_RATE*dt`, `ease(…, dt*ACCEL)`); the web's `game.ts` uses `SIM_STEP = 1/60` with an accumulator. The SSH engine must accumulate real wall-time and call its ported `stepSim(state, input, 1/60)` in a `while` loop _exactly_ like `game.ts` — a 30 Hz step integrates to different float state and the golden vectors won't match.

**Build pipeline + sync:** add a small fixture generator on the web side that runs the TypeScript sim with fixed seeds, synthetic `GameTrack[]`, and scripted inputs, then writes JSON golden vectors consumed by Go tests. The SSH binary has no embedded wasm and no wazero dependency unless Maurice explicitly chooses the custom-plugin path. The build gate is the Go parity test against the current generated fixtures.

**Parity — two distinct tests (corrected; a single "tolerant hash" is not a thing):**

1. **Intra-engine exact-hash regression guard:** fixed seed + synthetic `GameTrack[]` + scripted `SimInput` → committed state-hash checkpoints, asserted in Vitest (V8). Catches web sim regressions and fixture drift.
2. **Cross-engine field-wise tolerance:** the Go test runs the ported engine and asserts `abs(diff) < 1e-6` on ship pose + star positions vs the V8 vectors, and asserts the **Log-ID strings bitwise-equal**. Never hash-compare floats across engines — they may differ by a last ULP by construction (that's the whole behavioral-not-bitwise point, decision #2).

**Re-scope (the panel: Unit A isn't self-contained as first drawn):** A's parity covers only the **existing public surface** — `createSim`/`stepSim`/`departOrbit`/`resetSim`/`drainEvents`/`nearestCarrier`/`radarBlips`. `scopeContacts` parity moves to Unit B. A still defines the input shape even though C tunes the values.

**Per-session cost:** one Go `SimState` per connection, no QuickJS linear memory. The killed Javy spike observed ~1.38–1.44 MB linear memory before trapping, but that path is not the default anymore.

**Done = both parity tests green**, black-hole/asteroid behaviors fire in the SSH engine, and no stock-Javy bridge remains in the critical path. Nothing visual yet.

## 3. Unit B — The scope renderer + the read-the-log orbit screen

**Direction:** a top-down tactical radar scope (the web's `drawRadar` _is already_ a 2D vector drawing of exactly this — verified: heading-up ship tick at center, `bx = cx + sin(bearing)·range·r`, `by = cy − cos(bearing)·range·r`, `range = distance/radarRange`). Not the behind-ship 2.5D (it depends on sub-cell positioning + per-pixel streaks/grain a cell grid can't carry).

- **The loop:** a **15 fps render tick** via `tea.Tick` (re-armed each frame; confirmed available, and there's no existing loop to conflict). **Sim-step decoupled at the pinned `1/60`** via a fixed-timestep accumulator (cap `min(acc,0.25)` against a spiral-of-death), mirroring `game.ts`. **Elm wrinkle (the panel):** the model is a value receiver — per-frame mutable flight state (accumulator, hold-window timestamps) must live behind a pointer / on the `app` struct, not the value-copied `model`, or it's lost between `Update`s. `View()` returns **`tea.View`** with `view.AltScreen = true` (the v2.0.7 API — _not_ a returned `string`); the **Cursed Renderer diffs cells**, but note the diff win is **largely forfeited for the animated scope region** (most cells change each frame) — it helps the static HUD; budget bandwidth against a _fully-animated_ scope (spike).
- **Scope layout** (≥ 80×24; below it, the Fluncle-voiced "scope needs more room" plate, reusing the existing `<30`/`<21` ladder): near-circular (cell aspect ~1:2 → horizontal radius ≈ 2× vertical), ship fixed at center heading-up, star/carrier/Earth blips, dim ring + sparse star-noise, and the HUD panels porting the web 1:1 as text (fuel bar, the `LOCK`/`carrier N%`/`scanning` signal readout, telemetry feed, `N/N bangers`, help line).
- **Glyphs — unambiguous-width ONLY for anything placed at a computed `(x,y)`** (the panel's decisive catch): most of the first draft's vocabulary (`▲ ● ⊕ ▒ █ ▀ ▄ ▁..▇ ·`) is **East-Asian-Ambiguous** — width-1 by default but **width-2 under EAW mode** (CJK locales, some tmux/PuTTY fonts), which shoves the whole scope row out of alignment. `displaywidth.String()` returns 1 for these → a false-safe test. **Use only the unambiguous set for positioned elements: `✦ ✗ ◦` + ASCII** (ship `^` or `◦`; star `◦`/`✦`; carrier `✦`; Earth a letter or `✗`-class; hazards `✗` + ASCII). **The width unit test must assert under `runewidth.NewCondition{EastAsianWidth:true}` (pessimistic), not `displaywidth`.** The block-ramp `▁..█` is allowed only in the left-anchored VU meter (not on the grid), measured pessimistically.
- **Pulse without flicker (the panel):** at 15 fps a 1-cell blip can't ease a sampled `sin(t*5)` — it strobes. Pulse via a **slow 2–3-frame glyph/brightness alternation** (`◦`↔`✦` every ~4 frames), not a continuous sine.
- **`scopeContacts(state)` — the one genuinely-new sim API (lives in `sim.ts`, parity-tested here, not A).** `radarBlips` returns only uncollected stars + Earth (verified, `sim.ts`), so black holes + asteroids are **invisible death** on an instrument view. `scopeContacts` returns hazards within radar range. **It must surface the black-hole _influence radius_ (`BLACKHOLE_INFLUENCE = 260`), not just a point** — the gravity pull acts before any collision, so a single point-blip still ambushes the player; render a faint ring/boundary glyph at the 260-unit edge so the pull is legible.
- **The orbit "listening moment" → a read-the-log card** (where the terminal beats the web): drop the orbiting-ship animation; render the logbook in **`renderDetail`'s established field-label register** (the deep-register model — `Coordinate:`/`Pressed by:`/`Reads as:`, deadpan, the uncle still present), `fluncle://<id>` in gold, `Artist — Title` in cream, a decorative frame-seeded VU over the block-ramp (clearly "signal visualization," not a fake meter), and the Spotify link as an **OSC-8 hyperlink** (`terminalLink()` already exists). `departOrbit()` on any keypress. **The link is cream, not gold (One Sun guard — gold is the carrier + Log ID only).** (QR block = fast-follow, see §0.)
- **Color:** map `palette.ts` → Lipgloss truecolor (the menu already does); `colorprofile` auto-degrades truecolor → 256 → 16 per cell — author in hex, never pre-quantize; gold/cream/red/coolBlue stay distinct after a 16-color downgrade. Skip literal scanlines (no alpha; darkening alternate rows hurts legibility); the Light-Years Rule survives as a dim dithered ring + sparse `·`-class noise in dead space, and most strongly as the medium itself (a 16-color session _is_ a lossier transmission — the deepest Depth-Gradient surface).

**Edge cases:** recompute scope center/radii from `m.width/m.height` each frame (never cache across resize; the v2 renderer debounces resize).

## 4. Unit C — Input + control tuning + the latency gate

**Direction:** solve the two terminal realities — no key-release events, and SSH latency — with a hold-state machine that _bridges the initial-repeat delay_, and lock the feel from the latency spike. Highest-risk unit.

- **The Bubble Tea v2.0.7 API (corrected — the first draft cited symbols that don't exist):** there is **no** `WithKeyboardEnhancements`/`WithKeyReleases` option and **no** `SupportsEnhancedKeyboard`. The real API: set **`view.KeyboardEnhancements.ReportEventTypes = true`** on the `tea.View` returned from `View()`; detect support by handling the **`KeyboardEnhancementsMsg`** at startup and calling **`.SupportsEventTypes()`**; repeats are exposed via **`Key.IsRepeat`**. Also: `teaHandler` currently passes **empty `[]tea.ProgramOption{}`** through the Wish middleware — the wiring must be added.
- **The no-key-up problem and the start-of-turn stutter (the panel's decisive game-feel catch).** A PTY sends only keypresses; a held key yields OS auto-repeat _after a measured ~500 ms initial delay on Maurice's iTerm setup_, then an ~83 ms repeat stream. The first draft's "150 ms decay window" would therefore **stall every turn**: press → turn ~150 ms → **dead zone until the repeat stream begins (~500 ms)** → turn resumes. A single decay window can't both bridge the initial delay and keep taps short. **Fix:** a hold-state machine — first press **begins steering**; steering ends only after a **550-600 ms idle timeout** with no further press/repeat. Release over-runs slightly (invisible over SSH); far better than stalling at the start. On terminals that report releases/`IsRepeat` (the fast-follow Kitty path), use those directly for exact hold. Maurice's iTerm census returned `SupportsEventTypes() = yes`, so the Kitty/event-types fast-follow is worth keeping.
- **SSH latency vs a flight game:** same-continent key→render echo passed with P95 under 200 ms from Amsterdam plus two nearby VPS paths. Slow everything anyway — **~0.9–1.1 rad/s turn** (from `TURN_RATE = 1.9`), **−30–40% speed** — so a 200 ms lag over-rotates only ~12°; non-lethal asteroids cover laggy input.
- **The fuel/range rebalance (the panel — a real correctness bug, not tuning):** `tuneConfig` derives `cruiseBurn = CRUISE_SPEED·TANK_CAPACITY/range`. Lowering `cruiseSpeed` with `range` unchanged means **you cover less distance per tank → the SSH game can become unwinnable (dry before the carrier).** Slowing speed **must** rebalance `range`/`refuelRate` in lockstep; ships with a number and a "reachable-frontier" test, not a hand-wave.
- **`SimConfig` promotion (decision #3) is a ~6-site refactor, not a one-liner.** `TURN_RATE`/`CRUISE_SPEED`/`BOOST_SPEED`/`ORBIT_SPEED` are module consts read directly in `stepSim` (`targetSpeed`, the heading update), `launchShip`, `departOrbit`, and the module-level `warpShip`, plus `tuneConfig` derives burn from `CRUISE_SPEED`. Promotion threads them through `tuneConfig` as inputs; the web passes today's values (relative-fuel tests survive), SSH passes slower ones **+ the rebalanced range**. It's **a change to production game-feel code**; use the latency PASS numbers as the starting point, then let dogfooding tune within that range. (Framing: this is medium adaptation like touch-zones/reduced-motion, one parameterized sim — but cop to the surface area.)
- **Two gate-zero spikes (run against the live host BEFORE any unit code):**
  1. **Latency spike (throwaway harness, no WASM/renderer): PASS for same-continent, 2026-06-12.** Amsterdam + kaulobot + dokploy all stayed under the same-continent RTT and key→render thresholds; animated 80×24/15 fps output stayed under ~30 KB/s; Maurice's iTerm reported `SupportsEventTypes() = yes`; initial repeat delay measured ~500 ms. Transatlantic remains unmeasured, so US users may still warrant a later "best on a fast connection" note.
  2. **Engine-model spike (no renderer): KILL, 2026-06-12.** Stock Javy rejects WIT imports and stock static/dynamic Javy builds trap on the second zero-arg export call. The WASM bridge is replanned out of the default path; a custom Javy plugin is now an explicit alternate, not a hidden dependency.
- **Esc (the panel):** the new `screenGalaxy` must intercept `esc` _before_ the shared `handleKey` (which treats esc/q/b/backspace as back-to-menu everywhere) — in-game `Esc` = pause-then-confirm-exit; `ctrl+c` = hard quit.

## 5. Unit D — Canon, copy, IA, and the architecture boundary

**Direction:** wire into the existing menu/screen pattern, write telemetry in the deepest register _from the shipped strings_, make audio-less a feature, hold the thin-client boundary. Mostly independent; parallel from day one.

- **Entry & IA.** The app _rejects shell commands_ (`rejectCommandsMiddleware` — `ssh … galaxy` is closed; leave that copy). Add **"Enter the Galaxy"** as the **first** menu item (the flex — greet, don't hide it), a new `screenGalaxy` reusing the Elm screen + back-stack. The crew stat already renders on the menu you return to — that _is_ the crew narrative, free.
- **Telemetry — START from the shipped strings; adapt only with justification (the brand panel's central correction).** The first draft _silently rewrote_ the canon strings and drained their warmth. Defaults are the shipped `game.ts` lines:
  - `logged` → **`Logged fluncle://<id>.`** (NOT `LOG ENTRY WRITTEN` — that's running copy repeating per-event → ALL-CAPS banned by VOICE.md §6).
  - `refuelled` → `Tank full.` · `low-fuel` → `Tank low.` (echo live %).
  - `adrift` → `Tank dry. Adrift.` · `towed` → `Recovered adrift. Towed home.`
  - `warped` → **`Pulled under. Flung across the galaxy.`** (keep — vivid, warm-feral; NOT "horizon breach"; may _append_ `Re-acquiring.`).
  - `asteroid-hit` → **`Hull hit. Fuel knocked loose.`** (keep — Garnish Rule; NOT "cell venting", which is banned cosplay).
  - `all-found` → `No carriers left in the sector. Home, junglist.` (the one warm kinship beat — `junglist` used **once**, here, exactly as shipped; don't scatter it).
  - `home` → optional new line `Earth re-acquired. Welcome back, junglist.` (note: the shipped game has _no_ `home` telemetry — don't claim it does).
  - `bolt-fired`/`bolt-hit` → **no telemetry** (cosmetic).
  - **Case rule (closes the first draft's "open decision #4"):** sentence case per VOICE.md §6 — _not_ the all-lowercase the first draft drifted into. The deep register comes from `renderDetail`'s logbook _field labels_, not from jargon or lowercase styling.
- **The carrier read** renders `nearestCarrier().strength` (0..1) as a climbing % + bar + bearing, locking at orbit: `carrier detected … 71%` → `LOCK`. **`carrier` is safe and load-bearing:** it's the shipped on-screen signal-readout label (`render.ts`), the ROADMAP's own phrase, and it _sidesteps_ the banned `signal`-as-identity (it names the banger you're homing on, not a transmission). **Guard:** never let an editor "fix" `carrier` → `signal` (banned).
- **`LOCK` and a one-time `FLUNCLE'S GALAXY` boot plate are the ONLY sanctioned caps** (instrument stamp / brand-mark plate, both pass VOICE.md §6's test). Everything else is sentence case.
- **Audio-less as flavor (the best-realized voice idea):** `recovered flight log · instruments only` intro; in orbit, _"the audio didn't survive the trip out here. it's still playing back on Earth:"_ + the link. **"Earth," not "the mothership"** — "the mothership" is the newsletter's proper noun; don't collide it. This is the Light-Years Rule (loss as a property of distance), not an apology.
- **The cold-sci-fi trap — resolved, and the resolution must reach the _lines_.** The ROADMAP's "lonely operator" collides with the Mosh Pit Rule + PRODUCT.md's "cold lonely-derelict cliché" ban. Resolution: lonely is _texture_, never the landing — and the **warmth lives in the telemetry itself** (the shipped strings already are warm-feral: "Pulled under," "Home, junglist"), not bolted on via crew-blips. Test every line: _would the uncle say this, half-shouting over a tune?_ ("Fuel knocked loose" yes; "cell venting" no).
- **Cross-surface determinism + boundary (verified):** same `/api/tracks` + `placeStars` → identical galaxy + Log IDs; **no new endpoint** (the fields are already parsed). The SSH app **stays a thin read-client** (AGENTS.md: web owns Spotify/Telegram/Turso/publishing) — the game adds one public read, no mutation, no auth, session-only.
- **Crew presence (fast-follow, not v1):** v1 _is_ the menu's existing "N crew aboard" stat you return to. Ambient on-scope blips are fast-follow (and need _fake_ per-session positions — a polar scope has no "positionless" blip). True networked positions are deferred (a presence channel that doesn't exist + the cross-surface question + it compounds the latency risk).
