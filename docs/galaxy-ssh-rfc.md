## 0. Summary / the reframe

- **The unifying simplification: the sim is the portable unit, and the terminal is a third adapter.** PR #7 already split the game into a pure sim core (`sim.ts` + `placement.ts` + `types.ts` â€” deterministic, no DOM/audio/`Date.now`/`Math.random` in the hot path) and platform adapters (`render.ts`, `audio.ts`, `input.ts`). The SSH variant reuses the **exact same sim** and adds one new adapter pair: a Bubble Tea **scope renderer** and a **terminal input layer**. Nothing about the rules, entities, or galaxy is re-decided.
- **One JS sim, two engines, via WASM â€” and the honest cost.** `sim.ts` bundles with esbuild and runs unmodified inside **Javy/QuickJS â†’ WASM under wazero** (pure-Go, no cgo). The web runs the TS in V8; SSH runs the _same source_ as wasm in QuickJS, so `fnv1a`/`mulberry32` reproduce bit-for-bit. **But be honest (the panel pressed this):** WASM is the chosen _craftsmanship flex at a higher build/deploy cost_, not the cheaper engineering option. The golden-vector parity harness this RFC commits to would sync a hand-written **Go port** equally well, the cross-engine trig caveat means _both_ paths land on _behavioral_ (not bitwise) parity anyway, and the portable surface is ~1,200 lines (sim 721 + placement 396 + types 80), not "~600." The FFI contract is deliberately **mechanism-agnostic**: if QuickJS-in-wasm ever proves too heavy, a Go port drops in behind the same `projection()` contract with no architecture change. We proceed with WASM because Maurice chose the flex and "one literal source of truth" is a real, defensible taste call â€” stated as what it is.
- **The galaxy is identical across surfaces, and that part _is_ free.** `apps/ssh/main.go` already fetches the same public `/api/tracks` with the exact `GameTrack` fields, and `placeStars` is pure. ## 2. Unit A â€” The WASM sim bridge (the keystone)

**Direction:** compile the existing pure sim to one `.wasm` and run it in the Go app via wazero, one JS implementation for both surfaces. Ships after Gate 0, with the parity tests green, before any rendering leans on it.

**The path:** esbuild-bundle `sim.ts` + `placement.ts` + `types.ts` (via a new thin `apps/web/src/game/wasm-entry.ts`) â†’ **Javy** (QuickJS-based) â†’ `galaxy-sim.wasm` â†’ **wazero** (`github.com/tetratelabs/wazero`, the one new Go dep). **This is net-new tooling** â€” `turbo.json` has no `build:galaxy-wasm`, and esbuild/javy are not yet deps (greenfield, not a small add). Rejected: AssemblyScript (a rewrite = a second source of truth) and â€” as the _cheaper-but-not-chosen_ baseline â€” a Go port (same golden vectors sync it; documented as the leaner alternative behind the same contract).

**The FFI / state contract (corrected â€” the first draft was self-contradictory):**

- **All wasm exports are `func()` â€” zero-argument, no-return** (Javy's hard constraint, verified). Data does **not** cross as wasm params/returns. Inputs cross _in_ via **host-import callbacks the guest calls** (`readBoot()` â†’ the catalogue+seed bytes the host staged, `readInput()` â†’ quantized steer int + boost/fire bits); the projection crosses _out_ via a host-import the guest calls (`writeProjection(ptr,len)`), which the Go side reads with `mod.Memory().Read` + `json.Unmarshal`. Hostâ†’guest byte staging uses the module's exported `cabi_realloc` (host allocs, writes, signals length).
- **The exports:** `boot()`, `tick()`, `depart()`, `reset()`, `projection()` â€” each `func()`, operating on a **QuickJS module-global `SimState`**.
- **The unproven assumption, now a Gate-0 spike (the panel's top catch):** holding `SimState` in a module global _across many `tick()` calls_ requires the **reactor / persisted-global model**, which is **not** a flag on stock `javy build` â€” it needs a custom Javy plugin (Rust, `initialize-runtime`) or live proof the stock dynamic-linking path persists globals between invocations. **Gate 0, spike 2 proves this first** (`boot()` sets state â†’ two `tick()`s increment â†’ `projection()` reads N=2); if it fails on stock Javy, the unit grows by a custom-plugin sub-project â€” so it gates the whole approach, weighted equal to the latency spike.
- **The lean projection out (per frame):** `ship.{x,y,heading,speed,boosting,fuel}`, `phase`, `collectedCount`, `stars.length`, `time`; the selectors computed _inside_ wasm â€” `nearestCarrier()`, `radarBlips()`, and `drainEvents()` (the `SimEvent[]` since last tick). A few hundred bytes of JSON. (`scopeContacts()` is **Unit B's** addition â€” not in A's parity surface; see re-scope below.)
- **`boot()` payload:** the full `GameTrack[]` catalogue as UTF-8 JSON, staged via `cabi_realloc` â€” unbounded, ~75 KB at 500 tracks (fine, once per session). **This requires new SSH client code: the full paginated `/api/tracks` fetch loop** the web's `loadCatalogue()` does (the SSH app currently fetches only `?limit=16`/`?limit=1`). No new endpoint, but new pagination code.
- **The seed:** the **Go host generates the per-session seed** (`crypto/rand`) and stages it for `readBoot()` â€” wasm has no `Math.random`. Doesn't affect star/Log-ID determinism.
- **Sim-step rate â€” pin to `1/60`** (the panel: a parity-breaker otherwise). `stepSim` is dt-dependent (`heading += steer*TURN_RATE*dt`, `ease(â€¦, dt*ACCEL)`); the web's `game.ts` uses `SIM_STEP = 1/60` with an accumulator. The wasm `tick()` must accumulate real wall-time and call `stepSim(state, input, 1/60)` in a `while` loop _exactly_ like `game.ts` â€” a 30 Hz step integrates to different float state and the golden vectors won't match.

**Build pipeline + sync:** one shared core (`wasm-entry.ts` imports the _same_ sim files the browser imports â€” guard: the `registerBehavior("blackhole")` module-load side-effect must survive esbuild tree-shaking; the golden test confirms warps fire in wasm). A turbo `build:galaxy-wasm` task runs esbuild â†’ `javy build` â†’ `apps/ssh/galaxy-sim.wasm`, `//go:embed`-ed; wazero **compiles once** at boot (thread-safe `CompiledModule` on the `app` struct) and **instantiates per session** in `teaHandler` (one `ModuleInstance` per connection, never shared across goroutines). The SSH binary build gates on a fresh wasm.

**Parity â€” two distinct tests (corrected; a single "tolerant hash" is not a thing):**

1. **Intra-engine exact-hash regression guard:** fixed seed + synthetic `GameTrack[]` + scripted `SimInput` â†’ committed state-hash checkpoints, asserted in Vitest (V8). Catches sim regressions and bundler/QuickJS-version drift.
2. **Cross-engine field-wise tolerance:** the Go test runs the wasm and asserts `abs(diff) < 1e-6` on ship pose + star positions vs the V8 vectors, and asserts the **Log-ID strings bitwise-equal**. Never hash-compare floats across V8/QuickJS â€” they differ by a last ULP by construction (that's the whole behavioral-not-bitwise point, decision #2).

**Re-scope (the panel: Unit A isn't self-contained as first drawn):** A's parity covers only the **existing public surface** â€” `createSim`/`stepSim`/`departOrbit`/`resetSim`/`drainEvents`/`nearestCarrier`/`radarBlips`. `scopeContacts` parity moves to Unit B. A still pulls in the input-FFI shape (so define the `readInput()` host-import in A even though C tunes the values).

**Per-session cost:** compile-once at boot (~seconds); ~1â€“4 MB QuickJS linear memory per instance (plausible, to be measured in the engine-model spike); 50 crew â‰ˆ 50â€“200 MB â€” fine on the VPS. CPU is microseconds/step in the interpreter.

**Done = both parity tests green**, the engine-model spike passed, black-hole/asteroid behaviors fire in wasm. Nothing visual yet.

## 4. Unit C â€” Input + control tuning + the latency gate

**Direction:** solve the two terminal realities â€” no key-release events, and SSH latency â€” with a hold-state machine that _bridges the initial-repeat delay_, and lock the feel from the latency spike. Highest-risk unit.

- **The Bubble Tea v2.0.7 API (corrected â€” the first draft cited symbols that don't exist):** there is **no** `WithKeyboardEnhancements`/`WithKeyReleases` option and **no** `SupportsEnhancedKeyboard`. The real API: set **`view.KeyboardEnhancements.ReportEventTypes = true`** on the `tea.View` returned from `View()`; detect support by handling the **`KeyboardEnhancementsMsg`** at startup and calling **`.SupportsEventTypes()`**; repeats are exposed via **`Key.IsRepeat`**. Also: `teaHandler` currently passes **empty `[]tea.ProgramOption{}`** through the Wish middleware â€” the wiring must be added.
- **The no-key-up problem and the start-of-turn stutter (the panel's decisive game-feel catch).** A PTY sends only keypresses; a held key yields OS auto-repeat _after a 250â€“500 ms initial delay_, then a ~30â€“40 ms stream. The first draft's "150 ms decay window" therefore **stalls every turn**: press â†’ turn ~150 ms â†’ **dead zone until the repeat stream begins (~250â€“500 ms)** â†’ turn resumes. A single decay window can't both bridge the initial delay (needs â‰¥~500 ms) and keep taps short (needs ~150 ms). **Fix:** a hold-state machine â€” first press **begins steering**; steering ends only after a **longer idle timeout (~400â€“500 ms)** with no further press/repeat. Release over-runs slightly (invisible over SSH); far better than stalling at the start. On terminals that report releases/`IsRepeat` (the fast-follow Kitty path), use those directly for exact hold. **The latency spike must measure the _initial_ repeat delay over SSH, not the inter-repeat interval** (the first draft measured the wrong gap).
- **SSH latency vs a flight game:** slow everything â€” **~0.9â€“1.1 rad/s turn** (from `TURN_RATE = 1.9`), **âˆ’30â€“40% speed** â€” so a 200 ms lag over-rotates only ~12آ°; non-lethal asteroids cover laggy input.
- **The fuel/range rebalance (the panel â€” a real correctness bug, not tuning):** `tuneConfig` derives `cruiseBurn = CRUISE_SPEEDآ·TANK_CAPACITY/range`. Lowering `cruiseSpeed` with `range` unchanged means **you cover less distance per tank â†’ the SSH game can become unwinnable (dry before the carrier).** Slowing speed **must** rebalance `range`/`refuelRate` in lockstep; ships with a number and a "reachable-frontier" test, not a hand-wave.
- **`SimConfig` promotion (decision #3) is a ~6-site refactor, not a one-liner.** `TURN_RATE`/`CRUISE_SPEED`/`BOOST_SPEED`/`ORBIT_SPEED` are module consts read directly in `stepSim` (`targetSpeed`, the heading update), `launchShip`, `departOrbit`, and the module-level `warpShip`, plus `tuneConfig` derives burn from `CRUISE_SPEED`. Promotion threads them through `tuneConfig` as inputs; the web passes today's values (relative-fuel tests survive), SSH passes slower ones **+ the rebalanced range**. It's **a change to production game-feel code**, finalizable only _after_ the latency spike's numbers â€” surface it as a risk, not a pre-handoff one-liner. (Framing: this is medium adaptation like touch-zones/reduced-motion, one parameterized sim â€” but cop to the surface area.)
- **Two gate-zero spikes (run against the live host BEFORE any unit code):**
  1. **Latency spike (throwaway Bubble Tea harness, no WASM/renderer):** from â‰¥3 vantage points â€” baseline RTT (< 120 ms same-continent / < 220 ms transatlantic; gate behind a "best on a fast connection" note above ~250 ms); keypressâ†’render echo (P95 < 200 ms controllable, > 350 ms KILL); **initial** auto-repeat delay over SSH (sets the hold-idle timeout); frame bandwidth at 15 fps with a _fully-animated_ scope (< ~30 KB/s); and the `SupportsEventTypes()` census (informs whether the Kitty fast-follow is worth it).
  2. **Engine-model spike (no renderer):** prove persisted QuickJS globals survive across `tick()` on stock Javy (bootâ†’tickâ†’tickâ†’projection = N). KILL-to-replan (custom plugin) if it fails.
- **Esc (the panel):** the new `screenGalaxy` must intercept `esc` _before_ the shared `handleKey` (which treats esc/q/b/backspace as back-to-menu everywhere) â€” in-game `Esc` = pause-then-confirm-exit; `ctrl+c` = hard quit.
