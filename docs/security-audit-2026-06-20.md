# Security Audit — Full Codebase (2026-06-20)

## Intro

This is a full-codebase security audit of the Fluncle monorepo: the web app (`apps/web`, TanStack Start on Cloudflare Workers + Turso/libSQL), the CLI (`apps/cli`), the public SSH terminal (`apps/ssh`), the authoritative DNS server (`apps/dns`), the Raycast extension, and the supporting packages.

**Open-source threat model.** This repository is public. Source, build config, and the entire git history are visible to attackers, so the audit specifically checked three things beyond ordinary review: (1) that no secret or credential is committed anywhere in the tree or in history; (2) that the security model never relies on code obscurity — every gate must hold with the source in hand; (3) that the public↔admin trust boundary is sound. The headline result on (1) is clean (see "Committed secrets" below), which is the single most important property for a public repo.

**Remediation is deliberately deferred.** This slice produces the audit only. No code was changed. Fixes are intentionally out of scope here because they would clash with other in-flight work that cuts across the same files. Each finding below carries a concrete next step for whoever picks up remediation.

**How this audit was run.** Two passes. First, a rigorous manual review, decomposed across nested sub-agents per area (web admin auth + the trust boundary; Turso/SQL + the DB layer; secrets + git history + supply-chain; R2/presign + outbound OAuth/integrations; the public unauthenticated HTTP surface; the Go SSH/DNS services + the CLI). Findings were deduplicated and severity-ranked centrally, and every cited `file:line` was read and confirmed to say what the finding claims — no finding rests on an unverified citation. Second, the task tool itself — **deepsec** (Vercel Labs) — was run against the highest-value areas using its local-subscription fallback (no paid AI Gateway key), and its findings were folded back in, dedupe-marked against the manual pass. See "deepsec run" below. The two passes are highly consistent: deepsec independently confirmed nine of the manual findings (one of which it also strengthened — the rate-limit key, #12) and surfaced thirteen genuinely new ones (D1–D13: mostly in the Go SSH/DNS services, plus a few it traced deeper in the web surface than the manual pass). It found no committed secret, corroborating the manual headline.

## deepsec run (subscription-backed, partial coverage)

The task tool is deepsec (`https://github.com/vercel-labs/deepsec`). It defaults to the Vercel AI Gateway (a paid key sized for full-repo concurrency), but its README documents a no-key fallback: "When running locally, `deepsec` falls back to your existing `claude` / `codex` subscription if you've logged in on this machine." This machine is logged into `claude`, so the tool ran on that subscription with **no AI Gateway key and no paid-infrastructure change** — `pnpm` (which `deepsec init`/`install` require, and which is otherwise off-limits in this repo) was used only because the documented tool requires it.

**It ran for real.** `npx deepsec init` → `pnpm install` (deepsec 2.0.12) → `pnpm deepsec scan` (≈110 regex matchers, no AI: 386 files scanned, 181 with candidates) → `pnpm deepsec process --agent claude` against the Claude Opus 4.8 subscription. A project `INFO.md` was written from this codebase's shape (auth primitives, the public↔admin boundary, the shared-Spotify-credential threat model, the known-benign dev fallbacks) so the scanner was project-aware; the investigating agent also read this audit doc during its run, so its output is already reconciled against the manual pass and it cites the manual finding numbers it confirms.

**Coverage is partial, by design.** The README warns subscriptions "generally don't have enough headroom for full repo scans," and the calibration batch confirmed it (≈$3/file of Opus-reported cost, several minutes per batch — a full 181-candidate `process` was neither affordable nor in-scope on a subscription). The run was therefore scoped to the highest-value areas the task named: the `apps/web` unauthenticated HTTP surface + admin auth (`mcp.ts`, `spotify.ts`, `newsletter.ts`, `mixcloud.ts`, `agent-discovery.ts`, `mixtapes.ts`, `env.ts`, `admin-auth.ts`, `public-auth.ts`, `submissions.ts`, `tracks.ts`, `r2-presign.ts`, `og.$logId.ts`, `mixtape-cover.ts`) and the Go SSH/DNS apps. **21 files were AI-investigated, yielding 24 deepsec findings (0 critical, 0 high, 16 medium, 8 "bug").** The Go files needed `process --files`(direct mode): deepsec's regex`scan`only activated its bun/node/JS matcher set (Go matchers stayed dormant), so the 17 Go files registered zero candidates and would never have been queued by a normal`process`— direct mode forced`apps/ssh/main.go`and all five`apps/dns/_.go` through AI investigation anyway. **Not covered:** the CLI (`apps/cli`), Raycast, the `packages/_`, the bulk of `apps/web`routes/components, and the migrations — those rest on the manual pass only. deepsec's`revalidate`(FP-cutting) pass was not run (subscription cost); a few of its raw regex candidates were noted as false positives inside the relevant findings (e.g. "insecure-crypto" flags on SQL string fragments in`tracks.ts`/`mixtapes.ts`, XSS flags on the rasterized OG output).

**Reconciliation.** Findings deepsec independently confirmed are marked **[deepsec ✓]** at the manual finding. New deepsec-only findings are appended as **D1–D13** and folded into the severity ranking. deepsec's "bug" class (a correctness/reliability defect with security relevance — broken audit logging, a TOCTOU delete, a refresh-token race) is mapped to Low/Info here unless its security impact warrants Medium. Every deepsec `file:line` cited below was independently re-read and confirmed before inclusion; nothing is represented beyond what the tool reported.

## Summary

**No Critical or High findings.** The codebase is unusually security-conscious: the public↔admin trust boundary is well-built, the database layer is uniformly parameterized (zero SQL-injection / mass-assignment exposure), and no live secret exists in the tree or history (deepsec's run independently surfaced no committed credential, corroborating the headline). The findings are Medium and lower — chiefly availability/cost hardening on the unauthenticated surface (now spanning both Go services, not just SSH), minor information disclosure, and defense-in-depth / correctness notes. The deepsec pass moved the needle in two places: it confirmed the SSH listener's DoS gap and found the **DNS listener has the analogous gap** (no rate-limiting/RRL, negatives never cached → 1:1 amplification onto the paid upstream), and it **strengthened the rate-limit-key finding** (the User-Agent component defeats the 5/hour cap _through_ Cloudflare, not only on direct-origin access). The DNS server's systemd unit remains exemplary; the server's query-handling, less so.

Manual findings deepsec independently confirmed are tagged **[deepsec ✓]**. Deepsec-only findings are **D1–D13** at the bottom of the table.

## Committed secrets — the open-source headline

**No live credential was found in the current tree or in any of the 282 commits of git history.** This was verified, not assumed. Method: `git grep <pattern> $(git rev-list --all)` across all reachable commits for every credential shape — Telegram bot token `[0-9]{8,10}:[A-Za-z0-9_-]{35}`, Discord webhook URLs, JWT `eyJ…` (Turso/auth), AWS `AKIA…`, `sk-`/`sk_`/`sk_agent_` keys, `BEGIN … PRIVATE KEY`, Firecrawl/Postiz/Loops/`client_secret=` literals — all returned **0 hits** after excluding `op://` references, `process.env` reads, and placeholders. A broad entropy sweep (every unique 40+ char token across all history in `.ts/.tsx/.go/.sh/.toml`) resolved to a closed set of 8 benign strings (Cloudflare's generated `worker-configuration.d.ts` doc-hashes, SQL index names, and the gated dev fallback in finding #14). A `--diff-filter=A` scan confirmed no `.env`, `.dev.vars`, `.pem`, `.key`, `id_rsa`, or `.db` blob was ever committed. `.dev.vars.tpl` uses `op://` 1Password references for every secret, and the wrangler `vars` block plus the two `VITE_*` values expose only non-secret public IDs/URLs.

This is the property that matters most for a public repo, and it holds.

## Findings

### 1. [MEDIUM] `/api/search` has no rate limit — unauthenticated Spotify-credential cost amplification

- **Where:** `apps/web/src/routes/api/search.ts:10-31` (handler); downstream `apps/web/src/lib/server/spotify.ts` (`searchTrackCandidates` → `getSpotifyAccessToken` + `/v1/search`). Also reachable unauthenticated via the MCP `search_tracks` tool (`apps/web/src/lib/server/mcp.ts:89-113`).
- **What:** `GET /api/search?q=` is fully unauthenticated with only a 2-character minimum (`search.ts:16-18`) and no per-IP limit, no result caching, and no Cloudflare Rate Limiting rule (`wrangler.jsonc` declares none). Every request calls Spotify through the operator's single shared app credential — the same `getSpotifyAccessToken` the admin add/publish flow depends on.
- **Why it matters:** An anonymous attacker can drive unbounded volume through one shared credential, exhausting Spotify's rate limit and causing 429s that break the legitimate admin add+publish flow. Pure cost/availability amplification with a single anonymous GET; source is public, so the endpoint and its shared-credential design are known.
- **Next step:** Apply the same DB-backed per-connection limiter used for submissions (`submissions.ts`) — or a Cloudflare Rate Limiting rule — to `/api/search` and the MCP `search_tracks` path, and consider short-TTL caching of identical queries.
- **deepsec ✓:** Independently confirmed (high confidence) at `mcp.ts:92,99`. It traced the same chain and added that `submit_track` (`mcp.ts:118,125`) calls `searchTrackCandidates` _before_ `createSubmission` enforces its 5/hour cap, so every MCP `submit_track` fires a Spotify call regardless of the submitter limit — fold that into the fix.

### 2. [MEDIUM] OG/cover image routes re-render per request with no `Cache-Control` (CPU DoS)

- **Where:** `apps/web/src/routes/api/og.$logId.ts:110-122` (returns `ImageResponse`, no cache header; comment at `:13-15` claims "immutable + edge-cached"); `apps/web/src/lib/server/mixtape-cover.ts:102-109` returned by `apps/web/src/routes/api/mixtape-cover.$logId.ts:18-26`.
- **What:** Each request runs Satori + resvg WASM rendering plus `loadGoogleFont` network fetches (`og.$logId.ts:111-114`) and a remote image fetch+base64 inline. No `Cache-Control` header is set on either route (verified: no cache header anywhere in `og.$logId.ts`). workers-og's `ImageResponse` does not add cache headers by default, so the comment's promised caching does not happen — the og:image URLs already carry a `?v=<updatedAt>` query an attacker can randomize to bust any zone-default cache.
- **Why it matters:** A flood of `GET /api/og/<valid-logId>?v=<random>` forces a full WASM image render per request — disproportionate CPU for a cheap anonymous request. The route's own comment asserts a caching guarantee the code doesn't deliver.
- **Next step:** Set an explicit immutable `Cache-Control: public, max-age=31536000, immutable` on the `ImageResponse` so the existing `?v=` versioning actually caches (one render per version), matching the comment's intent.
- **deepsec ✓ (with a refinement):** Confirmed both routes (high confidence) at `og.$logId.ts:110` and `mixtape-cover.ts:102`. One nuance to carry into remediation: deepsec notes workers-og's `ImageResponse` _does_ emit `Cache-Control: public, immutable, …, max-age=31536000` by default — so the real gap on the OG route is not a missing header but that (a) Cloudflare does not auto-store a Worker fetch-handler response in the edge cache without the Cache API / a Cache Rule, and (b) the handler ignores arbitrary query params, so a random `?v=`/`?x=` busts the cache key and forces a fresh render. The durable fix is therefore a server-side Cache-API cache keyed on `logId + updatedAt` (params stripped) and/or per-IP rate limiting, not only a header. (`mixtape-cover.ts` does additionally lack any cache header.) deepsec also flagged the steady-state reliability gap → see D13.

### 3. [MEDIUM] Public SSH terminal has no idle timeout, session cap, or connection limit (DoS)

- **Where:** `apps/ssh/main.go:94-104` (`wish.NewServer` middleware stack). Only timeouts present are the 12s HTTP-client timeout and the 10s shutdown grace; no `wish.WithIdleTimeout`, `wish.WithMaxTimeout`, or max-concurrent-session cap (verified: grep for those returns nothing).
- **What:** `ssh rave.fluncle.com` is a public internet listener. `activeterm.Middleware()` requires an active terminal, but a client can open a PTY and idle indefinitely. Each session spawns a Bubble Tea program plus a galaxy sim ticking at 15 fps (`main.go:2194`), so abandoned sessions accumulate goroutines and CPU with no ceiling.
- **Why it matters:** An internet-facing listener with no idle timeout and no connection cap can be exhausted by holding many idle PTY sessions open — resource-exhaustion DoS. This is the most actionable item in the audit for a public listener.
- **Next step:** Add `wish.WithIdleTimeout(...)` and `wish.WithMaxTimeout(...)`, plus an in-process concurrent-session cap (the app already tracks sessions via `addRaver`/`removeRaver` at `main.go:240-246`, so a counter check in `sessionCountMiddleware` is a small change). Host-level firewall/fail2ban is complementary; the in-process timeout is the durable fix.
- **deepsec ✓ (with a sharper amplification path):** Confirmed (high confidence) at `main.go:94,104`, citing this finding by number. It added a second vector the manual pass missed: each PTY-less non-interactive deep link (`ssh host <coord>`/`latest`/`random`) runs a _synchronous_ API fetch in the connection goroutine (up to 12s), and these PNG-less connections return from `routeCommand` _before_ `sessionCountMiddleware` runs, so they are never even counted as crew aboard — an attacker can hold thousands of them open, each blocking a goroutine and firing one upstream request, entirely below the session radar. Apply a context deadline to that fetch in addition to the timeouts above. (deepsec also found the related D7/D9 SSH defects.)

### 4. [LOW] Admin grant cookie is a 30-day stateless HMAC with no revocation path

- **Where:** `apps/web/src/lib/server/env.ts:114` (`ADMIN_GRANT_MAX_AGE_MS = 30 days`); minted in `apps/web/src/lib/server/admin-auth.ts:45-47` (`signGrant` → `{ iat, role: "admin" }`); verified in `env.ts:189-217`; logout at `apps/web/src/routes/api/admin/logout.ts:8-13`.
- **What:** The admin session cookie is a signed `{ iat, role: "admin" }` token with a 30-day window and no nonce/jti or server-side session record. Logout only clears the client cookie; it cannot invalidate a captured copy. The only revocation lever is rotating `FLUNCLE_API_TOKEN` (which also invalidates the CLI bearer and all in-flight OAuth states, since they share the key — see #5).
- **Why it matters:** The design is public, so an attacker knows that capturing the cookie value once (transient XSS, shared/stolen device, log leakage) yields a 30-day admin credential that survives "log out" and cannot be individually revoked. There is exactly one operator, so blast radius is bounded, but there is no per-session kill switch.
- **Next step:** Acceptable for a single-operator tool. If revocation is ever needed, bind the grant to a server-stored session id (or a rotating epoch counter checked at verify time) so logout/compromise can invalidate one grant without rotating the shared token. At minimum, document that token rotation is the only revocation path.

### 5. [LOW] `FLUNCLE_API_TOKEN` triple-duties as bearer + grant-HMAC + OAuth-state key, with no strength floor

- **Where:** `apps/web/src/lib/server/env.ts:118` (`requireAdmin` compares it as the bearer), `:177-183` (`signState` uses it as the HMAC key for both the admin grant and OAuth state); read via `readEnv` (`:68-78`), which only checks non-empty.
- **What:** One secret is simultaneously (a) the bearer token compared against `Authorization`, (b) the HMAC key for the admin grant cookie, and (c) the HMAC key for OAuth state — with no minimum length/entropy enforced. A short or low-entropy value would simultaneously make the bearer brute-forceable and every HMAC forgeable.
- **Why it matters:** Source is public, so the dual/triple use of this one secret is known. If the deployed value is ever weak, it is a single point of failure for the entire admin boundary. This is deployment hygiene, not a code bug.
- **Next step:** Validate a minimum length (e.g. ≥32 chars) on first read, and/or derive the HMAC key from the token via HKDF rather than signing with the raw bearer, so a bearer leak doesn't equal a signing-key leak. Lowest-effort fix is a documented strong-token requirement plus a length assertion.

### 6. [LOW] `apiErrorResponse` returns raw internal error messages to clients on unexpected errors

- **Where:** `apps/web/src/lib/server/http-errors.ts:10`. Public handlers using it include submissions, newsletter, search, `tracks.$idOrLogId`, preview; the OAuth callbacks reflect provider error strings via the same pattern (`apps/web/src/routes/api/admin/spotify/auth/callback.ts:56-62` and the YouTube/Mixcloud equivalents).
- **What:** For any non-`ApiError` thrown in a handler, the 500 response body is `error instanceof Error ? error.message : String(error)`. Notably, `spotifyFetch`/`readApiError` (`apps/web/src/lib/server/spotify.ts:486-505`) throws raw `Error` strings carrying Spotify's upstream status/body, which would surface verbatim to an anonymous caller of `/api/search` on an upstream failure.
- **Why it matters:** Information disclosure of internal/upstream error detail to unauthenticated clients. These are single-line Workers messages, not stack traces, and no secret was observed in these paths, so impact is limited — but error text shouldn't be reflected.
- **Next step:** Return a generic message for the non-`ApiError` branch and log the detail server-side, reserving descriptive messages for intentional `ApiError`s.
- **deepsec ✓:** Confirmed (high confidence) at `spotify.ts:417,422,487,500,504` — it traced the exact `/api/search → searchTrackCandidates → getSpotifyAccessToken → requestToken/spotifyFetch` reflection chain into `apiErrorResponse`'s non-`ApiError` branch, and confirmed no token/secret appears in the reflected bodies (the secret rides the `Authorization` header, never echoed), so impact is upstream-detail disclosure, not credential leakage.

### 7. [LOW] Public track reads expose operator-internal `videoModelReasoning`

- **Where:** `apps/web/src/lib/server/tracks.ts:69` (`TRACK_SELECT` includes `video_model_reasoning`), mapped at `:146` (`videoModelReasoning`). Flows to `/api/tracks`, `/api/tracks/$idOrLogId`, `/api/tracks/random`, `/api/stories`, `/api/mixtapes` members, and MCP `get_recent_tracks`/`get_random_track`.
- **What:** Every public track read returns `videoModelReasoning` (which AI model rendered the video and why) — operator-internal pipeline metadata never rendered in the public UI.
- **Why it matters:** Minor information disclosure of internal tooling detail. No PII, submitter identity, or secrets. (Verified that `submitter_hash`, `user_id`, `contact`, and pending/draft submissions are correctly absent from public projections.)
- **Next step:** Drop `video_model`/`video_model_reasoning` from the public `TRACK_SELECT`, or split a public projection from the admin one.
- **deepsec ✓ (broadened):** Confirmed (high confidence) at `tracks.ts:69,70,145,146` and again on the MCP path (`mcp.ts:61,79`). It widened the leak set: the same public projection also surfaces `added_to_spotify` and `posted_to_telegram` (operator-workflow flags, mapped at `tracks.ts:118,134`) — reconsider those alongside the video fields when splitting the projection.

### 8. [LOW] Mixcloud `client_secret` sent in a GET query string

- **Where:** `apps/web/src/lib/server/mixcloud.ts:44-53` — the token exchange builds `client_id`/`client_secret`/`code`/`redirect_uri` into `URLSearchParams` and does `fetch(\`${mixcloudTokenUrl}?${params}\`)`, a GET with the secret in the query string.
- **What:** Secrets in URLs are logged by proxies, CDN access logs, and referrer chains more readily than secrets in headers/bodies. Here it is a server-to-server HTTPS call to `www.mixcloud.com`, so exposure is limited to Mixcloud's endpoint logging and any TLS-terminating egress proxy — but it is the weakest secret-handling pattern in the audited set. By contrast Spotify sends its secret as a Basic auth header and YouTube as a POST form body.
- **Why it matters:** Defense-in-depth; the residual leak channel is provider/proxy logs. Mixcloud's documented OAuth flow forces a GET-with-query shape, so it is partly a provider constraint.
- **Next step:** Confirm whether Mixcloud accepts a POST form body; if not, accept as a documented provider constraint. No other code change required.
- **deepsec ✓ (with an added reflection sink):** Confirmed at `mixcloud.ts:49,53`. It found a second-order channel the manual pass didn't: on a non-2xx token response the handler reads `response.text()` and embeds it in the thrown `ApiError` message (`mixcloud.ts:56,60`), and the callback route (`admin/mixcloud/auth/callback.ts:41-47`) reflects `authError.message` to the operator's browser — so if Mixcloud's error body ever echoes the request URL, the `client_secret` could surface there. Even if the GET shape stays, stop reflecting the raw body and scrub URLs/secrets from error text.

### 9. [LOW] Presigned R2 PUT does not constrain `Content-Length`

- **Where:** `apps/web/src/lib/server/r2-presign.ts:76-80` (signs `content-type` but not `content-length`); TTL is `PRESIGN_TTL_SECONDS = 3600` (`r2-presign.ts:22`). Minted only by the admin-gated `apps/web/src/routes/api/admin/tracks.$trackId.video.uploads.ts:26,78`.
- **What:** The presigned upload URL binds the object key (server-derived) and Content-Type into the signature, but not Content-Length. Within the 1-hour window, a presigned PUT for `<logId>/footage.mp4` could upload an arbitrarily large body.
- **Why it matters:** Minor abuse ceiling, and admin-only — an attacker would need the admin token to mint such a URL, and the key is one the server chose (no path traversal, no bucket-wide write — all verified sound). Worth noting as defense-in-depth on the storage-cost dimension.
- **Next step:** Bind a max `Content-Length` (or `Content-Length-Range`) into the presign so an oversized upload is rejected at the edge. Optional.
- **deepsec ✓:** Confirmed (low confidence, matching the Low rating) at `r2-presign.ts:69,76,77,78`. It re-verified the object key is fully server-derived with no path traversal and no bucket-wide write, so the residual risk is purely a storage-cost ceiling if a presigned URL leaks or the admin token is misused.

### 10. [LOW] `curl | sh` CLI installer has no checksum verification

- **Where:** `apps/web/src/routes/cli/latest[.]sh.ts:44,60` (served as `https://www.fluncle.com/cli/latest.sh`). The Homebrew formula does pin sha256 (`apps/cli/packaging/homebrew/fluncle.rb`), so the shell installer is the weaker channel.
- **What:** The generated installer downloads `fluncle-<platform>-<cpu>` from `github.com/.../releases/latest/download/...` over HTTPS and `chmod 755` + moves it into `$HOME/.local/bin` with no sha256 verification. The script is otherwise clean: `set -eu`, HTTPS-only, temp file + cleanup trap, unprivileged user path, no `sudo`.
- **Why it matters:** Standard `curl|sh` trust model plus no integrity check — a compromised GitHub release artifact (or a TLS-stripping MITM) would execute unverified. On par with most installers, hence Low.
- **Next step:** Publish a `.sha256` alongside each release asset and have the installer verify it before the `mv`. Acceptable as-is for the threat model.

### 11. [LOW] SSH→API client reads the response body with an unbounded `io.ReadAll`

- **Where:** `apps/ssh/main.go:2062` (`doJSON`: `io.ReadAll(response.Body)`). By contrast the DNS client caps reads at 1 MiB (`apps/dns/api.go:161`, `io.LimitReader(res.Body, 1<<20)`).
- **What:** The SSH app reads the Fluncle API response with no size limit. Upstream is Fluncle's own first-party API, so this only bites if that origin is compromised or returns an unexpectedly huge body — but it is an inconsistency with the DNS client's correct pattern.
- **Why it matters:** Low — same-origin trust limits exposure; flagged for consistency and defense-in-depth.
- **Next step:** Wrap with `io.LimitReader` mirroring the DNS client.
- **deepsec ✓:** Confirmed (medium confidence) at `main.go:2062`, citing this finding by number; it noted `fetchGalaxyTracks` (`main.go:1953-1976`) loops over all pages into a growing slice, so a hostile/buggy origin streaming a large body could drive memory pressure on the small VPS.

### 12. [MEDIUM] Submission/newsletter rate-limit key includes the attacker-controlled User-Agent — defeats the 5/hour cap _through_ Cloudflare

> **Raised from Low to Medium after the deepsec run.** The manual pass flagged only the spoofable `x-forwarded-for` _fallback_ (real only if the origin is reachable directly — Low). deepsec found the stronger, deployed-posture bypass: the User-Agent is also part of the key.

- **Where:** `apps/web/src/lib/server/submissions.ts:382-383` (`hashSubmitter`: `key = \`${connectingIp ?? forwardedFor ?? "unknown"}:${userAgent}\``). The same hash is reused by the newsletter limiter (`newsletter.ts:26`, `enforceRateLimit(hashSubmitter(request))`) and by both MCP tools (`submit_track`/`subscribe_newsletter`).
- **What:** The rate-limit identity folds in the request `user-agent` header (`submissions.ts:382`), which is fully attacker-controlled. Even in the deployed posture where Cloudflare always sets an unspoofable `cf-connecting-ip`, an attacker on a single IP rotates the User-Agent on each request to mint a fresh bucket (count = 0), so the 5/hour cap never trips. - **Next step:** Key the limiter only on `cf-connecting-ip` (treat its absence as a single `"unknown"` bucket); drop both the User-Agent and the `x-forwarded-for` fallback from the hash. Apply the same bucket to the MCP `submit_track`/`subscribe_newsletter` paths, add a Cloudflare Rate Limiting rule as an edge backstop, and optionally make the count+insert atomic to close the TOCTOU.
- **deepsec ✓ + strengthened:** Independently found (high confidence) at `submissions.ts:382,383,67,77`, explicitly distinguishing itself from this audit's original #12 ("works through Cloudflare").

## deepsec-only findings (D1–D13)

These were surfaced by the deepsec run and were _not_ in the manual pass's numbered list. Each `file:line` was independently re-read and confirmed. deepsec's `BUG` class (a correctness/reliability defect with security relevance) is mapped to Low/Info unless impact warrants Medium; the rationale is noted per finding.

### D1. [MEDIUM] Public DNS listener: no rate-limit/RRL, accepts `QR=1`/non-QUERY opcodes, un-minimized `ANY` → reflection-amplification

- **Where:** `apps/dns/dns.go:31` (validates only question count + `Qclass` — does **not** reject `QR=1` or non-`QUERY` opcodes), `:60-61` (`TypeANY` answered as a full TXT, not RFC-8482-minimized), `:66,83` (every child-label query triggers a synchronous blocking upstream HTTP lookup), `:163-187` (`run`: stock `dns.Server` on UDP/TCP `:53`, no RRL, no per-source rate limit, no in-flight/outbound concurrency cap).
- **What:** `dig.fluncle.com`'s authoritative server is a public UDP/TCP listener with no response-rate-limiting and no opcode/QR gate. A ~250–500-byte TXT answer is several times the ~45-byte query, UDP source addresses are spoofable, and `ANY` is honored as a full answer — so it can be used as a DDoS reflector/amplifier. Independently, each query fans out into a synchronous upstream `GET /api/tracks/<label>` with no concurrency cap, so a flood either spawns unbounded goroutines (FD/port/memory exhaustion on the small VPS) or serializes into head-of-line blocking.
- **Why it matters:** This is the DNS analogue of finding #3 (SSH) — the manual pass had marked the DNS server "not an open resolver… no amplification," which is true for recursion but missed the reflection/RRL and outbound-fan-out gaps on its own zone. Medium: unauthenticated, internet-facing, both an amplification surface and a self-DoS / paid-upstream-hammering vector.
- **Next step:** Add per-source-prefix RRL for UDP; drop packets with `QR=1` or unsupported opcodes early; minimize `ANY` (single record or HINFO per RFC 8482); bound handler and outbound-API concurrency with a semaphore (a singleflight also helps); set explicit `dns.Server` read/write timeouts.

### D2. [MEDIUM] DNS negative lookups are never cached → unauthenticated queries amplify 1:1 into paid upstream calls

- **Where:** `apps/dns/api.go:85-91` (`lookup`: `c.set(key, t)` is reached only on success; any `fetch` error returns early), `:111-112,121-123` (`errNotFound` for HTTP 404 / empty payload). The query label is attacker-controlled (`dns.go:58`, zone suffix trimmed).
- **What:** Only successful lookups are cached. NXDOMAIN / not-found results (and transient upstream errors) are never stored, so every query for a non-existent coordinate — `a1.dig.fluncle.com`, `a2…`, or the _same_ non-existent label repeated — is a guaranteed cache miss that fires a fresh outbound `GET https://www.fluncle.com/api/tracks/<label>` against the paid Worker + Turso backend. The 60s success cache (the only abuse protection present) is bypassed entirely for the most easily-abused case.
- **Why it matters:** Exactly the cost/availability amplification in the threat model, on the DNS surface: a small spoofable-UDP query stream converts 1:1 into upstream load with no rate limit and no negative caching to absorb it. Medium, and complementary to D1.
- **Next step:** Cache negatives too — store a tombstone for `errNotFound` honoring the config's existing `NegativeTTL`; bound the cache (LRU / max-entries); add a per-source rate limit and an outbound-concurrency cap; coalesce concurrent identical lookups (singleflight).

### D3. [MEDIUM] Admin gate can rely solely on a third-party-asserted Spotify _email_

- **Where:** `apps/web/src/lib/server/admin-auth.ts:34-43` (`isAllowedSpotifyUser`: returns `(email ∈ ADMIN_ALLOWED_EMAILS) || (profile.id ∈ ADMIN_ALLOWED_SPOTIFY_IDS)`); the callback (`routes/api/admin/spotify/auth/callback.ts:28-41`) mints the 30-day admin grant purely on that boolean. Per the comments (`env.ts:7-9`, `admin-auth.ts:21-24`), `ADMIN_ALLOWED_EMAILS` is the _required_ check and `ADMIN_ALLOWED_SPOTIFY_IDS` is _optional_.
- **What:** When the optional id list is unset (the documented default posture), a Spotify-asserted email is the **sole** gate to full publishing authority. Email is a mutable, provider-asserted attribute of `/v1/me`; trusting it rather than the stable, immutable Spotify user id for the top privilege is a fragile trust boundary.
- **Why it matters:** The manual pass verified the allow-list is exact/case-correct/fail-closed but did not question the email-verification assumption. Exploitation requires obtaining a Spotify profile that returns the operator's exact email (Spotify's signup verification normally prevents this), so this is defense-in-depth, not a demonstrated bypass — but it sits on the highest-privilege gate. Rated Medium for _where_ it sits; deepsec rated its confidence low for the same exploit-difficulty reason.
- **Next step:** Prefer the stable Spotify user id as authoritative — make `ADMIN_ALLOWED_SPOTIFY_IDS` required (and/or require **both** id and email to match), or document that email-only configuration is unsupported and treat the email as a secondary signal.

### D4. [MEDIUM] `/llms-full.txt` re-paginates the entire archive per request (CPU/DB amplification)

- **Where:** `apps/web/src/lib/server/agent-discovery.ts:282-292` (`llmsFullResponse`: loops `listTracks` at pageSize 100 up to `llmsFullMaxFindings=2000` — ~20 iterations, each running a `SELECT` + a `COUNT(*)` + a published-mixtape subquery, so ~40–60 DB queries), `:299` (sets `Cache-Control: public, max-age=3600`). Mounted ahead of the router (`server.ts`).
- **What:** Each request re-paginates the whole archive into a multi-hundred-KB markdown string. The `Cache-Control` header is present, but Cloudflare does not auto-cache a Worker-generated response without the Cache API / a Cache Rule, so every edge miss or cache-bust re-executes the full pagination. Cost grows linearly with the archive.
- **Why it matters:** Unauthenticated CPU/DB amplification (operator pays per Worker CPU-ms and per DB call). Slightly milder than the image routes because a header is present and work is bounded to 2000 findings — but it's the same class. Medium.
- **Next step:** Cache the rendered document via the Cloudflare Cache API keyed on a content/version stamp (render at most once per TTL); compute `totalCount` once instead of per page; consider a smaller hard page budget with an explicit truncation note.

### D5. [MEDIUM] Unbounded MCP JSON-RPC batch fan-out multiplies every tool's cost in one request

- **Where:** `apps/web/src/lib/server/mcp.ts:270-272` (`await Promise.all(payload.map((message) => dispatch(message, request)))` with no cap on the array length).
- **What:** When an MCP payload is an array, every entry is dispatched concurrently with no length limit. A single unauthenticated POST with thousands of `tools/call` entries fans out into that many concurrent tool executions — multiplying every other vector at once: Spotify calls (`search_tracks`/`submit_track`), DB reads (`get_recent_tracks`/`get_random_track`), or Loops contact-creates + confirmation emails (`subscribe_newsletter`). The attacker pays for one request; the operator pays the fan-out.
- **Why it matters:** Cloudflare's per-invocation subrequest limit caps the absolute worst case but still permits a large multiplier and removes the per-request bound the tools assume. Medium; compounds #1, #12, D6.
- **Next step:** Reject batches above a small fixed size (e.g. >10–20) with a JSON-RPC error and/or process entries through a bounded concurrency pool instead of an unbounded `Promise.all`.

### D6. [MEDIUM] Newsletter subscribe is abusable for email-bombing / Loops list pollution (User-Agent-rotation bypass)

- **Where:** `apps/web/src/lib/server/newsletter.ts:26` (only guard is `enforceRateLimit(hashSubmitter(request))` — the same UA-tainted key as #12), `:92-104` (`createContact` adds the address with `subscribed: true`), `:131` (`sendConfirmation` fires a transactional email when `LOOPS_TRANSACTIONAL_ID` is set). Reachable via `/api/newsletter`, `/api/v1/newsletter`, and MCP `subscribe_newsletter`.
- **What:** For any attacker-supplied email passing a loose format check, the endpoint (a) adds it to the operator's Loops audience as `subscribed: true` — so the weekly newsletter is later delivered to a non-consenting victim, turning Fluncle into an unwitting spam relay — and (b) sends a confirmation email to that victim, consuming paid Loops quota and harming sender reputation. The only brake is the #12 limiter, which is defeated by User-Agent rotation; the per-email `Idempotency-Key` only dedupes the _same_ address for 24h, not distinct victims.
- **Why it matters:** Directly contradicts the manual pass's "Coverage — verified sound" claim that the newsletter path was safe (it covered header-injection and enumeration but not list-bombing). Medium: unauthenticated, abuses a paid third party, and weaponizes the operator's domain reputation.
- **Next step:** Move to a durable `cf-connecting-ip`-keyed limiter (shared state — Durable Object / KV / Turso — not per-isolate memory) with a global per-window send ceiling; strongly prefer Loops double-opt-in so a subscribe can't deliver mail to an address that didn't initiate it; consider a CAPTCHA / proof-of-work on the unauthenticated form. (Fixing #12's key is a prerequisite.)

### D7. [LOW] Wish middleware list is reversed: PTY-less deep links are never logged; connect/disconnect timing is broken

- **Where:** `apps/ssh/main.go:97-103` — the list is `[logging, activeterm, bubbletea, sessionCount, routeCommand]`. Wish's `WithMiddleware` folds `h = m(h)` in list order, so the **last** entry is outermost; effective order is `routeCommand → sessionCount → bubbletea → activeterm → logging`.
- **What:** `logging.Middleware()` ends up innermost, reached only after `bubbletea` has run, so interactive sessions log `connect`+`disconnect` together at session _end_ with a meaningless ~0s duration. Worse, `bubbletea` returns _without_ calling `next` when there's no PTY, so for every PTY-less deep link (`renderNonInteractive`) and bare no-PTY connection, `activeterm` and `logging` never run at all — those connections are **never logged**. `activeterm` (now after `bubbletea`) is a dead gate (no PTY bypass results, since `bubbletea` still enforces its own PTY check).
- **Why it matters:** Security-relevant as degraded audit/abuse-detection visibility on an anonymous public listener — and it compounds #3/D-note exactly where it hurts: the flood traffic (PTY-less deep links) is precisely what goes unlogged, so connection logs can't surface a flood. Low (no auth bypass), but worth fixing alongside the SSH DoS hardening.
- **Next step:** Reverse the list to Wish semantics: `logging` last (outermost → correct `connect` at start + real duration), `activeterm` _before_ `bubbletea`, `routeCommand`/`sessionCount` inner. Add a test asserting connect/disconnect are logged for PTY-less sessions.

### D8. [LOW] DNS server does not honor EDNS0 / set the TC bit; oversized TXT may be unresolvable; `WriteMsg` errors discarded

- **Where:** `apps/dns/dns.go:82-101` (`answerTXT` appends a TXT RR but never echoes the request's EDNS0 OPT nor sets TC when the packed message exceeds the UDP limit), `:33,44,51,79` (`WriteMsg` error discarded on every path, e.g. `_ = w.WriteMsg(m)`). `txt.go`'s `splitStrings` only bounds each individual string to ≤255 bytes, not the total message size.
- **What:** A finding with a long title + multiple artists + album + both URLs can exceed the 512-byte classic-UDP limit; with no advertised EDNS0 buffer and no TC bit, an EDNS0 resolver may treat the answer as malformed or never learn to retry over TCP — so unusually large but legitimate findings become intermittently/fully unresolvable over UDP, silently (the send error is dropped).
- **Why it matters:** Primarily a correctness/availability defect (a "bug" in deepsec's taxonomy), low security weight, but it degrades the public surface and has no log signal. Low.
- **Next step:** Echo an EDNS0 OPT with a sane UDP buffer when the query carries one and call `m.Truncate(size)` / set TC so responses fit and signal TCP fallback; log `WriteMsg` errors instead of discarding them.

### D9. [LOW] No panic-recovery middleware on the long-lived multi-tenant SSH server

- **Where:** `apps/ssh/main.go:94-104` (middleware stack omits Wish's `recover.Middleware()`).
- **What:** `bubbletea`'s `program.Run()` executes in the per-connection goroutine; an unrecovered panic in any session's `Update`/`View` (or the galaxy sim) propagates and crashes the **entire** process, taking the listener down for all connected users. deepsec found no concretely attacker-reachable panic (indices clamped, nil-checks present, the type assertion always holds, divisors guarded), so this is defense-in-depth.
- **Why it matters:** On a shared, anonymous, long-lived server, one malformed session should not be able to abort the whole process. Low.
- **Next step:** Add `recover.Middleware()` to the stack so a panic is contained to the offending session.

### D10. [LOW] Free-text terminal fields are not ESC-stripped like the link fields

- **Where:** `apps/ssh/main.go:2180-2184` (`terminalLink` strips `\x1b` from URL + label before the OSC-8 wrapper — showing terminal-escape-injection awareness), but that stripping is **not** applied to free-text fields rendered elsewhere: track Title/Artists/Label/Tags and mixtape Title/Note go straight into lipgloss `.Render()` (`renderDetail`, `:1213`+ — e.g. `rowTitleStyle.Render(t.Title)`, `labelStyle.Render(artistLine(t.Artists))`, the `Pressed by:` label and `Reads as:` tags; `renderMixtapeDetail`, `:1314`+).
- **What:** A latent inconsistency: the link-field defense implies an intent to sanitize server-supplied text generally, but the free-text fields would pass raw ANSI/control bytes through to the viewer's terminal (cursor manipulation, screen clears, spoofing). **Not currently exploitable** — the data is operator-curated archive or Spotify-catalog text, submitted notes go to a review queue (never rendered to other users), and a user's own typed input is restricted to single-byte printable ASCII (`handleInputKey:919-923`). (`terminalLink` also strips only `\x1b`, not `\x07`/BEL.)
- **Why it matters:** Low — trusted source today, but worth closing because the partial defense already exists.
- **Next step:** Centralize a sanitizer that strips C0/C1 control bytes (incl. `\x1b` and `\x07`) from all server-supplied strings before rendering, and apply it to the free-text fields as well as the link fields.

### D11. [LOW] `deleteMixtape` is a non-atomic check-then-act that can delete a just-minted mixtape

- **Where:** `apps/web/src/lib/server/mixtapes.ts:474-482` (reads the row, throws unless `status === "draft"`), `:485-491` (then issues an unconditional batch `delete from mixtape_tracks … / delete from mixtapes where id = ?` with **no** status guard in the `WHERE`). `publishMixtape` (`:420-445`) concurrently flips a draft to `distributing` and mints a permanent, never-reassigned `sequence_number` + Log ID coordinate.
- **What:** If a delete reads `status = "draft"` just before a concurrent publish commits, the subsequent unconditional delete removes the now-minted mixtape — destroying a consumed coordinate (data loss / sequence gap). Both paths are admin-gated and single-operator, so likelihood is low (needs two concurrent admin actions on the same id — two tabs / double-submit), but the check-then-act is genuinely non-atomic.
- **Why it matters:** Integrity/availability of the spine's coordinate sequence; admin-gated so not attacker-reachable. Low (deepsec's "bug" class).
- **Next step:** Add `and status = 'draft'` to the `delete from mixtapes where id = ?` (and the `mixtape_tracks` delete) and verify `rowsAffected`, so a row that became `distributing`/`published` between read and write is not deleted.

### D12. [LOW] Concurrent Spotify token-refresh on the public path can wipe the operator's auth row

- **Where:** `apps/web/src/lib/server/spotify.ts:336` (refreshes within 60s of expiry, no lock/coalescing), `:353-354` (on `invalid_grant`, unconditionally calls `clearSpotifyAuth`), `:396-403` (`clearSpotifyAuth` runs `delete from spotify_auth`). Triggered by the **unauthenticated** `/api/search` path.
- **What:** Many concurrent anonymous requests in the 60s pre-expiry window each fire a refresh with the same stored refresh token (a thundering herd of token POSTs). Once Spotify's 6-month refresh-token expiry lands (the file's own comment notes 2026-07-20) and a rotation invalidates the prior token, the losing concurrent refreshes get `400 invalid_grant` and the handler deletes the whole `spotify_auth` row — breaking the admin add/publish flow until a manual reconnect. Attacker-influenceable by flooding `/api/search` at expiry (amplified by #1's missing rate limit).
- **Why it matters:** Availability of the operator's core publish credential, reachable from an unauthenticated route; latent today, sharper after 2026-07-20. Low (deepsec's "bug" class), but tied to the #1 rate-limit fix.
- **Next step:** Single-flight the refresh (cache the in-progress promise) and make `clearSpotifyAuth` conditional/idempotent — don't delete on a transient/racy `invalid_grant`; re-read and clear only if the stored refresh token still matches the one that failed.

### D13. [LOW] OG card fetches Google Fonts per request with no caching/fallback; an unhandled failure 500s the card

- **Where:** `apps/web/src/routes/api/og.$logId.ts:110-113` (`loadGoogleFont` called twice per request, no module-level/Cache-API memoization, no `try/catch`).
- **What:** Unlike `fetchImageDataUri` (which catches and degrades to the bare background), a failed/rate-limited Google Fonts fetch throws and propagates, returning a 500 instead of an image. This is both a steady-state reliability gap (every render depends on two live third-party fetches) and a self-amplifying failure mode: the #2/D-note render abuse can itself trigger Google Fonts rate-limiting, after which legitimate social-crawler OG requests start 500ing.
- **Why it matters:** Reliability of the public OG surface (deepsec's "bug" class); compounds finding #2. Low.
- **Next step:** Fetch the Oxanium bytes once and reuse (module scope or Cache API), and wrap the font load + `ImageResponse` in a `try/catch` that falls back to a system font / static placeholder card so the endpoint degrades instead of 500ing.

## Coverage — verified sound

These areas were reviewed in depth and found sound; recording them so the audit's coverage is auditable.

- **Admin trust boundary.** Constant-time bearer comparison via `timingSafeEqual` (`env.ts:117-131,227-236`) — no `===` on any secret. The allow-list (`admin-auth.ts:25-43`) is exact, case-correct, and fail-closed (empty/missing env denies all, never allows all). Every one of the 34 `/api/admin/*` route files gates with `requireAdmin` as its first statement, except the intentionally-public login front door and the OAuth callbacks, which are instead gated by HMAC-signed state with a 10-minute window. The `/api/v1/*` mirror re-imports the identical `serverHandlers` objects (no parallel ungated surface). The admin grant cookie sets `HttpOnly`, `SameSite=Lax`, `Path=/`, `Secure` (prod). HMAC is verified before `JSON.parse` (no parse-before-verify); a valid OAuth state cannot be replayed as an admin grant (role check).
- **CSRF / IDOR.** State-changing `/api/me/*` routes enforce JSON content-type + same-origin (origin/referer) + a signed double-submit CSRF token bound to `user.id` (`public-auth.ts:216-256`). Every `/api/me/*` read/write/delete/export is scoped server-side to the session's `user.id` — no IDOR. better-auth config is sound (secret required in prod, explicit `trustedOrigins`, no `disableCSRFCheck`, no `crossSubDomainCookies`). The public/admin auth modules are physically isolated, enforced by `public-auth-boundary.test.ts`.
- **Database layer.** Uniformly parameterized — zero SQL injection, ORDER BY/identifier injection, mass-assignment, or unbounded-pagination exposure across all SQL-building modules and all 23 Drizzle migrations. The generic admin update path (`track-update.ts`) and mixtape update use hardcoded `column = ?` allow-lists and never spread the request body. `LIMIT` is coerced-and-capped everywhere (max 48/50/54). Turso creds are read from `process.env` only, with no hardcoded fallback and no prod-from-test path; the URL/token are never logged. Migrations contain no destructive unscoped statements and no embedded secret values.
- **Public input surface.** Submissions enforce a strict `^[A-Za-z0-9]{22}$` Spotify-id regex + URL/id cross-check + honeypot + a DB-backed 5/hour per-submitter rate limit, and re-fetch authoritative metadata from Spotify rather than trusting client `title`/`artists` (no stored-XSS path into the admin UI). Newsletter caps email length, avoids header injection and enumeration (same `{ ok: true }` whether or not already subscribed), and is idempotent to Loops. **(Caveat from the deepsec pass: the _content_ of those validations is sound, but the rate limiter gating them is bypassable — see #12 and D6 — so "validated + rate-limited" holds for the validation, not the limit.)** The preview/OG/cover proxies have no SSRF (URLs are DB/provider-derived, never attacker-supplied) and no injection (params constrained by the lookup gate's charset; OG escapes title/artist) — though they are not server-side cached (#2, D4, D13). `/api/health` returns only `{ ok: true }`. CORS `*` appears only on `/api/preview` and `/mcp`, both serving public credential-free data with no `Access-Control-Allow-Credentials`. The MCP `submit_track`/`subscribe_newsletter` tools call the exact same validated functions as the HTTP routes — no softer back door (but they share the same bypassable limiter, and the JSON-RPC batch path adds an unbounded fan-out — D5).
- **R2 presign + OAuth integrations.** The presigned-upload object key is fully server-derived (`<logId>/<artifact>` from a DB row + a fixed artifact allow-list) — no path traversal, no bucket-wide write; key segments are `encodeURIComponent`'d and the bucket is a hardcoded constant. All OAuth `start` handlers mint signed state with a nonce + purpose + 10-minute window, verified constant-time on every callback; `redirect_uri` is a fixed env var for Spotify/YouTube; post-auth redirects are fixed relative paths (no open redirect). Refresh/access tokens are stored server-side, never in client-readable cookies; the login path discards Spotify tokens after reading identity. No SSRF in any integration client; Telegram/Discord are outbound-only (no inbound webhook to verify); secrets travel in headers/bodies (except Mixcloud, finding #8) and are never logged.
- **Go services + CLI.** The SSH server is intentionally anonymous and can reach only public endpoints; it carries no credential and has no shell-out/command-injection surface (commands parse into a closed enum; coordinates are `url.PathEscape`'d; terminal-escape bytes are stripped from server-supplied _link_ fields — though **not** from the free-text fields, see D10). The host key is generated/persisted to a gitignored path — no key committed. The DNS server is not an open _recursive_ resolver (anything outside the zone → REFUSED, never recurses), exposes no inbound control API (`api.go` is an outbound client), bounds malformed packets via `miekg/dns`, sanitizes TXT assembly, and its systemd unit is exemplary (`DynamicUser`, `CAP_NET_BIND_SERVICE` only, `NoNewPrivileges`, `ProtectSystem=strict`, syscall filter). **Correction from the deepsec pass:** while it does not recurse, the DNS server _does_ present a reflection/amplification surface on its own zone (no RRL, accepts `QR=1`/odd opcodes, un-minimized `ANY`) and amplifies queries 1:1 into the paid upstream because negatives are never cached — see D1 and D2; the "no amplification" line in the manual pass was scoped to recursion only. The CLI never writes/logs the token, talks HTTPS to a first-party origin with no command-line origin override, and its packaging pins sha256 for Homebrew + uses OIDC trusted publishing for npm. (The CLI, Raycast, and `packages/*` were covered by the manual pass only; deepsec did not investigate them.)
- **Secrets / supply chain.** No live secret in tree or history (method above). `.gitignore` coverage is complete (`.env`, `.dev.vars`, `apps/web/.dev/`, SSH host keys, build artifacts). CI workflows use `permissions: contents: read`, `persist-credentials: false`, reference secrets only by name, and publish to npm via OIDC (no token). `bun.lock` is committed with integrity hashes; no `git+`/`*`/`latest`/`file:` dependency ranges; lifecycle scripts are benign and first-party. Go modules have no `replace` directives and use reputable sources.
