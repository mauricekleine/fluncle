# Tonight's domain: Security & secret hygiene

This repo is **open source and world-readable forever, git history included** — a single committed secret VALUE or a slice of the secret-management topology is existential, un-leakable once pushed. The auth tiers are the platform's spine: every mutating and admin op must sit behind the right guard, and the publish boundary (only the Worker holds the keys) must stay intact. So this domain leans hard toward **filing, not fixing** — the highest-value findings here (a leaked secret, a mis-tiered op, a widened footprint) are exactly the human-eyes class, and the hard rails forbid you from editing an auth guard or a secret file yourself. Hunt relentlessly; touch almost nothing.

## The hunt

**1. Committed secret VALUES or the secret-management MAP (highest severity — file every hit, fix none).** Hunt the working tree for anything that should have stayed in 1Password or been a placeholder. Two distinct classes, both forbidden in committed files (`AGENTS.md` → "Public Repo"):

- **Secret values** — real tokens, keys, passwords, session secrets. gitleaks scans full history on every push (`.github/workflows/gitleaks.yml`, config in `.gitleaks.toml`); you are the second pair of eyes on the _current tree_, especially near new code that reads `env`/`process.env`, fixtures, test files, and docs.
- **The topology / reference map** — concrete `op://<vault>/<item>/<field>` 1Password paths, hostnames, IPs, ports, internal/webhook URLs, tailnet names, or local `/Users/...` paths. These grant nothing alone but hand out the layout. The CI backstop is a working-tree grep for `op://[A-Za-z0-9]` (the second step of `gitleaks.yml`); the correct form is a PLACEHOLDER — `op://$FLUNCLE_1PASSWORD_ENV_ITEM/<field>` or `op://<vault>/<item>/<field>`, as in `apps/web/.dev.vars.tpl`. A concrete `op://` path, a `rave-0N`/tailnet name, a real webhook URL, or a `/Users/<name>/...` path in a committed file is a finding.
- Public runtime IDENTIFIERS are **fine** and already allow-listed in `.gitleaks.toml` (the R2 account id `0651fd3b…`, the IndexNow token) — do not re-file those. If you find a genuine leak, **file it as high** and do not touch the file, do not rewrite history, do not "fix" it by deleting the line (the value is already public — rotation is the operator's call). Flag it loudly in the report.

**2. Auth-tier correctness (the spine — verify, never edit).** Every mutating/admin op must run at exactly one deliberate tier. The model lives in `apps/web/src/lib/server/orpc-auth.ts`: `adminAuth` (any admin principal, operator OR agent), `operatorGuard`/`operatorProcedure` (operator only — agent gets 403), and the `/me` `privateUserAuth` / `privateUserMutation` cookie-session tier. The exhaustiveness net is `orpc-auth-coverage.test.ts` — every contract op is either in `PUBLIC_UNAUTH_OPS` (a deliberate anonymous endpoint) or `EXPECTED_TIERS` with a chosen tier, and the derived middleware chain must match. Read the diff-worthy question: does any op that _publishes, mints a coordinate, edits a live surface, or spends/credentials_ sit at `admin` when it should be `operator`? Does a new public read belong on the unauth allow-list at all? The file-route carve-outs are outside the oRPC net and must guard themselves — confirm the upload/media routes under `apps/web/src/routes/api/admin/**` still call `requireOperator`/`requireAdmin` (e.g. the source-audio master read is operator-only; the silent-clip is agent-tier). **Touching an auth guard or a tier map is on the NEVER list — a suspected mis-tier is filed, with the op name, its current tier, and the tier you believe it needs.**

**3. Publish-boundary integrity.** The Worker (`apps/web`) is the only holder of the publish keys (Spotify/Telegram/Discord/YouTube/Mixcloud/Postiz); the CLI, Raycast, SSH, and the box crons are thin clients that trigger through the authenticated admin API. Flag anything that would let a client reach a publish credential directly, or a new server path that publishes without going through an operator-tier op.

**4. Input validation on public/admin surfaces.** Contract ops carry zod input/output schemas (`packages/contracts/src/orpc/**`) — the coverage tests (`orpc-coverage.test.ts` / `orpc-admin-coverage.test.ts`) enforce that every non-carve-out route has a contract. Hunt for a public/admin surface accepting unvalidated input: a loose `z.any()`/`z.string()` where a bounded shape belongs, an unbounded array or string that should have a `.max()`, a numeric limit with no ceiling, or a file-route carve-out parsing a body by hand with no validation. A tightening of a schema on an existing op is often a **safe direct fix** if you can verify it (typecheck + the op's test); a change that could reject currently-valid traffic is a **file**.

**5. Dependency posture / known-vulnerable pins.** The workspace uses a `minimumReleaseAge` freshness gate (`bunfig.toml`, 3 days). Note dependencies with known advisories or conspicuously stale majors — but a bump is the `fluncle-maintenance` skill's job and touches the supply chain, so **file it as a recommendation**, don't bump it here.

**6. R2 / external-service exposure & footprint.** Confirm presigned-upload ops are scoped to the key they own (not a wildcard bucket grant), that private buckets (e.g. `fluncle-source-audio`, the copyrighted-master vault) are never served publicly, and that no new route, CORS rule, or `.well-known` entry widens the public footprint beyond what the surfaces doctrine intends. Anything that broadens what the world can reach is a finding.
