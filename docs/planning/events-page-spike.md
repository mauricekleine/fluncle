# Events page spike — a `/events` surface for the world's drum & bass

> **This is a spike, not a spec.** A non-canonical planning brainstorm per AGENTS.md's doc rules: where it deviates from the codebase or canon, the codebase and canon win. Nothing here is committed direction until the operator rules on the open decisions at the bottom. Researched 2026-07-17; the source-availability facts are point-in-time and worth a re-check before any sign-up.

## The idea

A public `/events` page: date-based, location-filtered drum & bass events from around the world — gigs, club nights, festivals. Future events only (with an optional backfill question, answered below). It is a new kind of surface for Fluncle: the archive is what he found out there; `/events` is where the crew can go hear it live. It is deliberately NOT the existing `/calendar.ics` feed — that is Fluncle's OWN live sessions (plans + past mixtapes); this is the world's events. The two stay separate objects.

The make-or-break is not the page, it is the data. So that comes first.

## What the data reality is

The uncomfortable headline: **the sources that best cover EU drum & bass (Resident Advisor, DICE) are exactly the ones with no legitimate API, and the sources you can sign up for today are either genre-coarse, UK-only, or partnership-gated.** There is one honest self-serve starting point (Ticketmaster) and one strong UK second (Skiddle, with a permission email). Everything else is a negotiation, a ToS breach, or a corpse.

Two cross-cutting facts from the research worth holding onto:

- **The artist-keyed pattern beats genre search wherever it exists.** We hold 700+ artist entities with names, MBIDs, and Spotify ids. Ticketmaster's `attractionId` lookup keys cleanly off that graph: resolve each artist once, store the id, poll per-artist. 700 artists × 1 call each is well under the 5k/day budget even daily, and it sidesteps promoter tagging sloppiness entirely — a Sub Focus show is a Sub Focus show whether or not someone ticked the subgenre box. (Songkick's MBID-keyed calendar was the perfect fit for this and is exactly the thing that no longer exists.)
- **Every usable ToS points the same way: we are a refreshable, attributed mirror that links out, never a permanent independent copy.** Ticketmaster wants reasonable-period caching plus a 24h takedown path; Skiddle wants credit and live links; Bandsintown wants session caching. The architecture below is built to that posture, and it happens to be the posture we would want anyway (events expire; stale listings are worse than none).

## The architecture, in Fluncle's terms

This maps almost one-to-one onto shapes the repo already runs — the catalogue crawler's frontier discipline, the backfill sweeps' Worker-paced bounded batches, the entity pages' loader shape, and the registry fan-out.

### The `events` table

One row per event occurrence, minted by the sweep, never by hand:

| Column                          | What it is                                                                                                                                                                                                                                        |
| ------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `id`                            | Deterministic: `<source>:<source_event_id>` (e.g. `ticketmaster:vvG1iZ9...`) — the crawler's `crawl_frontier` id trick, so a re-sync collides on the PK and writes nothing. `on conflict do update` refreshes mutable fields (status, url, times) |
| `title`                         | The event/lineup name as the source states it                                                                                                                                                                                                     |
| `venue_name`, `city`, `country` | Country as ISO-3166 alpha-2 — the country filter's key. Raw strings kept as-is (the `tracks.label` immutability lesson)                                                                                                                           |
| `latitude`, `longitude`         | Nullable; Ticketmaster supplies them, a scraped venue page may not                                                                                                                                                                                |
| `starts_at`, `ends_at`          | UTC ISO; `ends_at` nullable. `starts_at` is indexed — every read is a future-window seek                                                                                                                                                          |
| `url`                           | The outbound ticket/detail link (the ToS-mandated link-out; the affiliate wrapper goes here later if we join TM's program)                                                                                                                        |
| `image_url`                     | Nullable; hotlink policy per source (TM images are served for this purpose; a scraped image is not — leave null there)                                                                                                                            |
| `source`                        | `ticketmaster` / `skiddle` / `firecrawl:<site>` / ... — provenance on every row, the tag-provenance habit                                                                                                                                         |
| `status`                        | `scheduled` / `cancelled` / `postponed` — maps straight onto schema.org `eventStatus`                                                                                                                                                             |
| `synced_at`                     | Last time the source confirmed this row — the staleness/purge key                                                                                                                                                                                 |
| `fold_key`                      | The cross-source dedupe fold (below), indexed                                                                                                                                                                                                     |

Plus an `event_artists` edge table (`event_id`, `artist_id`) — the `track_artists` twin — linking events to OUR artist entities where matchable, and a small `artist_event_sources` map (`artist_id`, `source`, `external_id`) storing each artist's resolved Ticketmaster `attractionId` (resolve once, reuse forever — the same lesson as storing the Bandsintown id instead of re-querying by name).

**Dedupe across sources** is the crawler's two-layer idempotence, adapted: layer one is the deterministic PK within a source; layer two is a `fold_key` for cross-source twins — `fold(venue_name) + date + fold(headline-artist-or-title)` — computed at write, with the later-arriving source's row marked a duplicate rather than deleted (provenance is kept, one row renders). Fuzzy venue folds will miss some; that is acceptable — a doubled listing is a cosmetic bug, not a certification breach. Nothing here touches `tracks`, `findings`, or any speaking queue: an event is not a finding and the rail never comes near it.

### The sweep

A deterministic `--no-agent` on-box sweep, `fluncle-events`, on a rave-02 host systemd timer — the `fluncle-backfill` discipline verbatim: the box only triggers; the **Worker** holds the vendor keys (the TM key is a Worker secret, never on the box) and runs the fetch. New agent-tier op `sync_events` (`POST /admin/events/sync`), CLI `fluncle admin events sync`, both through the oRPC contract + coverage tests like everything else.

- **Cadence: every 6 hours.** Events are announced weeks out and change slowly; four ticks a day keeps the mirror fresh well inside the 24h-takedown posture while spending a fraction of the rate budget. Each tick is bounded (a page of the artist rotation + one country of the genre sweep) and cursor-resumable, so "run again" and "resume" are the same command — the marathon-not-process rule.
- **Two acquisition modes per tick**, mirroring the crawler's seed-plus-walk: (a) the **artist-keyed pull** — walk the `artist_event_sources` rotation, one `attractionId` events query per artist, our 700-artist graph doing the genre filtering for free; (b) the **genre sweep** — the Dance/Electronic + Drum & Bass subgenre query per country, catching lineups with no logged artist on them. Mode (a) is the quality spine; mode (b) is the breadth net.
- **Expiry is part of the same sweep**: a past event (`starts_at` older than yesterday) is pruned unless we decide to keep a past-events archive; a row not re-confirmed by its source for N syncs goes stale-marked then pruned. This is also the TM takedown compliance mechanism — the mirror is purgeable by construction.
- **Rate politeness**: one shared per-vendor client with the MusicBrainz client's manners — serialized budget, `Retry-After` honoured, circuit-break on sustained 429/403 with the breaker state surfaced on `get_events_status` (the Spotify-anchor breaker pattern, reused not reinvented).

### The page

`/events`, a public route in the graph-page shape: plate masthead, then the list, dark, quiet, fast. Public route = loader + `useLoaderData`, no react-query (the `artist.$slug.tsx` exemplar); filters live in search params via `validateSearch` so a filtered view is a shareable URL.

- **The list**: grouped by date, soonest first, future-only window in the query (`starts_at >= now`, an indexed seek). Each row: date, title, venue, city + country, and the outbound ticket link. Where an event matches a logged artist, the artist name links to `/artist/<slug>` — the graph edge that makes this page ours rather than a listings clone, and the thing no other DnB events page can do.
- **Filters**: a date-range control and a **country dropdown** (built from the distinct countries actually in the table — no maps API, no geo service, zero cost). City narrows within a country the same way. "Near me" is deliberately v2: it needs browser geolocation plus a lat/lon distance ranked in SQL (the rank-in-SQL DB rule applies — bounded pre-filter by country/bounding-box first, never a full-table haversine into the isolate), still no maps API, but it is real work and the dropdown serves the need on day one.
- **Voice**: the events themselves are earthly facts, and the entity-dossier register is the precedent — this is a surface where cities and countries are named plainly (the cosmos-replaces-the-map rule lifts, as it does on the artist bios). Fluncle's voice lives in the lede and the empty state, not the rows. Chrome stays literal per the Chrome Rule. Copy examples (to be canon-reviewed, not ratified here): lede "Where the crew can go hear it live. I keep this chart updated from out here." Empty state "Nothing on the chart for this stretch. Check back, or widen the window."
- **SEO**: this is a genuine rich-result surface. Google's event rich results require `name`, `startDate` (ISO with timezone), and `location` (venue name + PostalAddress); recommended `eventStatus`, `offers.url`, `performer`, `image`. The supported multi-event pattern is an **`ItemList` of complete `MusicEvent` nodes** on `/events`; Google's richest treatment wants one event per page, but per-event pages for external events we add nothing to are thin-content/doorway territory — the same trap the label pages taught us — so v1 ships the ItemList and per-city pages (`/events/<country>`) become the "one strong page" play later, each with real depth before it earns a sitemap slot. `performer` in the JSON-LD points at our `/artist/<slug>` `@id` where matched, closing the graph the same way album → label does. Event schema is one of the types Google has NOT restricted (unlike FAQ/HowTo), and `eventStatus` covers cancelled/postponed/rescheduled natively.

### The registry fan-out (the add-a-surface checklist)

One `web.events` entry in `@fluncle/registry` (route `/events`, weight `web: secondary`, probeConfig, title + statusDescription) and one `cron.events` entry (`cronName: fluncle-events`, weight `status: hidden`) — and the fan-out lights up `/status`, the homepage dev-row, `llms.txt`, the sitemap, and the doctrine doc from the same two entries. Names run the `verb_noun` registry: `list_events` (public read), `sync_events` (agent tier), `get_events_status` (agent read). The timer unit pair goes in `docs/agents/hermes/events-timer/` with the roster row, installed by `install-host-timers.sh`. Box enable stays operator-gated like every sweep.

## Recommended path

**v1 — ship on Ticketmaster alone (the only zero-permission start):** the `events` table + `event_artists` edge, the `sync_events` Worker op with the artist-keyed pull + genre sweep, the `fluncle-events` timer, and the `/events` page with date + country filters, ItemList MusicEvent JSON-LD, and the artist-page links. Honest v1 coverage: the ticketed layer of the scene — arena tours, festivals, big club brands, Hospitality-scale nights — EU-decent, underground-thin. Say so on the page rather than pretending: the lede owns that this is the chart he can see from out here, growing as sources come online.

**v1.5 — Skiddle for UK club depth** the day the written approval lands: attributed rows, live links, the credit line in the source column. This is the single cheapest genuine-underground upgrade available anywhere.

**v2 candidates, in order:** the curated Firecrawl venue/promoter scrape lane (10–20 hand-picked EU venue sites, robots.txt honoured — the defensible scrape); a JamBase coverage evaluation; per-country pages once rows justify them; near-me.

**Not on any path:** Songkick (dead, and its ToS bans aggregation anyway), SeatGeek (closed), Edmtrain (US-only + anti-aggregator ToS), PredictHQ (wrong shape), Festicket (dead). **RA and DICE are not "later," they are a standing operator decision** — the data everyone actually wants lives there, and the only paths are a partnership ask or a knowing ToS breach.

**Effort estimate: ~4 agent slices**, each a worktree PR: (1) schema + migration + the TM client + `sync_events` + breaker/status, piloted on one country before fan-out; (2) the artist `attractionId` resolution + `event_artists` matching; (3) the `/events` route + filters + JSON-LD + copy, canon-reviewed against the exemplar; (4) registry + timer unit + doctrine/docs + coverage tests. Roughly a one-day orchestrated run at the usual review bar.

## Open decisions (operator-only)

1. **Ticketmaster sign-up.** A free self-serve key, but it is an external account + their ToS accepted in Fluncle's name. Green-light needed before slice 1.
2. **Scraping posture.** Three distinct rungs, decide each: (a) curated venue/promoter sites via Firecrawl with robots.txt honoured — defensible, recommended for v2; (b) DICE — no API, scraping their catalogue is ToS-risky; (c) **RA's GraphQL — explicitly prohibited by their ToS for commercial automated extraction.** The spike's recommendation is (a) yes, (b) no, (c) no — but send RA a partnership email; the worst answer is silence and the page loses nothing by waiting.
3. **Permission emails worth sending now:** dev@skiddle.com (commercial approval), API@bandsintown.com (partner program), RA partnerships. All free; all unblock real coverage.
4. **Future-only vs 2026-01-01 backfill.** Recommendation: **future-only.** The APIs serve upcoming events, so a backfill would mostly be unobtainable anyway — and because events are announced months ahead, the future window is already full on day one. If a past-events archive is wanted later, it accumulates naturally by keeping expired rows instead of pruning them (decide retention then; TM's caching posture prefers pruning).
5. **Near-me vs country filter.** Country/city dropdown for v1 (zero cost, no permissions). Near-me is real work with a scale-safe query shape — a v2 line item, not a default.
