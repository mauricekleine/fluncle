# Fluncle over Tor — the archive, off the grid

An onion is another surface in the Galaxy: the same findings, the same Log IDs, reached off the grid. A v3 `.onion` address is self-authenticating and end-to-end encrypted by Tor itself — no DNS, no CA, no exit node — so it gives a visitor metadata privacy and censorship-resistant reach to the archive. It does not protect Fluncle's infrastructure (the mirror proxies the known clearnet site), and the archive is public and read-only, so this is a principled, on-brand surface, not a load-bearing access path. Size the effort to a tasteful mirror.

This runbook mirrors `docs/dig.md`'s "gated production steps — run them yourself" model. Two things differ from `dig.fluncle.com`, both load-bearing.

- **There is no `apps/tor/` directory and no repo binary to build.** Unlike `fluncle-dns` (a repo Go binary with a hand-written systemd unit), Tor is a stock package with its own `debian-tor` user and config, and the rewriting proxy is generated by [onionspray](https://onionservices.torproject.org/apps/web/onionspray/), not hand-written. The only repo artifact is the `Onion-Location` header (Unit C below).
- **The onion opens no inbound public ports.** `fluncle-dns` opens `:53` at two firewall layers and the SSH app opens `:22`; an onion service reaches the network by outbound connections only and clients arrive through rendezvous points inside Tor. The firewall posture does not change. This is the most reassuring fact here and the easiest to get wrong by reflex.

Admin access to the public-edge box is private (over the tailnet, not the public internet; the `hetzner-devbox` profile), separate from the public app port — see the ops runbook note.

## What runs where

One Tor daemon on the existing public-edge box (the one that serves `rave.fluncle.com`), beside the Wish SSH app (`:22`) and `fluncle-dns` (`:53`). It hosts two onion identities, independent keys and lifecycles.

| Onion      | Backs                                                               | Reaches                                   |
| ---------- | ------------------------------------------------------------------- | ----------------------------------------- |
| web onion  | an onionspray rewriting proxy → the live `www.fluncle.com` (Worker) | web, `/api/v1/*`, `/rss.xml`, `/mcp`      |
| rave onion | a `HiddenServicePort` → the Wish app on `127.0.0.1:22`              | `ssh rave.fluncle.com`, the rave terminal |

API, RSS, and MCP are free riders on the web onion — they are just paths on `www.fluncle.com`, proxied like any other path. MCP is the cleanest case: the server is stateless, JSON-only, POST-only, with no SSE, so it needs no special proxy handling.

## Owner decisions to settle before building

- **SSH onion in v1, or a fast-follow?** The web onion plus `Onion-Location` captures essentially all the value and closes five of the six Tor checklist boxes. The rave/SSH onion is the thinnest-value unit (a public, no-auth terminal toy over Tor); it is nearly free since it rides the same daemon, so "do it for completeness" is defensible, but it is a genuine choice. Maurice decides.
- **Key custody item names.** The onion private keys go to the configured 1Password items (see the ops runbook note), matching the custody used elsewhere. Confirm the item names per that note.
- **Same box vs. a dedicated box.** Recommended: the existing public-edge box ($0 marginal). A dedicated box (~€4–5/mo) buys only blast-radius isolation and is paid infrastructure, so it needs an explicit yes.
- **Vanity prefix?** Optional novelty flex (mine a readable prefix with `mkp224o` before first start). Costs local CPU wall-clock, no money. Default: no.

The onionspray steps, the VPS steps, and the Cloudflare bypass below are operator-run gated production steps. They are not automatable from this repo — run them yourself.

## 1. Stand up the web onion (onionspray) — the real work

### 1a. Clear the Cloudflare origin bypass first

onionspray's upstream fetch is a server-to-server HTTPS request from the box's datacenter egress IP to the Cloudflare-fronted origin — exactly the profile bot management flags. If a challenge fires, every onion visitor sees a Cloudflare challenge they cannot solve. From the box:

```bash
curl -sI -H 'Host: www.fluncle.com' https://www.fluncle.com/   # expect 200 HTML, not a challenge
```

If it returns a challenge, add a WAF skip rule / IP Access Rule allow-listing the box's egress IP (or a secret header the Worker exempts) before standing up the onion. This is the one Cloudflare change Tor needs — it belongs to the web onion, not to the `Onion-Location` header.

### 1b. Install a pinned-recent onionspray and create the project

Install onionspray ≥ 1.6.0 (ideally the latest 1.8.0+) and create one project that maps `www.fluncle.com` → a generated web onion. The project `.conf` is where the real configuration lives. Two safeguards are **OFF by default** and must be turned on explicitly — pinning a recent version only makes them available:

```
# onionspray project .conf (illustrative — consult onionspray docs for exact keys)
set hardmode 1
onions www.fluncle.com
set nginx_proxy_ssl_trusted_certificate /etc/ssl/certs/ca-certificates.crt   # verify upstream cert (OFF by default)
set tor_pow_enabled 1                                                         # PoW DoS defense (OFF by default)
set tor_pow_queue_rate 250
set tor_pow_queue_burst 2500
set tor_intro_dos_defense 1                                                   # intro-DoS defense (OFF by default)
set tor_intro_dos_rate_per_sec 25
set tor_intro_dos_burst_per_sec 200
```

- **Upstream cert verification.** The whole design re-originates clearnet HTTPS to Cloudflare, so verifying Cloudflare's cert is the single most load-bearing config line. Without `nginx_proxy_ssl_trusted_certificate` set, the proxy is open to upstream MITM (Tor Project Security Advisory 002). Confirm it is configured, not merely that the version is ≥ 1.6.0.
- **DoS defenses.** Onion services are a known DoS target via introduction-point flooding. PoW and intro-DoS are disabled by default in C-Tor and in onionspray. Confirm both are actually enabled in the running config, not assumed.
- **Upstream `Host` must stay `www.fluncle.com`.** The router does a host-based rewrite (`apps/web/src/router.tsx` branches on `galaxy.` isomorphically), so a wrong upstream `Host` breaks the page after hydration even when the first byte looks fine. onionspray preserves the upstream `Host`; verify past hydration in a real browser, not just `curl | head`.
- **Strip upstream `Onion-Location`.** The Worker emits `Onion-Location` on HTML; have onionspray strip it from upstream responses so the onion never serves a pill pointing at itself.

Run onionspray's `config` → `make` → `start` lifecycle per its docs, and note the exact path where it writes the generated `HiddenServiceDir` — you need it for the backup below.

## 2. (Optional) Add the rave/SSH onion

A second `HiddenServiceDir` pointing at the Wish app:

```
HiddenServiceDir /var/lib/tor/rave_ssh/
HiddenServicePort 22 127.0.0.1:22     # the Wish app — NOT :2222 (that's admin OpenSSH)
```

Point the onion at `:22`, not `:2222`. Production binds the Wish app to `:22` (`FLUNCLE_SSH_PORT=22`), while the bootstrap moves the admin OpenSSH daemon to `:2222`, Tailscale-only. Pointing the onion at `:2222` would expose the wrong service. Verify on the box with `sudo ss -tlpn | grep -E ':22 |:2222 '` and confirm `fluncle-ssh` owns `:22`. Read the address with `sudo cat /var/lib/tor/rave_ssh/hostname`.

The SSH host key still authenticates the host process independently of the onion address, so first connect shows the usual TOFU fingerprint prompt. Publish the rave host-key fingerprint on the clearnet site (e.g. `/about`) so a first-connect user can verify it out-of-band. One incidental, on-theme effect: the onion forwards from `127.0.0.1`, so the crew counter shows a Tor raver as `VOID` (loopback maps to `VOID`) — an anonymous traveler from nowhere on the map. Tor adds latency (roughly 5× a direct path), so the terminal is usable but visibly laggier; the Galaxy mini-game feels it most.

## 3. Keys, and the CRITICAL backup (a hard checkpoint)

Each `HiddenServiceDir` holds `hostname` (the address) and `hs_ed25519_secret_key` / `hs_ed25519_public_key` (the identity). **The secret key IS the address.** Lose the dir and the `.onion` is gone forever — no re-issue, no recovery; every published link, the `Onion-Location` header, and the docs would break. A **leak is as bad as a loss**: whoever holds the secret key can run a malicious clone at the published address, so this is about confidentiality too. Keep on-box perms `0700 debian-tor:debian-tor` and treat the backup like a root credential.

**Back up each `HiddenServiceDir` off-box immediately after first start**, into the configured 1Password item (see the ops runbook note), matching the custody `docs/dig.md` uses for Cloudflare creds. This is the single irreversible step — gate the rest of the rollout on it being done.

## 4. Firewall — nothing to open

No new provider-firewall rule. No new UFW allow rule. The HTTP proxy binds `127.0.0.1`; the SSH onion reuses the already-open `:22`. Verify nothing was opened by reflex.

## 5. systemd

Use onionspray's / the package's own service management. The stock Debian `tor` unit is already fairly hardened (its own `debian-tor` user, `NoNewPrivileges`, `ProtectHome`, `PrivateTmp`, a syscall filter); confirm the running unit meets the bar set by `apps/dns/fluncle-dns.service`. If extra hardening is wanted, add a `systemctl edit` drop-in rather than forking the vendor unit, so package updates keep flowing.

```bash
sudo systemctl enable --now tor      # or onionspray's service
sudo systemctl status tor --no-pager
```

## 6. Ship the `Onion-Location` header (Unit C — repo + normal deploy)

This is the only repo change, and it is already merged inert. `appendOnionLocation` (`apps/web/src/lib/server/agent-discovery.ts`) advertises the onion twin on every HTML response, per-path, gated to `text/html` so the JSON/XML surfaces (`/api/v1/*`, `/rss.xml`, `/mcp`) stay clean. It reads the web onion from the `WEB_ONION_HOSTNAME` module constant, which ships empty — while empty the helper is a no-op, so nothing is advertised until the address exists.

**To go live, set the constant and push.** In `apps/web/src/lib/server/agent-discovery.ts`, set:

```ts
const WEB_ONION_HOSTNAME = "<the-web-onion-56-char-host>"; // no scheme, no trailing .onion
```

then `git push`. Workers Builds deploys it (watch for build coalescing if pushing rapidly). No DNS record, no Cloudflare dashboard toggle — an onion has no DNS and the header is pure code.

## 7. Verify

```bash
# Read the addresses off the box (use onionspray's actual HiddenServiceDir path from step 1):
sudo cat <onionspray-web-hsdir>/hostname
sudo cat /var/lib/tor/rave_ssh/hostname

# Before the onion: confirm Cloudflare doesn't challenge the proxy's origin fetch (step 1a):
curl -sI -H 'Host: www.fluncle.com' https://www.fluncle.com/   # expect 200 HTML, not a challenge

# From a Tor-enabled shell (a standalone `tor` exposes SOCKS5 on 127.0.0.1:9050):
torsocks curl -s http://<web-onion>.onion/ | head
torsocks curl -s http://<web-onion>.onion/rss.xml | head

# MCP over the onion (stateless JSON-RPC; the server ignores Accept and always returns JSON):
torsocks curl -s http://<web-onion>.onion/mcp \
  -H 'Content-Type: application/json' \
  -H 'Accept: application/json, text/event-stream' \
  -d '{"jsonrpc":"2.0","id":1,"method":"tools/list"}'

# The rave terminal over the onion (needs netcat-openbsd; see "What a user needs"):
ssh -o ProxyCommand='nc -X 5 -x 127.0.0.1:9050 %h %p' -o VerifyHostKeyDNS=no <rave-onion>.onion
ssh -t -o ProxyCommand='nc -X 5 -x 127.0.0.1:9050 %h %p' -o VerifyHostKeyDNS=no <rave-onion>.onion latest

# Onion-Location header on clearnet, after the Worker deploy (HTML pages only, per-path):
curl -sI https://www.fluncle.com/ | grep -i onion-location
curl -sI https://www.fluncle.com/log | grep -i onion-location
curl -sI https://www.fluncle.com/rss.xml | grep -i onion-location   # expect NO header (JSON/XML)
```

Then drive a real browser over the onion and check **past hydration** — the page must not flip to the wrong content after the client router hydrates. When `torsocks curl` returns the archive, the page survives hydration, the SSH onion opens the rave terminal, and Tor Browser shows the ".onion available" pill, the surface is live. Also confirm the DoS defenses and upstream cert verification are actually on in the running config — don't assume the version pin did it.

## What a user needs

Tor Browser handles the website onion (it shows the ".onion available" pill on desktop; Onion-Location is desktop-only, not Android). For SSH, MCP, or curl, the user runs their own `tor` client (SOCKS5 on `127.0.0.1:9050`):

```bash
# macOS:         brew install tor && brew services start tor
# Debian/Ubuntu: sudo apt install tor && sudo systemctl enable --now tor
```

Two footguns to keep in mind:

- A standalone `tor` daemon uses SOCKS port **`9050`**; the Tor Browser bundle uses **`9150`**. The wrong port is the single most common failure.
- The `nc -X 5` SOCKS5 form needs **`netcat-openbsd`** (macOS's stock `nc` works; GNU netcat does not support `-X`).

A persistent `~/.ssh/config` block lets the user just type the onion:

```
Host *.onion
  ProxyCommand nc -X 5 -x 127.0.0.1:9050 %h %p
  VerifyHostKeyDNS no
```

`VerifyHostKeyDNS no` is non-negotiable: without it OpenSSH tries an SSHFP DNS lookup before invoking `ProxyCommand`, which fails (no DNS for `.onion`) and leaks the lookup. For HTTP clients, point at `--proxy socks5h://127.0.0.1:9050` — the `socks5h` form makes Tor resolve the `.onion`, so it isn't leaked locally.

## A dormant caveat to record (not act on)

If Fluncle's MCP server ever gains streaming / server notifications (`text/event-stream`), the onion proxy would need `proxy_buffering off`, `proxy_cache off`, `chunked_transfer_encoding off`, and `proxy_read_timeout ≥ 300s`, or buffering breaks SSE. Today it does not stream, so this is a future flag, not a build item.
